> "The long, highly random, security codes could be identified by

> I'm not sure what this means? Are you saying that if someone attempted
> to guess the
> long and highly random passwords that it would be possible if they did
> so over a long period of time using a brute force attack?

Yes, because there is no lock out.



> Just btw, this example of people trying to hack smart power meters is a
> very relevant one. This happens all the time
> and people find very creative ways to bypass the built in mechanisms to
> prevent people from trying to hack them. Especially
> in my country (South Africa) this is something that people like to do

> Maybe an example to add
> is to detect if someone is just completely bypassing the meter but

> notice
> that it is being bypassed.

So a detection point might be "no use of electricity" or something like that?


>  People also often like to put these meters

Interesting. I wonder how the app would "know".

> - -----------------
> Another suggestion:
>
> I didn't see anything in the book(although at the time of writing this
> at his been some time since I read the book) about using app sensor to
> detect people who have gone past exploiting the web application and
> now have a shell on your box.  You could perhaps use app sensor to for
> example check any activity on
> the machine running as the www-data or www user and doing something
> out of the ordinary.  I'm not sure how this would be done , but just a
> thought.

I think the idea of detecting host, configuration and code changes was
mentioned briefly on this mailing list a few years ago, but it hasn't
really been built upon, and as you say does not appear in the book at
all. I suppose if the host is compromised perhaps we can't trust the
application either, but in our mobile app example we have this
potential issue too. Perhaps we should add a chapter about
physical/environment changes?
