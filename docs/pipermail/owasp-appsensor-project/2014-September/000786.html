Thanks Juan!

This is a really big win for us. Language interoperability was one of the 2
major design goals of v2 (along with client/server deployment). The work
Juan and team have done shows this is possible - big kudos there.

There has been a lot of work done in the last couple years around api code
generation, so I'm hoping to look into that as a way to support different
languages in a minimal way. However, real power will come with the ideas
Juan put forth about framework integrations. Having a module that handles
responses in a .NET specific way is a huge win to developers wanting to
implement appsensor. It means they can just get started. I think this will
be a huge area of possibility for us going forward. I think what the
netflix folks have done with ensnare (
https://github.com/ahoernecke/ensnare/wiki) is really remarkable. They
plugged into rails directly and got huge benefits. The base of the
appsensor mechanism is there now. Taking it to frameworks directly
(something owasp talks about all the time) would make it so much easier for
devs. This is what Juan and team are doing and I think it's awesome and the
path we should head towards.

Also, the dev list is cc'ed already.

Thanks,
John

> Looking for even further interoperability with a .NET technologies, we
> created a WSIT-compatible AppSensor Metro Web service . The intiial problem
> was that jersey SOAP web service is not compatible with .NET and consuming
> the REST interface would require manual effort and would be "unnatural" to
> what .NET dev teams are accustomed, plus making it .NET compatible will
> greatly facilitate its usage, as it will be just like if it was other .NET
> WCF Web Service.
>
> Now .NET developers just have to setup an AppSensor server and add a Web
> Reference to their .NET Project from Visual Studio. Then they will be able
> to report events, add attacks and get responses from the AppSensor server
> as .NET objects they can manipulate.
>
> We are also working on a .NET HTTP module for addressing those responses
> and block access or log out users as necessary, currently the module can
> blacklist URLs and cookies. This however will take much longer to be
> implemented at full. It might not be necessary if appsensor responses are
> implemented on ModSecurity rules or similar. So suggestions are welcome.
>
> I am attaching some screenshots of the service and the test .NET project
> using it.
