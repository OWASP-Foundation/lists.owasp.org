Oh good, that sounds interesting.  I'll update the ModSecurity
cross-references on the Detection Point page once the next rule set
comes out.

In the summit session about Enterprise Web Defense Roundtable...

   http://www.owasp.org/index.php/Summit_2011_Working_Sessions/Session068

... I've scribbled a note to myself about how an application could
inform other systems (e.g. SIEM, WAF, audit software) what its
allowable entry points were - some sort of application surface
definition language.  Ivan Ristic did work on this a while ago where
self-learning WAFs might be able to export their impression (profile)
of the entry points.  It would be nice if there was a common way of
expressing this.  Of course the application (should?) knows this and
doesn't have to guess.

Then for example, if AppSensor could tell a WAF what its surface was,
the WAF could say signal back to AppSensor details of missing static
content requests (ACE3: Force Browsing Attempt) which the application
might not otherwise know about, because the web server normally deals
with the error.  The application profile might have enough information
to assist with some of the other RequestException detection points
too.

Colin
