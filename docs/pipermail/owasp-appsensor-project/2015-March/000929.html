On Fri, Feb 27, 2015 at 5:11 AM, Colin Watson <colin.watson at owasp.org <mailto:colin.watson at owasp.org> > wrote:

John / All

 

I did begin a little on this yesterday, so it is timely to get some more encouragement (pressure!) to get on with it.

 

When John created the text for the developer flyer, and we got feedback on it, that made it really easy for me to do the subsequent design part. At the moment I feel stuck on some fundamental things that everyone on this list could help decide:

1.	The developer flyer is a two-sided A4 or US letter leaflet, which is great for handing out in large quantities
2.	I wondered if the CISO document should be something a bit thicker/plusher since it is a different audience

*	What size? (see Lulu options... http://connect.lulu.com/t5/Product-Pricing-Information/Binding-options-and-their-page-counts/ta-p/33673 we used Crown Quarto for the Guide, but that isn't an option for <32 pages) - US Letter or Square?

I'd prefer "printer paper" - something standard sized, which I'm guessing is A4 / US Letter. 

*	How many pages - just two or a little booklet say 10 12 or 16 sides

The overview in the guide is about 30 pages. There's some stuff that can be removed, but a significant amount of value remains. While a professional printing/binding is nice, I'd still plan for people who want to print it out and staple it. That can be done double-sided, so I'd say 12-15 pages sounds about right once you strip down the content, add a nice front page, and then account for section/page breaks for printing.

*	How bound (coil, saddle stitch or hardcover) see http://www.lulu.com/create/books/ for dependencies on sizes, binding, colours and number pages
*	Maybe colour inside?

Yeah, I imagine more people will print this one in color, so that'd be nice
 

Once we have a feel for the size of the thing, it will be easier to determine what text can go in it.

 

What is the key messaging we want to push? Some buzzy phrase ideas:

*	Real time
*	Telemetry and instrumentation 

You can add "cloud friendly" in this bucket since cloud makes obvious the need for visibility into applications

*	Intelligence driven security
*	Instrument applications
*	Active defense

Augment to say it's "automatic" (the system handles the response) and that you can respond however you want (not a canned set of responses) - if you can implement it in code, you can do it.

*	Integrates with existing monitoring systems and network defences
*	???
*	???
*	Already used at scale

I'd also add "forward thinking" using some of the references that have said this is the next direction to go

 

 

We could then put some draft text up in a Google doc for editing?

Yep - that makes the most sense - accessible and supports comments 

 

Colin
