IIRC, there was some discussion about eventually moving from Subversion
to Git, because it's supposedly much easier to manage branches and merging
back to the trunk with Git.  I don't recall where we left that conversation
at. I
know there was concern of looking the version history if we moved.  Google
Code now is supporting Git, so I don't think it's necessary to move to
GitHub
to do this.At least Google Code hasn't recently had any (public) security
breaches.
Anyhow, I still think that's open to debate as to whether we move and where
to.
I've not really given it any more attention since GSoC 2012 ended.

Presently, it seems to be on 2 AppSensor branches. I'm not sure if John has

 - Where might we put other future code examples?
>

I would suggest creating a "contrib' subdirectory and perhaps put them
there. Chris Schmidt did that for ESAPI Java. E.g., see:
<http://code.google.com/p/owasp-esapi-java/source/browse/#svn%2Fcontrib>

- How will they be differentiated?
>

I think that they clearly should be kept separately from AppSensor proper.
The
bigger question IMO is how do we manage who can upload what examples. I'm
personally not comfortable with just giving commit privs to anyone who wants
to upload an example. Until it becomes unmanageable, I think a few of us
should vet the examples and perhaps take responsibility of uploading them.
If it becomes a lot of work because many are submitting examples, then we
might have to create a separate project for it and allow the contributors to
have commit priv there.  But to start off with, I think it's sufficient to
just
make a 'contrib' subdirectory similar to how we did for ESAPI.


> - Do we have preferred names for the previous and new versions (e.g.
> Integrated Java implementation, and SOAP Web Services)?
>
> How about just AppSensor and AppSensor-WS? Or if we are going to
create a REST based web service and treat it separately, perhaps
AppSensor-SOAP-WS and AppSensor-REST-WS. If course if we do
that, we should probably refactor things to break out the common
components into a common core (AppSensor-Core?).


> It doesn't have to decided now, but I am hoping we don't lose
> older/other versions. I'm just flagging the issue up.
>

I agree that we don't want to lose the version history. There are ways
and tools to support moving from svn to git or svn to hg (Mercurial),
but I just don't think it's been on anyone's immediate radar.

BTW, as far as additional documentation is concerned, don't forget that
the AppSensor GSoC work is documented on the AppSensor Google
Code wiki. Specifically, <http://code.google.com/p/appsensor/wiki/GSoC2012>.
There's not a lot there--it's mostly status updates--but you might find
something
useful there.
