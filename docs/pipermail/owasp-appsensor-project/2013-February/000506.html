>On Fri, Sep 30, 2011 at 11:52 AM, Ryan Barnett <ryan.barnett at owasp.org>
>wrote:
>> We currently have InputException (IE) and this would be its counterpart
>>and
>> would include all issues related to inspecting the HTTP response.
>>
>> There are a number of current detection points that could move here -
>>
>> IE7: Detect Abnormal Content Output Structure ­ could actually move
>>here to
>> CIE2: Detect Abnormal Quantity of Returned Records
>>
>> There is also a new Detection Point for this category that I recommend -
>>
>> - Deviation from normal response time interval ­ which could detect if
>>an
>> application is under a resource DoS attack.
>>
>> So the new category could be this -
>>
>> OutputException (OE)
>>
>> OE1: Detect Abnormal Content Output Structure
>> OE2: Detect Abnormal Quantity of Returned Records
>> OE3: Deviation from normal response time interval
>>
>>
>> Thoughts?
>
>Could an OE detection point detect HTTP Response Splitting?
>Or be used as a crude DLP (similar to your proposed OE2, but
>the "abnormal quantity" may only be 1)?
>
>That would be sweet.

Exactly.  That is the concept that is used in many WAFs when inspecting
outbound data.  It can track where and the amount of sensitive data in the
app.  For instance if a certain page on a site displays 1 SSN to a user
when logged into their account and then suddenly 500 are shown (as a
result of a successful SQLi), then an alert should be triggered.
