My take is that we actually have 3 classes of detection points:

- static: the rules originally defined, these detection points are 
written into the application.
- dynamic: the rules Michael is proposing and I call statistical. These 
detection points 'bake' the threat model directly into the application.
- random: rules that behave as dynamic honey-pots, these detection 
points exist for every input in the application.

I would like us to have strong frameworks defined around all three.
However, this is only half the picture - the other half is response. I 
imagine that there is also three kinds of response as well, they are 
also:

- static: binary responses (e.g. we know this is bad)
- dynamic: this is statistical response, for example a simple dynamic 
rule a scale of 1-10:
	* 1-3: this is unusual but safe behavior so allow & log all further 
requests
	* 2-6: this is normal behavior so do nothing
	* 3-9: this is abnormal bad behavior so deny & log all further requests
	* 10: this is an attack so deny & raise an alert
- random response is a 'self pen-testing application' by using the 
random detection points; and using random responses we can produce a 
full monte-carlo situation where the application has the only knowledge 
about if something is actually vulnerable or not. So either the attack 
is guaranteed to be real or it can be an application produced 'random 
attack.' The application can both test how quickly the humans 'detect 
and respond' as well as 'randomly allow' a real attack in order to 
collect information about the vulnerability only to block it at a later 
point so that a patch can be created from the resulting information 
collected.
