>> AppSensor Team,
>>
>> I'd like to create a child project within AppSensor called "Automated 
>> Attack
>> Prevention". This project will be a study of current techniques to 
>> thwart
>> automated attacks against application. The deliverable will be well 
>> documented
>> knowledge and best practices (just like appsensor). The deliverable 
>> could be
>> both stand alone and also as a chapter within the overall AppSensor 
>> book.
>>
>> This idea is very similar to the defensive nature of AppSensor; 
>> however, the
>> differences of the material and approach warrants it to be slightly 
>> separated
>> as a child project.
>>
>> Here are a few examples of material we would cover.
>> - Brute Force Login Protection :
>> -- Defensive options: Threshold based captcha, IP blocking, Response 
>> time
>> delay
>> -- Triggers: Can be tied to attacker (IP) or victim (targeted 
>> account).
>> Example: Multiple failed logins from a single IP against any number 
>> of target
>> accounts OR multiple failed logins from multiple IPs against the same 
>> username
>>
>> - Hashcash concepts and how they might apply in web apps
>> (http://en.wikipedia.org/wiki/Hashcash)
>>
>> - Detection of common web scanner requests and optional response 
>> tactics
>> (drop, ignore, block ip, feed fake response data)
>>
>> - and more
>>
>>
>> I think the material is related enough to fly under the AppSensor 
>> umbrella,
>> but different enough to become a child project. Thoughts?
>>
