Hate to be a fly in the ointment, but I think we need to think<br>
this through a bit more.<br><br>
I think one question that ought to be asked if a person is normally<br>
using a phone or other mobile device with GPS or other location<br>
info made available and then all of a sudden, that information<br>
stops coming, how would / should this detection point react?<br>
Would it even be triggered? Might we consider it a normal thing<br>
that someone disables their GPS? (I do that all the time to save<br>
battery. Even more often since the Snowden leaks, but I digress.)<br>
But if the geo-location information is based on something like<br>
GPS or the courser-grained cell-tower triangulation and then an<br>
app stops getting that? Do we flag it or not? How do we recommend<br>
an application using this detection point in AppSensor proceed<br>
if that information suddenly becomes unavailable?  Note that<br>
having an answer for this is important. If we simply say "okay,<br>
it is normal for people to disable these services to save batteries<br>
or whatever", then an attacker can just not send that information.<br>
On the other hand, if you try to locate this simply based on<br>
information for IP addresses, that is liable to have a lot of false<br>
positives. For instance, if I login to my company VPN from home<br>
and access your site, it puts me at one location. However, if I<br>
access your site from the same PC directly (while still at my home),<br>
it may look like I'm instantly a thousand miles or more away.<br><br>
The other issue that I have with basing location solely on IP<br>
addresses is I'm not sure how well that is actually going to work<br>
once (someday!) we all regularly start using IPv6.<br><br>
