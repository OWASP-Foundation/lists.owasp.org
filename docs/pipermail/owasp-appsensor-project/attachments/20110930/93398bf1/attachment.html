This is quite interesting, since I do believe that growing the capabilities and adding detection points is a good thing. :-) <div><br></div><div>What concerns me is that since we have a stochastic anomaly detection model we can adapt the behaviour of the application based on algedonic responses, those algedonic responses are the result of a threshold of pain inside the stochastic anomoly system from input from the detection points. <div>

<br></div><div>This is important, because if we go around and start naming detection points after exploits; people forget that it is the stochastic anomaly system that is providing the protection - especially if you haven&#39;t got your heard around the idea yet. And this will lead to people building all kinds of attack signatures based on exploits, and we have just become another web-application-firewall-signature based device. </div>

<div><br></div><div>Where as I think the real ideal goal would be for every application to actually send every single input validation violation to the stochastic anomaly detection engine. Thus I think we should use the terminology necessary to communicate - but we need to be very, very careful. I would not even want to try and name every kind of throw that results from an input validation violation. </div>

<div><br></div><div>In my mind - we throw all exceptions, the stochastic anomaly detection catches and a decision is made by the rules engine and the application adapts accordingly. </div><div><br></div><div><br></div><div>
