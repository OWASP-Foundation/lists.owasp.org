<div><br></div><div>As for your point about Mod Security / Incompleteness.</div><div><br></div><div>AppSensor describes a 'best practice' and can be built into new applications. However, the very important question remains "what about legacy applications?" How do you do an AppSensor then? And this is the primary use case for most of the world. So while you can add tags, and they are not as specific as the application detection points - I personally think that is a giant win and move forward for the vast majority of people who have legacy issues. After all, AppSensor is also a "Security Architecture" pattern - and having mod security provide censors for applications that would otherwise never have them is a giant win. </div>

<div><br></div><div>Sure it isn't complete, but maybe other IDS/IPS/SEIM/WAF vendors will start doing the same and we can grow that coverage from precisely zero to something more than nothing, and something less that building it in. I just can not see how this doesn't create value for security architects, and thus create an opportunity for additional security controls custom to the architecture to be put in place to provide better protection.</div>

<div><br></div><div>Remember the key idea of AppSensor is a risk base approach of treating the impact instead of the vulnerability or threat. This is not unlike fire suppression systems in buildings. What is important is not the quality of the sprinklers in the ceiling (although that is a factor) but the coverage of the sprinklers in the building. </div>

corresponding AppSensor detection points - the fact is that this is not a<br>
robust option.  While I do think that WAF alerts can funnel into AppSensor<br>
alerts, it is not complete enough in its normal default configuration to<br>
be considered a true "AppSensor" imlementation.  The key piece here is<br>
that most WAFs work as "attack detection systems" for a very narrow<br>
category of attacks - mainly Injection flaws.  In order to have good<br>
AppSensor coverage, users need to customize the detection points for their<br>
application.  It is this key piece that has caused me to re-think strategy<br>
here for ModSecurity.  What I am considering is to still have basic<br>
mappings in the OWASP CRS to AppSensor detection points, but to possibly<br>
develop a completely separate "AppSensor" ruleset for ModSecurity.  These<br>
rules would utilize the Lua API to implement the detection points and it<br>
would take some customization by the user.<br>
