&gt; Question for you all – I am updating the OWASP CRS for more coverage of AppSensor Detection Points.  I am working through these step by step.  When reviewing RE1 and RE2 – it seems to me like the detection logic for these should be switched.<br>




&gt;<br>
&gt; RE1 should be the initial check to see if a request method submitted is not a valid one at all per the RFC.  So, if someone sends JUNK or TEST as the request method, then RE1 would trigger.<br>
&gt; RE2 should then be the check to see if the request method submitted is not one that is expected for the current resource (expecting a POST but receiving a GET).<br>
&gt;<br>
&gt; The reason why I came to this conclusion is that when implementing these rules in ModSecurity, if we keep the current ordering and using blocking, RE2 would never trigger as an unknown HTTP Request method would first violate RE1.<br>




&gt;<br>
&gt; To summarize – I think the detection logic of these two are fine but that the detection point labels should be switched.<br>
