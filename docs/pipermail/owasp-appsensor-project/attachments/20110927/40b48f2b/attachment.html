<b>Background:</b><br>
<br>
During the summit it was proposed that the Intrusion Detection Analysis Engine (not sure what it is called) would be separated out as a separate process, but kept in Java.&nbsp; Then the code that implements the detection points and response actions could be written
 in Java, .Net, PHP and so on to interface with this engine I imagine via some form of inter-process communication.&nbsp; I offered to write some initial detection-point PHP code as a test that would allow developers to insert detection points and echo any detections
 back to the screen (or to a file, or to syslog, since PHP already has syslog integration) as a first pass until John was able to start on the next gen Engine.<o:p></o:p></p>
</blockquote>
<div>
<p class="MsoNormal">Yes, and we're stoked about that. <o:p></o:p></p>
</div>
<blockquote style="border:none;border-left:solid #CCCCCC 1.0pt;padding:0cm 0cm 0cm 6.0pt;margin-left:4.8pt;margin-right:0cm">
<p class="MsoNormal"><br>
<br>
<b>Some Thoughts, Observations and Questions:</b><br>
<br>
1. The current AppSensor was designed to play well with ESAPI.&nbsp; Based on the very little I know about ESAPI, I would imagine that the detection-point code should continue to depend on ESAPI.&nbsp;
<span style="color:black">Would the Engine also have to be dependent on ESAPI?</span><o:p></o:p></p>
</blockquote>
<div>
<p class="MsoNormal">The intrusion detector bit, yes, should probably stay at least ESAPI-compliant since we want to &quot;play together well&quot; for sure. However, we will also likely provide non-esapi hooks for the client in case someone is anti-esapi for some reason,
 or just can't doesn't want to bring it in, and that version should still talk to the engine just fine. The engine component certainly does not HAVE to use ESAPI at all. We'll make that determination during the design, and will use it only if it makes sense.<o:p></o:p></p>
</div>
<blockquote style="border:none;border-left:solid #CCCCCC 1.0pt;padding:0cm 0cm 0cm 6.0pt;margin-left:4.8pt;margin-right:0cm">
<p class="MsoNormal"><br>
2. I was going through some of the current AppSensor/ESAPI Java code, and also looking at the ESAPI PHP code and noticed that the Java version used a &quot;.properties&quot; format for the config, but the PHP ESAPI used an XML config file.&nbsp; Based on this, I will probably
 use an XML config file for any AppSensor-specific configs for the detection-point code.&nbsp; However, it should be trivial to fall back to a &quot;.properties&quot; file format if need be.<o:p></o:p></p>
</blockquote>
<div>
<p class="MsoNormal">This is DEFINITELY up in the air for me at the moment. I want to hear what the ESAPI folks are doing here, and will probably stay with that. I know we're a different project, but we are going for &quot;drop in&quot; replacement and changing our config
 files does not make us drop in. Having said that, I don't know the status on the PHP side for ESAPI. For now, go w/ XML as you said and just build some abstraction like we discussed so you can swap in a properties file reader if necessary
<o:p></o:p></p>
