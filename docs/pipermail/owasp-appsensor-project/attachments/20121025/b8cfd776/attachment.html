<blockquote
 cite="mid:CAOPE6Pi4rxadZxUZTdiEJ5wFZ4xa25kWFmxCgAwn9iyCL8e+Xw@mail.gmail.com"
 type="cite">I think how you would do that would depending on what
cookie it was.<br>
For any other cookie other than the "session" cookie (e.g., JSESSIONID,<br>
ASPSESSIONID, etc.), you could store the original value of the session<br>
in the associated HTTP session object and whenever you receive it,<br>
compare it to what you have there.<br>
  <br>
If it is something like JSESSIONID though, storing it there won't work.<br>
I'd recommend trying to detect something like that using a JavaEE<br>
servlet filter or (in IIS, an HTTPModule).<br>
  <br>
Of course, you could always make the tampering self-evident by<br>
adding an HMAC to it and then validate the HMAC whenever you<br>
validate the cookies' value. That would be the best approach, but<br>
if you are dealing with cookies from 3rd party libraries / applications<br>
you may not be able to do that.<br>
  <br>
Hope that makes sense.<br>
