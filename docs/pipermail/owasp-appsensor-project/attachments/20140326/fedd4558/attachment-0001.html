<div dir="ltr">Colin, <br>I haven't collected the full POST request, but the only header we care about is the one you specified ('X-Appsensor-Client-Application-Name' is the default, but you can change this via configuration). <br>

<div id=":1ef" class=""><br></div><div id=":1ef" class="">As for the multiple hosts question, I am simply assuming that application is the level of granularity you care about. The system is flexible enough to support multiple hosts, but you would handle it using application name suffixing as you mentioned above. The only thing you'd want to do there is ensure that you add all your hosts for a given application to a correlation set so that end users are considered the same across "client applications", in this case hosts of a single client application.<br>

<br></div><div id=":1ef" class="">Regarding attacks and responses ... I don't have the JSON handy (in the middle of a machine rebuild), but ...<br></div><div id=":1ef" class="">- The request for attacks would look the same, except to '/attacks' <br>

</div><div id=":1ef" class="">- The request for responses would be a 'GET' request to '/responses?earliest=12345' where the query parameter 'earliest' is a unix timestamp that represents the "start time" to retrieve responses from. If you sent a timestamp of 5 minutes ago, you would get the responses created in the last 5 minutes. It's a simple polling mechanism and the client will simply update that timestamp after every successful call. That way they only pull the responses they haven't retrieved yet.<br>

<br></div><div id=":1ef" class="">We are not doing identity authentication or message integrity checking. The basic assumption I'm working from at this point (after talking with Kevin Wall a good bit on this topic) is that we'll just rely on web servers to do authentication, then pass the client name header through and then we'll do authorization. By using the reverse proxy setup, we save ourselves a good bit of effort and give users a lot of deployment flexibility.<br>
