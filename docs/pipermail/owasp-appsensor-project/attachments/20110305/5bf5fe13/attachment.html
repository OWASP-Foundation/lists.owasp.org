<br>
In the main log,<br>
<br>
- `id_log` is an auto-incrementing ID<br>
<br>
- `log_timestamp` is determined when the log entry is written - not<br>
when the event was detected<br>
<br>
- `severity` is<br>
      0 (Emergency/Application unavailable for all users)<br>
      1 (Alert/Function unavailable for all users),<br>
      2 (Critical/Function or application unavailable to a single user),<br>
      3 (Error/Other security events not included in codes 0, 1, 2 or 4),<br>
      4 (Warning/A security event but user allowed to continue),<br>
      5 (Notice: normal but significant condition)<br>
      6 (Information/Normal user behaviour)<br>
<br>
- `event_timestamp` is the event date/time<br>
<br>
-  `event_type` is a broad description e.g. &quot;Input Validation&quot;,<br>
&quot;Authentication&quot;, &quot;Session Management&quot;, &quot;Output Validation&quot;<br>
<br>
-  `action` was the original intended purpose of the request e.g. Log<br>
In, Refresh Session ID, Log Out, Request<br>
<br>
-  `object` is the affected component or other object (user account,<br>
data resource, file) e.g. URL, Session ID, User Account<br>
<br>
- `result_status` is whether the ACTION aimed at the OBJECT was<br>
successful e.g. Success, Fail, Defer<br>
<br>
- `reason` is why the status above occurred e.g. User not<br>
authenticated in database check ..., Incorrect credentials<br>
<br>
- `http_status_code` is the status code returned to the user (often<br>
200 or 301, need to imagine many log entries here for a single<br>
request)<br>
<br>
- `event_confidence` is a percentage 0-100...confidence the event<br>
identified occurred as stated above - often 100% for whitelist input<br>
validation, but maybe less than that for data from a WAF about<br>
possible XSS attack<br>
<br>
- `location` is where event was detected - script or component (not the URL)<br>
<br>
- `host` e.g. <a href="http://www.example.com" target="_blank">www.example.com</a><br>
<br>
- `service_name` e.g. &quot;Colin&#39;s App&quot;<br>
<br>
- `service_port` e.g. 443<br>
<br>
- `service_protocol` e.g. http<br>
<br>
- `http_method` e.g. GET, POST<br>
<br>
- `entry_point` a URL e.g. /cw/Bench.php<br>
<br>
- `request_number` is a unique integer for every single request<br>
received by the application, and will be shared by a number of ecent<br>
log entries, so they can be correclated with each other i.e. one<br>
rqeuest may have multiple input validation errors<br>
<br>
- `source` varchar(45) CHARACTER SET latin1 NOT NULL COMMENT &#39;IP address&#39;,<br>
<br>
- `user_identity` might be a user ID, username or perhaps 0 for<br>
unauthenticated&#39;,<br>
<br>
- `http_user_agent` is HTTP_USER_AGENT<br>
<br>
-  `user_fingerprint` is a SHA!1 hash of user agent/request properties<br>
   sha1(substr(trim($_SERVER[&#39;REMOTE_ADDR&#39;]),0,7).trim($_SERVER[&#39;HTTP_USER_AGENT&#39;]).trim($_SERVER[&#39;HTTP_ACCEPT_ENCODING&#39;]).trim($_SERVER[&#39;HTTP_ACCEPT_LANGUAGE&#39;]));<br>
<br>
- `hash` a SHA1 hash of a salt concatenated with the hash of the<br>
immediately previous log entry and the log record above (including<br>
id_log) for integrity checking<br>
<br>
the only other thing I had wondered about adding was a field for which<br>
(e.g. type-identifier like RE3-2001) AppSensor detection point fired.<br>
This may be NULL for some entries.  Then perhaps also what responses<br>
ocuured (e.g. a list A,F,G or NULL).  That way it&#39;s easier to<br>
monitor/tune AppSensor.  Or possibly those two should be logged in<br>
another DB table..<br>
<br>
  id_log<br>
  appsensor_detectionpoint<br>
  appsensor_response (allow NULL)<br>
<br>
allowing for more than one record per id_log<br>
<br>
