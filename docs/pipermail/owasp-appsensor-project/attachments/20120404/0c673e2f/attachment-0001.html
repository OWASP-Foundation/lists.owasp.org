You should implement localised mis-use prevention for these 3 examples<br>
in your own code. For example "API abuse" is probably being detected<br>
by authorisation and input validation checks?  AppSensor can't be used<br>
to protect an application that isn't doing these things already, or<br>
has known or easily found vulnerabilities.<br>
<br>
AppSensor might then be used to detect an attack if it saw (for<br>
example) 6 or 10 instances of any of the three things you mention by<br>
an IP address range or subnet (e.g. 2 login account lockouts, plus 3<br>
new accounts and 1 API abuse = 6 events).<br>
<br>
1. Consider where in your architecture you could detect these sort of<br>
events (existing checks?), and whether this can be achieved with an<br>
common code<br>
<br>
2. Make sure you have application logging capabilities<br>
<br>
3. Start to log the events<br>
<br>
4. Consider what type of responses are appropriate (time delays, lock outs, etc)<br>
<br>
5. Build mechanisms to undertake/enforce the responses<br>
<br>
6. Use something (engine, query on logs, etc) to detect the thresholds<br>
you have set e.g. "6 of any", "3 account lockouts"<br>
<br>
7. Undertake the predefined responses<br>
