<h1></h1><h1>WBox</h1>
<h2>HTTP testing tool</h2>
<p>
Wbox aims to help you having fun while testing HTTP related stuff.
You can use it to perform many tasks, including the following.
</p>
<ul><li>Benchmarking how much time it takes to generate content for your web application.</li><li>Web server and web application stressing.</li><li>Testing virtual domains configuration without the need to alter your local resolver.</li>
<li>Check if your redirects are working correctly emitting the right HTTP code.</li><li>Test if the HTTP compression is working and if it is actually serving pages faster.</li><li>Use it as a <b>configuration-less HTTP server</b> to share files! (see the server mode documentation at the end of the Wbox tutorial in this page, but it's as simple as <i>% wbox servermode webroot /tmp</i>)</li>
</ul>
Wbox is <b>free software</b> under the BSD license and was written in ANSI C (POSIX runtime required) by <a href="http://invece.org">Salvatore 'antirez' Sanfilippo</a>.
<br><br><h1>Wbox tutorial</h1>
<h2>HTTP client mode</h2>
<p>
The following is a short tutorial. Wbox is trivial to use but you may 
want to read this in order to understand better what wbox is and how to 
use it.
</p>
<h3>Basic usage</h3>
The simplest way to use wbox is to call it with just one argument, an url:
<pre>% <b>wbox <a href="http://www.google.it">www.google.it</a></b>
WBOX <a href="http://www.google.it">www.google.it</a> (64.233.183.99) port 80
0. 200 OK    3774 bytes    407 ms
1. 200 OK    (3767) bytes    273 ms
2. 200 OK    3767 bytes    304 ms
3. 200 OK    3767 bytes    260 ms
user terminated
</pre>
As you can see wbox in this mode of operation resembles the <a href="http://en.wikipedia.org/wiki/Ping">ping utility</a>. For every performed HTTP request it shows the sequence number of the request (<b>0.</b>), the HTTP reply code and reason string (<b>200 OK</b>), the number of bytes received from the server (<b>3774 bytes</b>), and the time the request took to complete in milliseconds (<b>407 ms</b>). There are two details to note.
<ul><li>The number of received bytes includes the HTTP reply header, so it is not equal to the document length.</li><li>Every time it changes compared to the previous HTTP reply, the number of received bytes is put between parens, like in <b>(3767)</b></li>
</ul>
<h3>Limiting the number of requests</h3>
By default wbox will continue to perform requests forever, until you 
don't press Ctrl+C. After the url argument you can pass a number of 
options to wbox to alter its behaviour, one of this is an integer number
 specifying how many requests to perform.
<pre>% <b>wbox <a href="http://www.google.it/notexistingpage.html">www.google.it/notexistingpage.html</a> 1</b>
WBOX <a href="http://www.google.it">www.google.it</a> (64.233.183.103) port 80
0. 404 Not Found    1495 bytes    310 ms
</pre>
This time wbox terminated after the first request was received.
<h3>Obtaining splitted timing information</h3>
Usually web applications don't spend the same time in the generation of 
every part of the page. With wbox it's possible to obtain time 
information for different parts of the document using the <b>timesplit</b> option.
<pre>% <b>wbox <a href="http://digg.com">digg.com</a> timesplit 1</b>
WBOX <a href="http://digg.com">digg.com</a> (64.191.203.30) port 80
0. 200 OK    44134 bytes    1326 ms 
       [0] 0-4095 -> 728 ms
       [1] 4096-8191 -> 254 ms
       [2] 8192-12287 -> 16 ms
       [3] 12288-16383 -> 17 ms
       [4] 16384-20479 -> 225 ms
       [5] 20480-24575 -> 12 ms
       [6] 24576-28671 -> 16 ms
       [7] 28672-32767 -> 18 ms
       [8] 32768-36863 -> 16 ms
       [9] 36864-40959 -> 16 ms
       [10] 40960-44133 -> 8 ms
</pre>
<h3>Dumping data</h3>
We may like to see what kind of HTTP reply google is emitting on 404 errors, the option <b>showhdr</b> tells wbox to dump the HTTP reply header.
<pre>% <b>wbox <a href="http://www.google.it/notexistingpage.html">www.google.it/notexistingpage.html</a> 1 showhdr</b>
WBOX <a href="http://www.google.it">www.google.it</a> (64.233.183.103) port 80

HTTP/1.1 404 Not Found
Content-Type: text/html
Server: GWS/2.1
Content-Length: 1368
Date: Tue, 05 Jun 2007 16:49:58 GMT

0. 404 Not Found    1495 bytes    540 ms
</pre>
We may like to use wbox inside a shell script in order to collect different headers, so the option <b>silent</b> tells wbox to don't show status lines.
<pre>% <b>wbox <a href="http://www.google.it/notexistingpage.html">www.google.it/notexistingpage.html</a> 1 showhdr silent</b>
HTTP/1.1 404 Not Found
Content-Type: text/html
Server: GWS/2.1
Content-Length: 1368
Date: Tue, 05 Jun 2007 16:51:52 GMT
</pre>
It's also possible to dump the whole reply using the <b>dump</b> option.
<pre>% <b>wbox <a href="http://www.google.it/notexistingpage.html">www.google.it/notexistingpage.html</a> 1 dump silent</b>
HTTP/1.1 404 Not Found
Content-Type: text/html
Server: GWS/2.1
Content-Length: 1368
Date: Tue, 05 Jun 2007 16:53:44 GMT



<html><head>
<meta http-equiv="content-type" content="text/html;charset=utf-8">
<title>404 Not Found</title>
<style><!--
body {font-family: arial,sans-serif}
div.nav {margin-top: 1ex}
<b>(a lot of data omitted)</b>
</pre>
You can use <b>dump</b> and <b>timesplit</b> at the same time in order to get a dump of the reply with time annotations, but <b>warning!</b> using this feature the time used in order to produce the actual output on screen will alter the results, so you <b>must</b> redirect the output into a file. The right command line to use is the following:
<pre>% <b>wbox <a href="http://digg.com">http://digg.com</a> dump timesplit 1 > /tmp/dump.txt</b>
</pre>
Then open the file with an editor to see the HTTP reply splitted in chunks of data separated by time annotations.
<h3>Stressing servers and apps</h3>
Instead to make a single request it's possible to make multiple concurrent requests using the <b>clients <i>number</i></b> option. Wbox will spawn the required number of processes, and every process will perform a query every second.
<pre>% <b>wbox <a href="http://localhost">http://localhost</a> clients 4</b>
WBOX localhost (127.0.0.1) port 80
0. 200 OK    9939 bytes    14 ms
0. 200 OK    9939 bytes    22 ms
0. 200 OK    9939 bytes    26 ms
0. 200 OK    9939 bytes    31 ms
1. 200 OK    9939 bytes    4 ms
1. 200 OK    9939 bytes    7 ms
1. 200 OK    9939 bytes    7 ms
1. 200 OK    9939 bytes    4 ms
2. 200 OK    9939 bytes    4 ms
2. 200 OK    9939 bytes    6 ms
2. 200 OK    9939 bytes    10 ms
2. 200 OK    9939 bytes    3 ms
3. 200 OK    9939 bytes    3 ms
3. 200 OK    9939 bytes    4 ms
3. 200 OK    9939 bytes    7 ms
3. 200 OK    9939 bytes    7 ms
user terminated
</pre>
Every client will print a status line, so you can get an idea about 
delays under load. Note that's much better to perform this tests on 
localhost or over LAN in order to filter bandwidth limits that may 
otherwise alter the numbers.
<h3>Modifing requests period</h3>
By default wbox will perform a request per second, you can change this using the option <b>wait <i>seconds</i></b> (for every process). Set it to zero in order to perform continuous requests without delay between them using <b>wait 0</b>.
<h3>Setting cookies and referer</h3>
Many web applications perform different tasks for the same url if some 
cookie is set to the right value. One obvious example is a login cookie.
 Wbox is able to perform requests containing cookies using the option <b>cookie <i>name</i> <i>value</i></b> like in the following example.
<pre>% <b>wbox <a href="http://hping.org">hping.org</a> cookie secret ff8a45a7</b>
</pre>
It is possible to use the <b>cookie</b> option multiple times to set 
more than a cookie. Another parameter that sometimes may change the 
target site behaviour is the Referer header. It is possbile to set it 
using the <b>referer <i>url</i></b> option.
<h3>Testing virtual domains</h3>
Sometimes after you configured a new virtual domain in your web server 
it's handly to check if everything is working as it should before DNS 
propagation and without the need to alter by hand your /etc/hosts. You 
can use the <b>host <i>name</i></b> wbox option in order to do this.
<pre>% <b>wbox 64.233.183.99 host <a href="http://www.google.it">www.google.it</a> dump</b>
WBOX <a href="http://www.google.it">www.google.it</a> (64.233.183.99) port 80
HTTP/1.1 200 OK
Cache-Control: private
Content-Type: text/html; charset=UTF-8
Set-Cookie: PREF=ID=81ccf41f3364e7fa:TM=1181063780:LM=1181063780:S=yd7lfPo-SBZTngH_; expires=Sun, 17-Jan-2038 19:14:07 GMT; path=/; domain=.<a href="http://google.it">google.it</a>
Server: GWS/2.1
Transfer-Encoding: chunked
Date: Tue, 05 Jun 2007 17:16:20 GMT

d6f
<html>... (data omitted)
</pre>
<h3>Testing HTTP compression</h3>
Using the <b>compr</b> option wbox will include the right headers to 
accept the gzip/deflate data encoding. If the web server supports 
compression it will reply with compressed data, and wbox will show this 
fact in the reply status lines.
<pre>% <b>wbox <a href="http://antirez.com">antirez.com</a> compr</b>
WBOX <a href="http://antirez.com">antirez.com</a> (62.149.233.114) port 80 [compr]
0. 200 OK    6553 bytes    837 ms    compr
1. 200 OK    6553 bytes    301 ms    compr
user terminated
</pre>
It is sometimes interesting to check the time required to complete the 
request with and without the HTTP compression just including/excluding 
the <b>compr</b> option from the command line.
<h3>A few more options</h3>
<ul><li><b>http10</b> tells wbox to use HTTP/1.0 in the request, avoiding chunked data output.</li><li><b>head</b> forces wbox to use the HEAD method instead of GET.</li><li><b>close</b> close the connection after few bytes received. Handy to
 check just latency against big pages/files, to check what happens on 
end of file from client conditions, ...</li><li>Use <b>wbox -h</b> in order to list all the available options.</li></ul>
<h2>HTTP server mode</h2>
Wbox implements a minimal configuration-less HTTP server. It is useful 
as a quick way to share files in a LAN or via Internet, so it implements
 directory listing by default. In order to start this mode just use the 
following command line (assuming documents you want to share are in <i>/tmp/mydocs</i>):
<pre>% <b>wbox servermode webroot /tmp/mydocs</b>
</pre>
Than play with it conneting to <a href="http://127.0.0.1:8081">http://127.0.0.1:8081</a> since Wbox for default uses the <b>8081</b> TCP port. You can change this default using the <b>serverport <i>port</i></b> option.
<h3>limiting number of clients in server mode</h3>
The default in wbox server mode is to handle up to 20 concurrent clients. It is possible to tune this parameter using <b>maxclients <i>number</i></b>.
 When a clinet connects but there are already the maximum number of 
clients connected the connection is closed before to spawn a child. This
 allows for an easy DOS attack so the behaviour should be changed in two
 ways: a timeout should be implemented in order to avoid that a client 
can hang forever, also on a new connection that can't be server because 
too many clients are already connected a random old connection should be
 killed.
