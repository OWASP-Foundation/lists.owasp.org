<p class="MsoNormal"><span style="font-family:"Courier New"">Hello Hiranmayi,<br>
<br>
On Tue, Mar 20, 2018 at 02:19:50PM +0000, Hiranmayi Palanki wrote:<br>
> Having recently tried out ModSecurity, it appears that lower paranoia levels (Basic, PL1, PL2, PL3) are not fully blocking all flavors of XSS and SQL injection.<br>
> <br>
> Encoded XSS attacks are bypassing the CRS rules. Example below:<br>
> <a href="http://localhost:8082/xss2?uid=%3Balert%281%29%3B">http://localhost:8082/xss2?uid=%3Balert%281%29%3B</a><br>
<br>
This results in a 3 rule alerts in CRS 3.0.2 for me:<br>
942110 PL2 SQL Injection Attack: Common Injection Testing Detected<br>
920273 PL4 Invalid character in request (outside of very strict set)<br>
942432 PL4 Restricted SQL Character Anomaly Detection (args): # of special characters exceeded (2)<br>
<br>
> Similarly for SQLi, the below pattern is getting through.<br>
> <br>
> <a href="http://localhost:8082/sqli2?uid=3-2">http://localhost:8082/sqli2?uid=3-2</a><br>
<br>
What's wrong with this parameter. "3-2" is by no means an SQL Injection. The<br>
very fact that a security scanner is launching a request does not make it an<br>
attack.<br>
<br>
> When the Paranoia Level is set to PL4, the encoded XSS attack is blocked,<br>
> however it crashes the application. <br>
<br>
I am not sure what you mean by "crashes the application". You are probably<br>
meaning to say that the application is no longer working properly with PL4.<br>
That is expected behaviour at this high paranoia level. What you are facing<br>
are false positives, that you need to tune away.<br>
<br>
There is a tutorial dedicated to this work over at <a href="https://isolate.menlosecurity.com/1/3735928037/https:/www.netnea.com">https://www.netnea.com</a><br>
<br>
> Is it possible to selectively add custom rules (for encoded XSS attacks) to<br>
> the basic level or other lower levels that do not adversely impact the<br>
> application?<br>
<br>
One of the criterias for putting rules in the higher paranoia levels is<br>
there tendency to cause false positives. If they were less likely to have this<br>
effect, we would shift them to a lower PL.<br>
<br>
With that being said: No, you can not easily move rules around even if the<br>
false positives were a non-issue.<br>
