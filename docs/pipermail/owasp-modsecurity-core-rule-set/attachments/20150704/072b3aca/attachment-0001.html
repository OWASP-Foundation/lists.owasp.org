<p class="MsoNormal"><span style="font-size:11.0pt;font-family:"Calibri","sans-serif";color:#1F497D">ModSecurity will indeed will block HTML tags because they are exactly how HTML injection/XSS is introduced. Sending HTML is often indicative of these vulnerabilities
 being exploited, in general it is considered poor form/dev to send such content across the wire. However, you can always, easily create exceptions for rules in particular places. Just look at the particular rule that is triggering and the particular parameter
 that is triggering them and you can add an exception by doing something similar to the following ‘</span><span style="font-size:10.5pt;font-family:"Helvetica","sans-serif";color:#333333;background:white">SecRuleUpdateTargetById 950907 !ARGS:test’</span><span style="font-size:11.0pt;font-family:"Calibri","sans-serif";color:#1F497D">.
 For more information see </span><a href="https://www.trustwave.com/Resources/SpiderLabs-Blog/ModSecurity-Advanced-Topic-of-the-Week--(Updated)-Exception-Handling/"><span style="font-size:10.5pt;font-family:"Helvetica","sans-serif";color:#4078C0;background:white;text-decoration:none">https://www.trustwave.com/Resources/SpiderLabs-Blog/ModSecurity-Advanced-Topic-of-the-Week--(Updated)-Exception-Handling/</span></a>.
<span style="font-size:11.0pt;font-family:"Calibri","sans-serif";color:#1F497D">In addition, if you are using CRS 3.0 this feature is already built in see your config file:
</span><a href="https://github.com/SpiderLabs/owasp-modsecurity-crs/blob/v3.0.0-dev/modsecurity_crs_10_setup.conf.example#L420"><span style="font-size:10.5pt;font-family:"Helvetica","sans-serif";color:#4078C0;background:white">https://github.com/SpiderLabs/owasp-modsecurity-crs/blob/v3.0.0-dev/modsecurity_crs_10_setup.conf.example#L420</span></a>.<o:p></o:p></p>

<p class="MsoNormal" style="margin-bottom:12.0pt">it seems to me that ModSecurity is 'rightly' detecting XSS attempts due to the HTML tags that you submit when creating blog posts. As far as I know, what you would need to do is create exceptions for the URLs
 and parameters you know contain HTML tags, and if possible make them specific to the tags that your site allows. This is a tedious (for a lack of a better word) work that comes with the installation of ModSecurity in front of a new application. Think that
 the default CRS is generic and doesn't understand the idiosyncrasy of your application, so the rules need adjusting.<o:p></o:p></p>
