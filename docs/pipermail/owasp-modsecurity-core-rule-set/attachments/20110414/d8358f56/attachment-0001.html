Subject: CRS Directory Format Question

The current OWASP CRS archive has a number of directories that hold different rules -

&nbsp;* &nbsp; base_rules
&nbsp;* &nbsp; optional_rules
&nbsp;* &nbsp; slr_rules
&nbsp;* &nbsp; experimental_rules

I am thinking that most ModSecurity users want to use Apache Include wild-carding when activating rulesets -

&lt;IfModule security2_module&gt;
&nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; Include conf/modsecurity_crs/*.conf
&nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; Include conf/modsecurity_crs/base_rules/*.conf
&lt;/IfModule&gt;

While this is certainly convenient, this does cause a problem. &nbsp;The various rules files have a numbering scheme whose purpose to to help ensure that the rules file are executed in the proper order when wild-carding with includes. &nbsp;Activating these rules are challenging when separated into the different directories.

&lt;IfModule security2_module&gt;
&nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; Include conf/modsecurity_crs/*.conf
&nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; Include conf/modsecurity_crs/base_rules/*.conf
&nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; Include conf/modsecurity_crs/optional_rules/*.conf

&lt;/IfModule&gt;

So, what I am thinking is that we should add an empty directory called -

&nbsp;* &nbsp; activated_rules

The sole purpose of this directory would be for the local Admin to copy all files that they want to run into that one directory. &nbsp;When they do this, then the file name numbering scheme will work and it will allow for easier Include wild-carding -

&lt;IfModule security2_module&gt;
&nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; Include conf/modsecurity_crs/*.conf
&nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; Include conf/modsecurity_crs/activated_rules/*.conf
&lt;/IfModule&gt;

How does this approach sound to everyone?

-Ryan
