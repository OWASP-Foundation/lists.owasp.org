<div>
<div>I will investigate the idea of a rule mapping, but keep in mind that following -</div>
</div>
</div>
<ol>
<li>One of our goals with the update is to make the rule accuracy better (less false positives).  So, just because a specific rule is currently a false positive and you chose to add an exception to disable it, that does not mean that you would need to keep
 the exception with the new version of the rule.</li><li>ModSecurity v2.7 (will be released very soon) has the new "accuracy" and "maturity" actions.  This will allow us (Trustwave) to give a relative score for each of these settings.  This will help you to decide which rules to run default or not.</li><li>As for directory structure groupings, it is my belief that all rules in the "base_rules" directory should have a very high quality of accuracy (low FP rate).  If a user wants to get more aggressive with detection, then they will be able to activate other
 rule files under the "experimental_rules" directory in which the accuracy/maturity values would be lower.</li></ol>
<div><br>
</div>
<div>-Ryan</div>

<p class="MsoNormal"><font size="1" color="black" face="Calibri"><span style="font-size:7.0pt;font-family:Calibri;color:black">One of the items we are considering with the CRS v3.0 update is to start from scratch with all of the rule IDs.  We have a defined
 range reserved for the OWASP CRS - </span></font></p>
</div>
<div>
<p class="MsoNormal"><font size="1" color="black" face="Calibri"><span style="font-size:7.0pt;font-family:Calibri;color:black"><a href="https://sourceforge.net/apps/mediawiki/mod-security/index.php?title=Reference_Manual#id">https://sourceforge.net/apps/mediawiki/mod-security/index.php?title=Reference_Manual#id</a></span></font></p>
</div>
<div>
<p class="MsoNormal"><font size="1" color="black" face="Calibri"><span style="font-size:7.0pt;font-family:Calibri;color:black"> </span></font></p>
</div>
<div><span style="text-align:-webkit-auto;float:none;word-spacing:0px">
<p class="MsoNormal"><font size="1" color="black" face="Arial"><span style="font-size:6.5pt;font-family:Arial;color:black;background:white">900,000–999,999: reserved for the OWASP ModSecurity Core Rule Set </span></font></p>
</span><font size="1" color="black" face="Calibri"><span style="font-size:7.0pt;font-family:Calibri;color:black"><a href="http://www.owasp.org/index.php/Category:OWASP_ModSecurity_Core_Rule_Set_Project" title="http://www.owasp.org/index.php/Category:OWASP_ModSecurity_Core_Rule_Set_Project" style="text-align:-webkit-auto;background-repeat:no-repeat;word-spacing:0px"><font size="1" color="#3366bb" face="Arial"><span style="font-size:6.5pt;font-family:Arial;color:#3366bb;background:white;text-decoration:none">[12]</span></font></a></span></font><font size="1" color="black" face="Arial"><span style="font-size:6.5pt;font-family:Arial;color:black;background:white"><span style="text-align:-webkit-auto;float:none;word-spacing:0px"> project</span></span></font><font size="1" color="black" face="Calibri"><span style="font-size:7.0pt;font-family:Calibri;color:black"></span></font></div>
<div>
<div>
<p class="MsoNormal"><font size="1" color="black" face="Calibri"><span style="font-size:7.0pt;font-family:Calibri;color:black"> </span></font></p>
</div>
<div>
<p class="MsoNormal"><font size="1" color="black" face="Calibri"><span style="font-size:7.0pt;font-family:Calibri;color:black">The rationale for this change is to make it easier to users to do exceptions and disable groups ofrules by using SecRuleRemoveById,
 etc… That directive has the capability to remove <b><span style="font-weight:bold">ranges</span></b> of rule IDs, however over the years the rule IDs were added ad-hoc and they are not properly grouped in this manner.  For instance, groupings could be something
 similar to this -</span></font></p>
</div>
<div>
<p class="MsoNormal"><font size="1" color="black" face="Calibri"><span style="font-size:7.0pt;font-family:Calibri;color:black"> </span></font></p>
</div>
<ul type="disc">
<li class="MsoNormal" style="color:black"><font size="1" color="black" face="Calibri"><span style="font-size:7.0pt;font-family:Calibri">Protocol Violation – 900000-900099</span></font>
</li><li class="MsoNormal" style="color:black"><font size="1" color="black" face="Calibri"><span style="font-size:7.0pt;font-family:Calibri">Protocol Policies - 900100–900199</span></font>
</li><li class="MsoNormal" style="color:black"><font size="1" color="black" face="Calibri"><span style="font-size:7.0pt;font-family:Calibri">Cross-site Scripting - 900200–900299</span></font>
</li><li class="MsoNormal" style="color:black"><font size="1" color="black" face="Calibri"><span style="font-size:7.0pt;font-family:Calibri">…</span></font></li></ul>
<div>
<p class="MsoNormal"><font size="1" color="black" face="Calibri"><span style="font-size:7.0pt;font-family:Calibri;color:black">We would give each rule group approx 100 rule IDs.  The initial number would be less than thatwhich would allow some growth.  After
 some analysis of the current rules, we really shouldn't ever exceed those allotments as there ends up being rule logic duplication and you just end up with bloated rule amounts.</span></font></p>
</div>
<div>
<p class="MsoNormal"><font size="1" color="black" face="Calibri"><span style="font-size:7.0pt;font-family:Calibri;color:black"> </span></font></p>
</div>
<div>
<p class="MsoNormal"><font size="1" color="black" face="Calibri"><span style="font-size:7.0pt;font-family:Calibri;color:black">Keep in mind that the active rule IDs were going to change a bit anyways with this update as we will be consolidating many rules into
 highly optimized regexes, etc..  </span></font></p>
</div>
<div>
<p class="MsoNormal"><font size="1" color="black" face="Calibri"><span style="font-size:7.0pt;font-family:Calibri;color:black"> </span></font></p>
</div>
<div>
<p class="MsoNormal"><font size="1" color="black" face="Calibri"><span style="font-size:7.0pt;font-family:Calibri;color:black">The only concern that I have with making this move would be if users have extensively implemented exceptions based on the current
 rule IDs.  They would need to be aware of this when they migrate to v3.0.</span></font></p>
</div>
<div>
