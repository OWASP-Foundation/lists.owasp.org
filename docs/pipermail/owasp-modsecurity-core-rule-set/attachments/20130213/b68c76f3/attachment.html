<div>I actually like both ideas.  I think having the @detectSQLi operator with the path the fingerprints.txt file is a good approach as it would allow the rule itself to stay the same but the fingerprints.txt file can simply be update as new payloads are captured.</div>
</div>
</div>
<div><br>
</div>
<div>As for adding a new t:normalizeSQLi transformation function, I think it could have value, however we would need to revalidate/update all of the other SQLi rules to account for the normalizing that libinjection would do.  I am not saying that this is a
 bad thing, just needed.</div>
<div><br>
</div>
<div>I think adding libinjection code in any capacity would be a great step forward.</div>
