<div>Yes, these rule work by first checking various TARGETS for possible SQLi keywords (ARGS_NAMES is one of the var collections checked)-</div>
<div><br>
</div>
<div>SecRule REQUEST_COOKIES|REQUEST_COOKIES_NAMES|REQUEST_FILENAME|ARGS_NAMES|ARGS|XML:/* "@pm select show top distinct from dual where group by order having limit offset union rownum as (case" "phase:2,id:'981300',t:none,t:urlDecodeUni,t:lowercase,nolog,pass,nolog,setvar:'tx.sqli_select_statement=%{tx.sqli_select_statement}
 %{matched_var}'"</div>
<div><br>
</div>
<div>If there is a match, that data is appended to the TX.SQLI_SELECT_STATEMENT value which is then checked against the other individual rules and anomaly scores are then increases.</div>
<div><br>
</div>
<div>The final rule in this group simply checks to see if there are 3 or more SQLi keyword matches within the TX value.</div>
