<p class="MsoNormal"><span lang="EN-US" style="font-family:"Arial","sans-serif"">I noticed that the version 3 of CRS ruleset sometimes transforms the rule phase: from
<b>phase:1</b> (v2.2.9) to <b>phase:request</b>. But <b>phase:request</b> just translates to
<b>phase:2</b> (request body) and, if I understand the phase processing correctly, requests that do not contain a body will not be checked against
<b>phase:2</b> rules. (Well, this is what I observe in my debugging logs. Don’t know if it is the ModSecurity standard behavior or if it comes from my implementation.)<o:p></o:p></span></p>
<p class="MsoNormal"><span lang="EN-US" style="font-family:"Arial","sans-serif""><o:p> </o:p></span></p>
<p class="MsoNormal"><span lang="EN-US" style="font-family:"Arial","sans-serif"">A typical example is the rule id 960911 (REQUEST-20-PROTOCOL-ENFORCEMENT) which validates the HTTP request line: it seems to me that it should be validated for requests including
 those without body, but the phase is now <b>phase:request</b> on CRS v3.<o:p></o:p></span></p>
<p class="MsoNormal"><span lang="EN-US" style="font-family:"Arial","sans-serif""><o:p> </o:p></span></p>
<p class="MsoNormal"><span lang="EN-US" style="font-family:"Arial","sans-serif"">Besides, there are a lot of other examples in REQUEST-21-PROTOCOL-ATTACK where it seems to me that the rules should be put on
<b>phase:1</b>. (E.g. smuggling, response splitting.)<o:p></o:p></span></p>
<p class="MsoNormal"><span lang="EN-US" style="font-family:"Arial","sans-serif"">The same applies to REQUEST-10-IP-REPUTATION.<o:p></o:p></span></p>
