<div><br>
</div>
<div>The "TAOP01U10RhvxuuKxxempg01U10monamzy1" string is NOT detected as SQL injection, but simply</div>
<div>the name of the PARAMETER that contains the malicious data. The data that triggers your rule</div>
<div>is</div>
<div>       "7 Andr"</div>
<div><br>
</div>
<div>which you can see in the   [data "7 Andr"]   part of the rule message.</div>
<div><br>
</div>
<div>The problem is probably, that the regex of the rule has something like</div>
<div><br>
</div>
<div>     ...|div|like|between|and)\\s*...</div>
<div><br>
</div>
<div>which says "something with 'and' followed by ANY number of whitespaces". Unfortunately "ANY"</div>
<div>also includes 0. So "andr" matches "and" followed by 0 whitespaces and the "r" probably matches</div>
<div>some of the rest of the complex regex.</div>
<div><br>
</div>
<div>I hope that makes sence :-)</div>
