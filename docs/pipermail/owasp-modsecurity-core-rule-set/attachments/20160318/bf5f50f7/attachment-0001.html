<p class=""><b>Sending alert emails</b></p>
<p class="">As an example, suppose that we wanted to execute a script to email us an alert message whenever an attempted SQL injection exploit was detected. To do this, we need two things:</p>
<p class="">1. A script file that has the ability to email an alert to a specified email address.</p>
<p class="">2. A rule that will invoke the email script when a rule match is detected.</p>
<p class="">For the script, we will use a standard shell script that invokes <span class="">/bin/sh</span>, though we could have easily used Perl or any other scripting language. We will email the alert to <span class=""><a href="mailto:user@example.com">user@example.com</a></span>. </p><p class="">Create a file named <span class="">email.sh </span>in the directory <span class="">/usr/local/bin </span>and type the following in it:</p><p class="">#!/bin/sh</p><p class="">echo "An SQL injection attempt was blocked" | mail –s "ModSecurity Alert" <a href="mailto:user@example.com">user@example.com</a></p><p class="">echo Done.</p><p class="">The script invokes the <span class="">mail </span>binary to send an email with the subject <b>ModSecurity Alert </b>to <span class=""><a href="mailto:user@example.com">user@example.com</a></span>. The last line of the script writes the string <b>Done. </b>to <span class="">stdout</span>. This is so that ModSecurity will recognize that the script has executed successfully. </p><p class="">We now have to make the script executable so that it can be invoked when a rule matches:</p><p class="">$ chmod a+rx /usr/local/bin/email.sh</p><p class="">Now all that is left is to create a rule that will trigger the alert script:</p><p class="">SecRule ARGS "drop table" "deny,exec:/usr/local/bin/email.sh"</p><p class="">
















</p><p class="">You can now test out this rule by attempting to access <span class=""><a href="http://yourserver/?test=drop%20table">http://yourserver/?test=drop%20table</a></span>. If you've substituted your own email address in the example above you should get an email telling you that an SQL injection attempt has just been blocked.</p><p class="">







</p><p class="">Receiving such an email can be useful to quickly be alerted of any ongoing attacks. However, what if we wanted the email to contain a little more information on the attempted exploit; would that be possible? Yes, it's not only possible, it's also a very good idea, since more information about an alert can allow us to decide whether it is something to investigate more in-depth (such as when we detect that it's not just an automated vulnerability scanner pounding away at our server but actually a hacker probing for weaknesses with manually crafted exploit URLs). </p><p class=""><b>Sending more detailed alert emails</b></p><p class="">ModSecurity allows us to set environment variables via the <span class="">setenv </span>action. By populating environment variables with suitable data we can record more information about the request that was blocked. </p><p class="">Suppose we would like to gather the following data when an attempted SQL injection is detected: </p><p class="">The hostname of the server where the alert occurred</p><p class="">The remote user's IP address and hostname</p><p class="">The full request URI</p><p class="">The values of all arguments, whether they were sent using the <span class="">GET </span>or <span class="">POST </span>method</p><p class="">The unique ID for the request, so we can find this alert in the log files</p><p class="">We will place this information in six separate environment variables, which we will call <span class="">HOSTNAME</span>, <span class="">REMOTEIP</span>, <span class="">REMOTEHOST</span>, <span class="">REQUESTURI</span>, <span class="">ARGS</span>, and <span class="">UNIQUEID</span>. Our modified rule now looks like this: </p><p class="">SecRule ARGS "drop table" "deny,t:lowercase, <span class=""> </span>setenv:HOSTNAME=%{SERVER_NAME}, <span class=""> </span>setenv:REMOTEIP=%{REMOTE_ADDR}, <span class=""> </span>setenv:REQUESTURI=%{REQUEST_URI}, <span class=""> </span>setenv:ARGS=%{ARGS}, <span class=""> </span>setenv:UNIQUEID={%UNIQUE_ID}, <span class=""> </span>exec:/usr/local/bin/email.sh"</p><p class="">Now all we have to do is modify the email script so that it places the environment variables in the email body:</p><p class="">#!/bin/sh</p><p class="">echo "</p><p class="">An SQL injection attempt was blocked:</p><p class="">Server: $HOSTNAME</p><p class="">Attacking IP: $REMOTEIP</p><p class="">Attacking host: $REMOTEHOST</p><p class="">Request URI: $REQUESTURI</p><p class="">Arguments: $ARGS</p><p class="">Unique ID: $UNIQUEID</p><p class="">Time: `date '+%D %H:%M'`</p><p class="">" | mail –s 'ModSecurity Alert' <a href="mailto:user@example.com">user@example.com</a></p><p class="">





























</p><p class="">Echo Done. </p><p class="">As you can see, we use a multi-line <span class="">echo </span>statement to get all the information nicely formatted. Since this is a shell script, it will replace <span class="">$HOSTNAME </span>and the other environment variables with the value we set the variables to in our ModSecurity rule. The last line of the <span class="">echo </span>statement also adds a timestamp with today's date and the current time by invoking the <span class="">date </span>command and placing backticks (<span class="">`</span>) around it, which causes the shell to execute the command and substitute the command's output for it. Finally, the data is piped into the <span class="">mail </span>binary, which sends an email with the subject line <b>ModSecurity Alert </b>to the specified email address.</p><p class="">








</p><p class="">Again, at the end of the script we make sure to echo a dummy text to <span class="">stdout </span>to make ModSecurity happy. If you test this script you should get a nicely formatted email with all of the attacker's details.</p><p class=""><br></p><p class="">From: ModSecurity 2.5 Securing your Apache installation and web applications<br></p></div><div class="gmail_extra"><br><div class="gmail_quote">2016-03-18 12:00 GMT+00:00  <span dir="ltr"><<a href="mailto:owasp-modsecurity-core-rule-set-request@lists.owasp.org" target="_blank">owasp-modsecurity-core-rule-set-request@lists.owasp.org</a>></span>:<br><blockquote class="gmail_quote" style="margin:0px 0px 0px 0.8ex;border-left-width:1px;border-left-color:rgb(204,204,204);border-left-style:solid;padding-left:1ex">Send Owasp-modsecurity-core-rule-set mailing list submissions to<br>
