<html><head></head><body style="word-wrap: break-word; -webkit-nbsp-mode: space; -webkit-line-break: after-white-space; color: rgb(0, 0, 0); font-size: 14px; font-family: Calibri, sans-serif; "><div><div><div>Please forgive the cross postings, but I wanted to make sure that all relevant parties were informed of this update. &nbsp;I have begun the process of implementing the OWASP AppSensor Detection Points (<a href="https://www.owasp.org/index.php/AppSensor_DetectionPoints">https://www.owasp.org/index.php/AppSensor_DetectionPoints</a>) within the OWASP ModSecurity Core Rule Set (<a href="https://www.owasp.org/index.php/Category:OWASP_ModSecurity_Core_Rule_Set_Project">https://www.owasp.org/index.php/Category:OWASP_ModSecurity_Core_Rule_Set_Project</a>). &nbsp;</div><div><br></div><div><b>I am pleased to announce that I have just made an update to the OWASP CRS SVN repository that fully implements the Request Exception (RE) category</b> -&nbsp;<a href="https://www.owasp.org/index.php/AppSensor_DetectionPoints#RequestException">https://www.owasp.org/index.php/AppSensor_DetectionPoints#RequestException</a>. &nbsp;See the following blog post for more details -&nbsp;<a href="http://blog.spiderlabs.com/2011/08/implementing-appsensor-detection-points-in-modsecurity.html">http://blog.spiderlabs.com/2011/08/implementing-appsensor-detection-points-in-modsecurity.html</a></div><div><br></div><div>The major change in this version vs. the earlier one outlined in this blog post (<a href="http://blog.spiderlabs.com/2011/02/modsecurity-advanced-topic-of-the-week-real-time-application-profiling.html">http://blog.spiderlabs.com/2011/02/modsecurity-advanced-topic-of-the-week-real-time-application-profiling.html</a>)&nbsp;is that both the profiling and detection logic has been moved to Lua scripts. &nbsp;With the increased logic capabilities of Lua, we are now able to more accurately profile the application in real-time by analyzing traffic and automatically generating profiles for the following resource characteristics -</div><ul><li>Enforcing the expected Request Method(s)</li><li>Enforce the number of expected parameters (min-max range)</li><li>Enforce parameter names&nbsp;</li><li>Enforce parameter lengths (min-max range)</li><li>Enforce Character Classes</li><ul><li>Flag (e.g. - /path/to/foo.php?param)</li><li>Digits&nbsp;&nbsp;(e.g. - /path/to/foo.php?param=1234)&nbsp;</li><li>Alpha&nbsp;&nbsp;(e.g. - /path/to/foo.php?param=abcd)</li><li>AlphaNumeric&nbsp;&nbsp;(e.g. - /path/to/foo.php?param=abcd1234)</li><li>Email&nbsp;&nbsp;(e.g. - /path/to/foo.php?param=foo@bar.com)</li><li>Path&nbsp;&nbsp;(e.g. - /path/to/foo.php?param=/dir/somefile.txt)</li><li>URL&nbsp;&nbsp;(e.g. - /path/to/foo.php?param=http://somehost/dir/file.txt)</li><li>SafeText&nbsp;&nbsp;(e.g. - /path/to/foo.php?param=some_data-12)</li></ul></ul><div>The updated rules files are in the /experimental_rules directory -&nbsp;<a href="http://mod-security.svn.sourceforge.net/viewvc/mod-security/crs/trunk/experimental_rules/">http://mod-security.svn.sourceforge.net/viewvc/mod-security/crs/trunk/experimental_rules/</a></div><div>Look in the /lua folder to find the 2 scripts -&nbsp;<a href="http://mod-security.svn.sourceforge.net/viewvc/mod-security/crs/trunk/lua/">http://mod-security.svn.sourceforge.net/viewvc/mod-security/crs/trunk/lua/</a></div><div><br></div><div>I encourage people to test out these new rules and to report back their experiences &#8211; both good and bad.</div><div><br></div><div>FYI &#8211; I also wanted to thank Josh Zlatin for assisting with the initial Lua script creation.</div><div><br></div><div>Cheers.</div><div><font face="Calibri,Verdana,Helvetica,Arial"><span style="font-size:11pt"><br></span></font><font size="2"><font face="Consolas,Courier New,Courier"><span style="font-size:10pt">--<br>
