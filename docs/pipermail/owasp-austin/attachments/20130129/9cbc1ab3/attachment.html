<br>
*Topic:* Data events, or why security is cloudier than you think.<br>
<br>
*Abstract:* Data security doesn't involve just securing data at rest or<br>
in transit. It also needs to be secured in use ? which means that at any<br>
point, the characteristics of the data can change. We call this<br>
situation a "data event," and it can mean that security requirements<br>
have to change as a result.<br>
<br>
This is not the same thing as logging event data; this is taking into<br>
account changes in the combination, use or business context surrounding<br>
specific data. For example, a press release is confidential and requires<br>
a certain set of security policies in the areas of access control, DLP,<br>
key management (if encryption is involved), and so on. But once the<br>
business event occurs, the press release suddenly becomes the opposite<br>
of confidential, and all the policies have to change immediately as a<br>
result.<br>
<br>
Data events can also occur when data elements are combined in particular<br>
ways so that they become covered by regulations. A query might produce a<br>
small enough sample size that it needs to be treated as protected<br>
information, or a doctor becomes a patient so that her name is now<br>
protected by HIPAA. Data events are often tied closely to the business<br>
context, and as such can mirror transactions and workflows.<br>
<br>
Data events are important because traditional security policies have<br>
been applied to the current container of the data: this database is<br>
confidential because some rows are confidential, or this Word document<br>
requires access control (but its content can be copied and pasted<br>
somewhere else). Container-centric security is too static for today's<br>
high-speed, big-data, cloud-based (pick as many buzzwords as you like)<br>
processing.<br>
<br>
This talk will describe the concept of data events, and will invite<br>
audience discussion on how security controls can be adapted to them.<br>
