You are on the right track to question the handling of passwords. I
assume your company (or client) has a policy on the management of
computer
passwords (change at least 1 per year, except when security exception
signed off by a responsible person (preferably a C-level/VP) is
provided each year). <br>
<br>
As a matter of security policy,&nbsp; passwords should never be stored in
code:<br>
&nbsp;(1) difficult to change passwords (it will be a change request/release
for every change, which will be part of the on-going cost)<br>
&nbsp;(2) no separation of duties (every programmer that has access to the
CVS will know the production password for that application)<br>
<br>
In addition, the level of access by the application that is granted
should be the minimum necessary to perform the function, i.e., no
administrator access account or group, and an user name/password unique
to the application within a group of application accounts that have
similar needs of access.<br>
<br>
I assume that the applications you are working with are batch processes
(the following would be different if part of a web service). The only
alternative is either store in a "start-up" or "ini" file or an elegant
scheme, the latter causes security people to be not taken very
seriously there-after. Depending on your risk posture (how much will it
cost the company if the security element is breached), you might
consider the following to secure the credentials:<br>
&nbsp;(1) if stored in a file, the file should not be on the same drive as
the application. You might want this file as part of a directory
protected by the system, with limited access to it.<br>
&nbsp;(2) the password should be stored in encrypted form (symmetric key
greater than 128 bits). I have seen applications that read the file,
and, upon detecting that the password is in clear-text, rewrites the
file with the encrypted password, where the key being stored is in the
keystore (either of the application servers or an enterprise solution).<br>
<br>
And then there are the more elegant ideas, such as using the kerberos
ticket of the signed-in user initiating the process as a "trust" to
access the database (as in Windows networking). This assumes that it is
a user initiated process. If it is a "batch" process timed to run in
the middle of the night, then you might use the "key" of the "keystore"
of the application to decrypt an attribute value found in the (Active)
Directory of an account assigned to the script/user.<br>
<br>
The best advice I can give is that the solution you come up with is
cost-effective: it will cost the enterprise no more than the value of
the asset to the enterprise.<br>
