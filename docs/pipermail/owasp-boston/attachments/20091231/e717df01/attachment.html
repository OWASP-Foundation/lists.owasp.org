<font size="-1"><font face="Verdana">LB:<br>
I call it the "Turtle All The Way" problem. No matter where you put it
(in a Java keystore, for example), you will still need a password
either hardcoded in your software, or left in the clear on disk.<br>
<br>
Here are some solutions. See which fits your risk tolerance and budget.<br>
<br>
First, commercial and expensive: <br>
<a class="moz-txt-link-freetext" href="http://www.cyber-ark.com/identity-access-management-solutions/application-password-management.asp">http://www.cyber-ark.com/identity-access-management-solutions/application-password-management.asp</a><br>
</font></font><font size="-1"><font face="Verdana"><br>
This works well<br>
<br>
</font></font><font size="-1"><font face="Verdana">There may be others,
I am not familiar with any other product.<br>
<br>
Hard-coding passwords is generally not a good idea. If you must use
that approach, set the filesystem permissions so that not everyone can
read the script containing the password. (for compiled programs,
reading the password is not trivial, but it makes changing the password
very difficult)<br>
<br>
NEVER (assuming this is some *nix) supply passwords to a script as a
command-line argument (then a ps -aef might show the password)<br>
<br>
My preferred, cheap (read: free) approach is below. Note that this is
NOT cryptographically secure, and will give up the password to someone
who can follow the code (hence the file permission)<br>
</font></font><font size="-1"><font face="Verdana"><br>
I know this isn't foolproof, but better than clear-text pw on disk.<br>
<br>
</font></font><font size="-1"><font face="Verdana">1. Put the password
in a file. Encrypt it (some strong cipher) with a key. Do not store the
key anywhere on that machine--remember it or write it down and put it
in a safe. Naturally, back up the password file as well.<br>
(if you have more than one application or script, you can put each
password in the file)<br>
<br>
2. Write another program. At each system boot, it will prompt the
sysadmin for the key, decrypt the password file, and store the
passwords in an obfuscated form in shared memory<br>
<br>
3. Your scripts (that talk to the db) need to know how to de-obfuscate
the password, and where to look for it in shared memory.<br>
<br>
Good: No password in clear-text--you pass the audit<br>
Bad: If the system goes down, it will require human intervention (but
for only one key that will unlock all the passwords)<br>
Bad: People reading the script will know where to look for the
obfuscated PW and how to make it readable.<br>
<br>
Happy New Year!<br>

do not know of as better solution.
