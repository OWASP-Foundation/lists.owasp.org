Since attending OWASP Boston in November, I have been wondering if we<br>
are missing an opportunity to try and define an access-control header<br>
that can do more to combat xss.<br>
<br>
<br>
Specifically, access-control-allow-origin seems limited in scope<br>
primarily to xmlhttprequest, or at most to a DOM object (though if I<br>
am reading this spec incorrectly, call me out  :-). It also seems to<br>
be focused on incoming xsrf, and to leave outgoing xss alone. Perhaps<br>
another header, with two modifications, might make it much easier to<br>
contain outbound xss. It could:<br>
<br>
1) allow a page to provide a whitelist of authorized URIs which the<br>
page&#39;s DOM may request (the whole DOM, not just a particular object).<br>
<br>
For example, &quot;access-control-authorized-request-uri&quot; could define an<br>
&#39;origin list&#39; of URIs that the page might possibly request. Any<br>
request the DOM makes outside this origin list would be considered<br>
malicious and would be dropped by the browser. This would allow<br>
<a href="http://secure.banksite.com" target="_blank">secure.banksite.com</a> to make a promise to the browser that its pages<br>
will never, ever request anything from anywhere other than<br>
<a href="http://secure.banksite.com" target="_blank">secure.banksite.com</a>, <a href="http://www.banksite.com" target="_blank">www.banksite.com</a>, and <a href="http://banksite.affiliate.cdn.com" target="_blank">banksite.affiliate.cdn.com</a><br>

<br>
2) extend the origin controls granted to xmlhttprequest via<br>
&quot;access-control-allow-origin&quot;, and to flash via &quot;allow-access-from,&quot;<br>
and to other individual object types, UP one level to the entire page<br>
<br>
3) allow browsers to report requests outside of this whitelist back to<br>
the originating server<br>
<br>
This is half in jest, but half serious: a browser could automatically<br>
issue another http request indicating the problem URIs whenever a page<br>
tells it to make a request outside its whitelist. This would allow<br>
site administrators to detect xss, or (more often?) misconfigured URI<br>
whitelists.<br>
<br>
4) be optional; like Access-Control-Allow-Origin, would be optional<br>
and would default to allow-all<br>
<br>
5) not apply to traditional hyperlinks, unless the DOM contrives to<br>
execute them - HTTP referer links would be left alone<br>
<br>
<br>
What does everyone think? I am relatively new to OWASP and to web app<br>
sec, but I haven&#39;t been able to think myself out of the belief that we<br>
need something like this. It seems somewhat analogous to memory<br>
randomization for compiled code, in that it doesn&#39;t combat the root<br>
cause of insecure code, but it can offer a relatively low-cost control<br>
to which applications could opt-in. And in that many applications seem<br>
to be hurting without it.<br>
<br>
All comment and thoughts are welcome.<br>
