Hi Juan Carlos and the list, 

IMHO, configuration management at application and infraestructura criteria
impacts in a well done application. You can develop a very good application
but you must have it in a correct enviroment. For example, an incorrect
application server configuration, like the posibility of execute servelts by
class instead of mapping it in web.xml could allow execute admin or util
servelts, incorrect logging could bring protection law penalties, etc. in
other hand and not less important, technical infraestructure disclosure its
not a good pratice at all. All this questions have to be test in the
process.

In the guideline of the document these parts are separated but integrate in
the process of testing. The items are general (a web server banner is always
the same for every manufacturer. Sometimes in the testing process you must
adapt the checklist to the minor casualties of your auditing target, like
OS, web server manufacturer, that as you said automated tools identify them
very good, but not if the log its not activated so you have to review it by
hand. 

In this way, and taking your good suggestion we can group the different
issues into the configuration chapters by the part of the infraestructure
that could impact (Web Server, Application server, .. etc) and always
separating of the manufacturer or technology.

Here goes another suggestion

Do not mix "pure" web application issues with web application infrastructure
issues, this is do not mix SQL injection which is a purerly programming
(bad) practice with Web Server configuration issues like "HTTP Server
banner". I've always fight for this disctintion to be made.

In my oppinion it is very important to separate OS, HTTP Server, application
server (like a servlet engine) and application itself, they are different
things and different ways to test them. For example, you hardly will perform
a source review on a HTTP Server and only an automated blackbox testing for
the "web software"* (very innacurate), but, blackbox tools are very good for
http server issues (exploits) and source review for web application.

*Web Software = the web application itself, e.g. a portal made of some php
pages (liked this term from Mark)

What do you think people?
