A number of comments on the intro section...

1.	Following the discussion on where ASP, and JSP fit, I think that it is important to identify where the technologies were supposed to be used, rather than where they are being used.  Both ASP and JSP (and for that matter PHP) were designed as presentation layer technologies.  ASP was designed to work with COM components to do the application tier logic, JSP was designed to work with Java classes (aka Java Beans) and initially PHP wasn't designed to do any back end processing.  Before I get shot down for this comment, a lot has changed since then and these languages have advanced considerably, so perhaps they do belong in the Application tier.  My preference is to see ASP and JSP in the presentation tier and their associated technologies ie COM/COM+/Serviced Components and Java beans, in the application tier

2.	Section "What are Web Applications"
	
	Replace "developed at the Speed of Light" with "evolved at a staggering rate" - Speed of Light is a bit to SciFi for a professional document
	
	Replace "Java (or Similar Language)" with "an object-oriented language" - More generalist

	Replace "Through Complex Business logic tiers" with "Using complex business logic" - Use of tier is unnecessary


3.	Rearrangement of paragraphs - The "What are Web Services" section should be before the paragraph starting "It can help to think of a web application....".  A new section heading should be inserted after this section titled "3-Tiers of logic" or something similar.  Web Services really only differ from web applications in the presentation tier, instead of displaying data via a web browser they pass data via a remote method call.  Instead of being viewed by a human user (ie web application), web services are consumed by other applications/services.

	To this end, I would also reword the paragraph "It can help to think of a web application being made up of three logical tiers or functions" to "Most modern applcations can be divided into three logical tiers or functions".  The three tier separation is becoming more common in even desktop applications where the GUI is separated from the functional classes, which is separate from the data/persistence layer.

4.	Section "Presentation Tier"

	Replace "Presentation Tiers are responsible" with The presentation tier is responsible"

	Replace "It may also include application compoenets that create the page layout" with It may also include components that determine the presentation logic and create the page layout.

5.	Section "Data tier"

	Should be renamed "Persistence Tier" - The other two layers talk about "Presentation" (The way things are presented) and "Application" (The logic that brings the data together).  Following this format the word "Persisence" (The way data is stored) is more consistent than data (what is being stored).

6.	Section "Application Security Development Lifecycle"

	Ammend "Those tasks ... development and testing" to "Those tasks ... development, testing, deployment, maintenance and retirement" - The deployment and maintenance areas are especially important as it is these areas that security is of upmost importance as there is real data involved.  The omission of the last three sections in a lifecycle is one of the most common and unforgivable mistakes that any software engineer can make!!


Anyway, there is enough ramblings from me.  They are just my thoughts, so take what you want from it.

Cheers

Nick
