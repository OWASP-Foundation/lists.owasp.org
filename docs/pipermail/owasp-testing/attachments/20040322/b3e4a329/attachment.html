<p>Note: This sub-modules must be considered part of the main module “Internet Application
Testing” however, due to the particular issues related to the web deployment
and applications they are considered as a separate testing framework in itself). </p>


<h1>Web Application Testing</h1>


<p>
Security testing of web services is a crucial aspect of internet security
testing that must be tackled separted to the traditional testing of systems, in
which most vulnerabilities are related to issues in the operating systems,
products or configuration of both. Web services are an exception due to these
platforms usually being developed by organisations in an ‘ad hoc’ way and, even
if they use COTS products there is development that is unique to the
organisation. Thus, in the case of web services, the tasks are targeted at
determine the security issues due to the particular applications in use by the
organisation.</p>

<p>
Due to e-commerce being nowadays dependant on these very applications, many
security issues arise and the applications must be tested thoroughly since
issues in applications, even if they do not compromise the system they are
being executed in, might compromise the information that is use (either by
making it accesible to an attacker or giving him a chance to modify it). </p>

<p>
It is important to note that even if vulnerability assesment for security
vulnerabilities in systems and products can be done through automatic assesment
tools, assesment of applications developed in-house by the organisation can be
rarely automated due to the many factors that need to be tested.</p>

the presented in the adjointed figure. Steps for web security application
testing that need to be taken (in order):</p>

<li>
<p>retrieval of information regarding the web services used by the
organisation from public sources</p>

<li><p>analysis of the web site structure, location of interactive applications
(forms) and sections for which an authentication is required</p>

<li><p>malversation through input modification </p>
</ul>

<h2>Tasks to perform in the Web Application Test:</h2>

<ul>

<li><p>Information on the web services offered:</p>

<li><p>Determine what is the web server destined for (publicity, e-commerce, news,
webmail...)</p>

<li><p>Determine the layout (structure) of the web pages</p>

<li><p>Determine previous techologies and services offered (through archive.org)</p>

<li><p>Information on the web deployment:</p>

<li><p>Determine which technology is used for pages development</p>

<li><p>Determine authenticated zones</p>

<li><p>Determine interactive applications</p>

<li><p>Determine which technology is used for web applications (servlets, CGI
scripts, ASP, SOAP)</p>

<li><p>Determine publishing of web contents (FrontPage, WebDAV, FTP servers,
propietary administrative interfaces..)</p>

<li><p>Determine arquitecture of the service (application server? Back-end
databases?) </p>

<li><p>Analyse quality of certificates used for encryption (if any)</p>

<li><p>Determine if old versions of published information is available</p>

</ul>
<p>
Information on interactive applications:</p>

<ul>
<li>
<p>Analyse HTML frontends of interactive applications</p>

<li>
<p>Determine variables given to back-end processes</p>

<li><p>Check input validation done in the client-side (javascript) and determine
possible variable limitations (length, input type...)</p>

<li>
<p>Locate hidden variables</p>

<li>
<p>Try to retrieve (fragments of) source code used for application development</p>

<p>Re-engineering of interactive applications</p>

<li>
<p>Determine if interactive applications are COTS components or developed
internally</p>

<li>
<p>Try to retrieve binary code or source code used in the web service from
legitimate sources (vendors, public source code)</p>

<li>
<p>Decompose or deconstruct binary code</p>

<li>
<p>Guess program logic from error/debug messages in application output </p>

<p>
Application and web servers configuration testing and vulnerability
explotation:</p>

<li>
<p>Check for directories with indexing enabled</p>

<li>
<p>Check if there are user’s personal pages</p>

<li>
<p>If access to the source code of dynamic content is possible, gather all
source code for analysis</p>

<li>
<p>If publishing or administrative interfaces are enabled and vulnerable,
create trojans running on the web server to permit access to the system or
back-end databases</p>

</ul>

<p>Authentication analysis:</p>
<ul>
<li>
<p>Analyse authentication methods in use</p>

<li>
<p>Create a valid user (if possible) and determine session identification
methods (session URLs, cookies...)</p>

<li>
<p>Analyse randomness quality of session information given on different
sessions.</p>

<li>
<p>Determine if encryption is being used for session authentication.</p>

<li>
<p>Try to determine direct relationships between user identification and
session identification.</p>

<li>
<p>Brute force of authentication tokens (passwords, user certificates..) to
bypass authentication</p>

<li>
<p>Try to bypass authentication through direct access of locations behind
authentication areas </p>

</ul>

<p>Input modification:</p>

<ul>
<li>
<p>Study results of controlled modification of input variables used by
applications (a single metacharacter)</p>

<li>
<p>Study results when empty inputs, out of range inputs are sent through variables to the
application.</p>

<li>
<p>Determine vulnerability to buffer overflows in applications.</p>

<li>
<p>Compare checks/limits done on the client side against those enforce by
applications.</p>

<li>
<p>Analyse error pages (if any) for source code or internal information
disclosure</p>

<li>
<p>Determine if multiple (GET and POST) access methods are used for
applications</p>

<li>
<p>Determine if cross-side scripting is possible when input is returned back
to the user</p>

</ul>

<p>
Access to backends through input modification:</p>
<ul>
<li>
<p>Use SQL injection to determine if access is posible to back-end databases</p>

<li>
<p>Determine rights of applications accesing databases (available tables,
leaked information, etc.)</p>

<li>
<p>Determine if sensitive information is available in the database</p>

<li>
<p>Determine if web modification can be done through database manipulation</p>

<li>
<p>Determine if user authentication information is available in the databases
available</p>

<li>
<p>Determine if it’s possible to execute system commands through stored
procedures in the database.</p>

</ul>

<h1>FTP Service Testing</h1>

<p>Ftp services are commonly used as both a method of giving away legitimate
content to clients and providers through the Internet an as a method of
publishing (pushing) information on other systems (like web systems). Security
issues in these services must be thoroughly tested since one such issue could
lead to a compromise of remote systems (be it clients or servers) through the
established trust relationships. The tasks described in this module, specific
to the FTP service in itself are focused at understanding how are the FTP
systems used by the organisation and what impact might its malversation cause
to the organisation.</p>

<h2>Expected results</h2>

<ul>
<li><p>Available directories and
permissions</p>

<li><p>Sensitive information
available on the FTP system</p>

<li><p>Use of the FTP system and
trust relationships with other systems</p>

</ul>

<h2>Tasks to perform</h2>

<p>Use of the FTP system</p>

<ul>

<li><p>Determine the use of
the FTP system by it’s legitimate users</p>

<li><p>Determine trust
relationships of other systems with the FTP systems</p>
</ul>

<p>Authentication</p>
<ul>
<li><p>Anonymous
authentication possible?</p>

<li><p>Try to download
account information from the service</p>

<li><p>Brute-force
authentication of legitimate users</p>
</ul>

<p>Configuration of the FTP
system</p>
<ul>
<li><p>Determine the
structure of the service</p>

<li><p>Contents available
of the service</p>

<li><p>Attempt brute force </p>

<li><p>Permissions
available for contents and directories by different users</p>

<li><p>Determine if it’s
possible to access other parts of the system through the FTP system (users not
restricted to a given directory)</p>

<li><p>Make use of FTP bouncing
techniques to gather information of internal hosts</p>

</UL>

<h1>Mail Service Testing</h1>

<p>
Mail service, is the single most used service in the Internet, even if the
systems that route the mail might be considered secure, misconfigurations of
the mail system might give away information on how the internal systems are
setup (usually through error messages or mail headers). This module’s tasks are
directed both at attempting to extract internal information through the use of
the mail system and to compromise the mail system through impersonation of
legitimate users.</p>

<p>
The tasks describe are not only directed towards the mail gateways
themselves but also towards the different Internet applications that make use
of the mail system, this includes remote mailbox services such as POP, IMAP and
webmail services.</p>


<h2>Expected results</h2>

<ul>
<li><p>Information of valid
users in the system</p>

<li><p>Route of incoming mail</p>

<li><p>Route of outgoing mail</p>

<li><p>Virus/trojan detection
capabilities of the mail service</p>

<li><p>Internal information
leaked by the mail service</p>

<li><p>List of users and passwords
of mail services (POP, IMAP, webmail services)</p>
</ul>

<h2>Tasks to perform</h2>

<p>Route tests:</p>
<UL>
<li><p>Send forged e-mails
to valid accounts and try to get response</p>

<li><p>Send e-mails to
invalid accounts</p>

<li><p>Attempt e-mail
source routing to locate non-visible e-mail gateways</p>
</UL>

<p>Internal information
leaks:</p>
<UL>
<li><p>Send e-mails to
administrative accounts in the mail system</p>

<li><p>Send e-mails to
internal machines</p>
</UL>

<p>Authentication:</p>

<ul>
<li><p>Attempt retrieval of
valid/invalid users through misconfiguration of mail gateways</p>

<li><p>Attempt brute force
of mail accounts sending forged e-mails to gather valid/invalid accounts</p>

<li><p>Attempt brute force
of passwords in interactive mail services </p>
</UL>

<p>Virus/trojans detection capabilities:</p>

<ul>

<li><p>Send virus test
(EICAR) to invalid and valid accounts</p>

<li><p>Send known trojans to invalid and valid accounts</p>

<li><p>Determine which mail
gateway anti-virus system is in use (if any)</p>

</ul>


<h1>DNS Application Testing</h1>


<p>The DNS servers of an organization is a source of information for legitimate and
ilegitimate users alike. The DNS service is the <em>backbone</em> of the Internet
and if it is malversed then many other services (mail, web...) can be malversed
too.</p>

<p>The tasks described in this module are oriented towards extracting the available
information from the DNS service and to determine if it is possible to subvert it.</p>



<h2>Expected results</h2>

<ul>
<li><p>Complete association of IP addressess and their names</p>

<li><p>Information of the type and version of DNS server</p>
</ul>

<h2>Tasks to perform</h2>
<ul>
<li><p>Determine version of the DNS server (by requesting for the 'chaos' information on Bind servers)</p>

<li><p>Query the nameserver for hosts in the same organisation</p>

<li><p>Query the nameserver for hosts in other organisation (recursive queries)</p>

<li><p>Try to retrieve the zone information fro the server (zone transfer)</p>

<li><p>Try reverse queries for each IP address of the organisation</p>

<li><p>Determine if server is authorative name server for any given zone</p>

<li><p>Determine if the names of systems are related to specific system information (operating system, users..)</p>

<li><p>Identify potential targets (mail servers and name servers) from the name server information</p>


</ul>

<h1>NTP Application Testing</h1>

<p>The Network Time Protocol are a necessary service that is mainly used for sincronisation purposes in
internal networks. However, both the server and the service itself can be subverted by an attacker. The 
purpose of this task is to determine if (and how) time synchronisation is done in the organisation's
network and if it can be subverted.</p>

<h2>Expected results</h2>

<ul>
<li><p>Location of NTP servers</p>

<li><p>Values </p>
</ul>

<h2>Tasks to perform</h2>
<ul>

<li><p>Determine locations of NTP servers by sending NTP queries</p>

<li><p>Determine locations of NTP clients by sending forged NTP responses</p>

<li><p>Send multicast/broadcast NTP request to locate NTP servers</p>

<li><p>Determine which NTP version (1-4) is used</p>

<li><p>Determine stratums of NTP servers</p>

<li><p>Determine wether or not authentication or encryption is used</p>

<li><p>Determine which type of encryption is used (symetric or public-key)</p>

<li><p>Determine if remote configuration of servers is possible</p>

<li><p>Determine resilience of the syncrhonisation network against rogue NTP servers</p>


</ul>

<h1>Finger Application Testing</h1>


<p>The finger service is commonly used to provide information within an organisation 
on what users are available in the organisation's system as well as wether or not they 
are connected. However, the information provided by this service can sometimes leak to the 
Internet. This information itself can be used to obtain both internal and personal 
information from the users of the organisation. The goal of this sub-module is to test the
information provided by the finger service and determine if it can be useful for an 
external attacker.
</p>

<h2>Expected results</h2>

<ul>
<li><p>List of users that access the system</p>

<li><p>List of login times of the users (and their connection profiles)</p>

<li><p>Personal information provided from the service</p>

</ul>

<h2>Tasks to perform</h2>
<ul>
<li><p>Attempt finger connections to systems in the organisation</p>

<li><p>Determine finger servers in use</p>

<li><p>Retrieve information of users connected to the system ('finger @')</p>

<li><p>Brute force users in the system through the finger service</p>

<li><p>Determine which personal information (phone, location....) can be recovered from the service</p>

<li><p>Attempt of monitorisation of the users through the finger service to derive the connection profile of users</p>

</ul>


<h1>RADIUS Application Testing</h1>


<p>The Remote Authentication Dial-In User Service (RADIUS) is a client/server protocol used for authentication
of remote users in systems. This service, commonly used for dial-in systems, should be audited in order to determine
if it can be subverted by a remote attacker.</p>

<h2>Expected results</h2>

<ul>
<li><p>Location of RADIUS systems</p>

<li><p>User accounts and their passwords</p>

<li><p>User accounts and their passwords</p>

</ul>

<h2>Tasks to perform</h2>
<ul>
<li><p>Send RADIUS requests to servers in the organisation (dial-in modems or other PPP tunnel servers)</p>

<li><p>Determine RADIUS servers in use</p>

<li><p>Brute-force users and accounts in the RADIUS servers</p>

<li><p>Analyse information provided in the answer packets</p>

</ul>
