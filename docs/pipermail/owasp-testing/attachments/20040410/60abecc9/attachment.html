<P>Mark, </P>
<P></P>
<P>Here are a few more items to beef up the access control section. </P>
<P></P><BR>
<P><I><FONT size=2>Ensure that the application allows users to access only those 
functions and assets they are specifically authorized for.</FONT></I><I></I> 
</P>
<P><FONT size=2>Verifiy that users are allowed to access assets and functions as 
described in the matrix.&nbsp; Also verify that users cannot access assets and 
functions outside their authorization. Because this set is generally quite 
large, choose a set of specific tests to exercise the access control mechanism. 
For example, the least privileged user should attempt to access resources and 
functions of more privileged users.</FONT> </P><BR>
<P><I><FONT size=2>Ensure that the access control mechanism is implemented in a 
centralized fashion, not distributed throughout the 
application.</FONT></I><I></I> </P>
<P><FONT size=2>Ensure that the access control mechanism behaves consistently 
across the entire application. Distributed mechanisms are impossible to 
implement and configure correctly.</FONT> </P><BR>
<P><I><FONT size=2>Verify that all accesses to the application are subject to 
the access control check.</FONT></I><I></I> </P>
<P><FONT size=2>Attempt to access the application in a variety of ways that are 
outside the normal user's path. A proxy can be helpful here in generating 
communications that would not ordinarily be expected.</FONT> </P><BR>
<P><I><FONT size=2>Evaluate whether the application relies on any external 
information to make access control decisions.</FONT></I><I></I> </P>
<P><FONT size=2>Examine all information that enters the application and 
manipulate it to attempt to subvert the access control decision. A proxy can be 
useful here to manipulate these values.</FONT> </P><BR>
<P><I><FONT size=2>Ensure that the application uses only the identity determined 
by the identification and authentication mechanism to make access control 
decisions.</FONT></I><I></I> </P>
<P><FONT size=2>Verify that the application does not use any identifiers or 
names as a proxy for the authenticated identity.</FONT> </P><BR>
<P><I><FONT size=2>Assets and functions shall be clearly associated with the 
information required to make access control decisions.</FONT></I><I></I> </P>
<P><FONT size=2>Examine the assets and functions to be accessed by the 
application. It should be clear what part of the access control matrix they 
belong to.</FONT> </P><BR>
<P><I><FONT size=2>Verify that both coarse-grained URL based access control and 
fine-grained access control to specific functions and assets is properly 
implemented.</FONT></I><I></I> </P>
<P><FONT size=2>Attempt to access the application in a variety of ways that are 
outside the normal user's path. A proxy can be helpful here in generating 
communications that would not ordinarily be expected.</FONT> </P><BR>
<P><I><FONT size=2>Verify that users have been assigned the minimum privileges 
and authorizations necessary to perform their tasks.</FONT></I><I></I> </P>
<P><FONT size=2>Verify that users do not have privilege to perform functions 
that they do not need.</FONT> </P><BR>
<P><I><FONT size=2>Ensure that administrators have been assigned the minimum 
privileges and authorizations necessary to perform their 
tasks.</FONT></I><I></I> </P>
<P><FONT size=2>Verify that administrative users do not have privilege to 
perform unnecessary functions.</FONT> </P><BR>
<P><I><FONT size=2>Verify that only the authorized types or modes of access to 
assets and functions are granted to users.</FONT></I><I></I> </P>
<P><FONT size=2>If the application requires specific privileges, such as read, 
write, execute, etc&#8230;, verify that these privileges are accurately enforced. Be 
sure that there is no way a user with read access can cause the system to 
perform a write function.</FONT> </P><BR><BR>
