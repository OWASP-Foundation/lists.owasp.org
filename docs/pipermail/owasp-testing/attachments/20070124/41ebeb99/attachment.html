Sure. <br>Orizon borns in the late of september when I was collecting info and tools to an italian speech I took for Owasp Italy chapter @ eAcademy (inside the SMAU expo, <a href="http://www.smau.it/">http://www.smau.it/</a>
) in Milan. The speech was about  application security and code review.<br>I realized the importance about dynamic code review combined with static pattern matching approach. I realized too that a dynamic approach needs a sort of automatization, also if not all part of code review process could be automatizated (and all of us we&#39;re happy about this).
<br>I found a list of tools both opensource than closed like pdm, lapse, fortify, flawfinder, rats. Wondering about these tools I realized that they don&#39;t share information among them and sometimes they are also tied to the environment used to write software.
<br><br>The security checks not to be shared between tools arises me that there would be the need of a sort of library, or a framework in which code reviewer experience in writing tests will be translated in some way and available to anyone who want to write a code review tool.
<br><br>So I started planning Orizon with the idea to create an infrastructure, a sort of code review engine usable by anyone who want to create a code scanning tool. So Orizon greatest value will be the security check library written by experts (we, guys @ owasp :)) and available to anyone&#39;s tool.
<br>Orizon of course will be a tool using itself as framework (like a demo).<br>Orizon will be a tool not tied to IDE, so I started by now a command line tool.<br><br>For static analysis Orizon won&#39;t be tied to programming language. That&#39;s because I&#39;ll translate source code to be checked, in XML (by now only java works but I&#39;m planning to extend translation list to C#, C and other). Over XML security checks (written in XML too) will be applied in a pattern matching fashion.
<br>In the architectural document (<a href="https://www.owasp.org/index.php/Image:Orizon_internal_draft.doc">https://www.owasp.org/index.php/Image:Orizon_internal_draft.doc</a>) I planned a sort of style check but this part needs further engineering over it.
<br><br>For dynamic analysis Orizon will (better, it aims to):<br>* disassemble class or source in methods or functions<br>* create simple test tools that invokes method or function to be checked supplying an input that in a first project stage will be created by reviewer (I&#39;m reading something about fuzzy logic and I&#39;d like to implement something but in a future)
<br>* prints out unhandled exceptions, strange behaviour, ...<br><br>Of course results validation is an human task and demanded to reviewer.<br><br>By now I&#39;m focused create the static part of the framework. I&#39;ll plan to have something usable in the late spring (may 2007).
