<p>I&#39;m afraid that I&#39;d be more inclined to agree with Stephen - controls in the app should be built around protecting the session ID (e.g. general input validation, httponly and secure flags on the cookie, re-issuing the token and invalidating the old one when moving from HTTP to HTTPS, moving from unauthenticated to authenticated and again when logging out, stopping concurrent logins, etc, etc, blah, blah, and all the stuff you suggested!). </p>

<p>If someone could just get access to the token locally and copy and paste it to another browser then you have bigger problems. When testing apps, I&#39;ve never been convinced that the old &#39;copy and paste the session token from one machine to another&#39; test provides any great benefit. I suppose it&#39;s always helpful to try and think of a practical way that someone can just get access to the token like this and if you can&#39;t, then that risk that an attacker is going to be able to get the token like this is pretty low.</p>

<div>Tying the session ID to a particular location/IP is difficult as suggested. Although I&#39;m all for the cheesy &#39;security in depth&#39; approach, it depends on the purpose of the application and the type of data it is processing - the solution should be fit for the purpose of the application. I.e. a variation of all of the above is great if you&#39;re protecting a high-value application, but who really cares about implementing all of this for a brocureware site that sells lawn-mowers!<br>
</div>
