<blockquote class="gmail_quote" style="PADDING-LEFT: 1ex; MARGIN: 0pt 0pt 0pt 0.8ex; BORDER-LEFT: rgb(204,204,204) 1px solid">
<div>Assuming you are talking about some sort of session token in the cookie that can be reused on another client some other relevant tips from <a href="http://www.owasp.org/index.php/Session_Management" target="_blank">http://www.owasp.org/index.php/Session_Management</a><br>
</div>
<div>
<li>All session tokens in high value applications SHOULD be tied to a specific HTTP client instance (session identifier and IP address). </li>
<li>Application servers SHOULD use private temporary file areas per client/application to store session data. </li>
<li>All applications SHOULD use a cryptographically secure page or form nonce in a hidden form field. </li>
<li>Each form or page nonce SHOULD be removed from the active list as soon as it is submitted. </li></div>
<div>
<li>Session tokens should be regenerated prior to any significant high value transaction. </li>
<li>In high-value applications, session tokens should be regenerated after a certain number of requests. </li>
<li>In high-value applications, session tokens should be regenerated after a certain period of time. </li></div>
<div> </div>
<div>In general you can say the less predictable a session token is, the more secure it is.</div>
<div>So to make your appplication as secure as possible:</div>
<div>- make it difficult to predict</div>
<div>- link it to the client with an difficult to predict property</div>
<div>- reuse each token as little as possible and regenerate as often as possible</div>
<div>- make use of a second (unpredictable) token and validate and reissue that for every request (also helps against CSRF)</div>
