<DIV><FONT face=Verdana size=2>&nbsp;Ten Principles for Application Security Testing</FONT></DIV><FONT face=Verdana size=2>
<DIV><BR><STRONG>There is No Silver Bullet</STRONG></DIV>
<DIV>&nbsp;-&nbsp;One glove (testing approach) does not fit all<BR>&nbsp;-&nbsp;Design based on requirements and other governing factors i.e. limitations, costs, etc.</DIV>
<DIV><BR><STRONG>Think Strategically, Not Tactically</STRONG></DIV>
<DIV>&nbsp;-&nbsp;Build security into the test plan<BR>&nbsp;-&nbsp;Establish test methods<BR>&nbsp;-&nbsp;Identify security issues during the design review phase<BR>&nbsp;-&nbsp;Build in security in-depth (layered approach)<BR>&nbsp;-&nbsp;Ask questions regarding future requirements/enhancements<BR>&nbsp;-&nbsp;Future growth</DIV>
<DIV><BR><STRONG>Test Early and Test Often</STRONG></DIV>
<DIV>&nbsp;-&nbsp;Substantial cost savings <BR>&nbsp;-&nbsp;Test in development, QA, and production environments <BR>&nbsp;&nbsp;&nbsp;&nbsp; (limited access prior to "going live")<BR>&nbsp;-&nbsp;Work closely with responsible teams during the development <BR>&nbsp;&nbsp;&nbsp;&nbsp; process to remediate or mitigate issues <BR>&nbsp;-&nbsp;Establish a Due Care process based on risk level<BR>&nbsp;-&nbsp;Educate developers about Web App sec issues</DIV>
<DIV><BR><STRONG>Understand the Scope</STRONG></DIV>
<DIV>&nbsp;-&nbsp;Governing regulatory requirements<BR>&nbsp;-&nbsp;Level of security required i.e. sensitive, Top Secret <BR>&nbsp;-&nbsp;Timeframe (schedule)<BR>&nbsp;-&nbsp;Interviews with developers, involved teams, and project leads<BR>&nbsp;-&nbsp;Complexity of project<BR>&nbsp;-&nbsp;Approval requirements<BR>&nbsp;-&nbsp;Proof of concept (yes or no)<BR>&nbsp;-&nbsp;Cost limitations (time to market)<BR>&nbsp;-&nbsp;Classification of data e.g. GLBA, Cardholder, CA SB1386, etc. </DIV>
<DIV><BR><STRONG>Mindset</STRONG></DIV>
<DIV>&nbsp;-&nbsp;Think like a hacker or cracker<BR>&nbsp;-&nbsp;Customize your approach and or methodology<BR>&nbsp;-&nbsp;Think outside of the box<BR>&nbsp;-&nbsp;Try the unexpected; be creative</DIV>
<DIV>&nbsp;</DIV>
<DIV><STRONG>Know Thy Target</STRONG></DIV>
<DIV>&nbsp;-&nbsp;User &amp; Administrative documentation<BR>&nbsp;-&nbsp;Architectural diagrams and data flow charts&nbsp; <BR>&nbsp;-&nbsp;Technical system and application documentation<BR>&nbsp;-&nbsp;Disallowed functionality/intentions<BR>&nbsp;-&nbsp;Understand the threat and known exploits<BR>&nbsp;-&nbsp;Understand regulatory requirements<BR>&nbsp;-&nbsp;Thoroughly research application &amp; systems being used<BR>&nbsp;-&nbsp;Conduct passive &amp; active recon</DIV>
<DIV><BR><STRONG>Use the Right Tools</STRONG></DIV>
<DIV>&nbsp;-&nbsp;Use a combination automated and manual techniques<BR>&nbsp;-&nbsp;Frequently update tool &amp; technique inventory<BR>&nbsp;-&nbsp;Become intimately familiar with technology being used and&nbsp;<BR>&nbsp;&nbsp;&nbsp;known/probable security issues<BR>&nbsp;-&nbsp;Shouldn't bring a knife to a gun fight</DIV>
<DIV><BR><STRONG>Attention to Detail</STRONG></DIV>
<DIV>&nbsp;-&nbsp;Weed out false positives<BR>&nbsp;-&nbsp;Thoroughly explore logic in an attempt to expose flaws<BR>&nbsp;-&nbsp;Look for the less obvious security issues<BR>&nbsp;-&nbsp;Become intimate with the application<BR>&nbsp;-&nbsp;Evaluate every aspect</DIV>
<DIV><BR><STRONG>Use Manual Source Code Inspections Whenever Possible</STRONG></DIV>
<DIV>&nbsp;-&nbsp;Allows for the identification of security holes and&nbsp;<BR>&nbsp;&nbsp;&nbsp;vulnerabilities often missed by black or gray box testing<BR>&nbsp;-&nbsp;Reinforce secure coding practices <BR>&nbsp;-&nbsp;Should include User Interface development </DIV>
<DIV><BR><STRONG>Use the System Development Life Cycle (SDLC)</STRONG></DIV>
<DIV>&nbsp;-&nbsp;Structured approach that provides guidance regarding the handling&nbsp;<BR>&nbsp;&nbsp;&nbsp;of IT Security activities throughout the life of the system or application</FONT></DIV>
