I agree that any source-code analysis tool will have limitations (and in fact KNOWing what those limitations are is as important as the limitations itself).<br><br>BUT since we are talking here about source-code-review guidelines, where there is a &#39;guideline&#39; about something to look at, I think that most of those &#39;look for this&#39; actions can be automated. In most cases the code-automation tool will just a grep on steroids where it only indicates &#39;areas to look at and review&#39;, but in others (specially in VM based languages like .Net and Java) it should be able to provide better results.
<br><br>I agree with the compiler concept, and in .Net and Java I would also use the &#39;sandbox&#39; paradigm where code that is able to be executed in &#39;safe&#39; sandboxes can be automatically considered to be &#39;safe from a series of vulnerabilities.
<br><br>I think all I am saying is that with the current focus on the Code Review project it would be a great opportunity to co-develop such code-auditing-tool (which 99% of current developers desperately need). And for the technical difficulties of implementing such tool, If I can, I will be very active in resolving them
