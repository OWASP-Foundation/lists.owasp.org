<H1 dir=3Dltr align=3Dleft>Title Cross Site Scripting<BR></H1>
<H2>Date 17th November<BR></H2>
<H2>Author Javier Fern=E1ndez-Sanguino<BR></H2>
<H1>Short Description of Issue</H1>
<P>Cross Site Scripting attacks (XSS) are attempts to subvert the =
relationship=20
between a user's browser and a trusted site in order to retrieve =
sensitive=20
information from the user's browser. Attackers can exploit this attack =
due to=20
flaws in the web server code (such as the way error request are shown) =
or in the=20
way it stores information that is later on presented to user (comments =
to news,=20
forums, web-based e-mail readers, etc.). If the information is not =
sanitized=20
before it is stored or&nbsp; if it is not santized when introduced into =
an HTML=20
object, it might lead to code execution in the users's browser, this =
code is=20
executed as in the context of the trusted site. Execution of code, =
including=20
Javascript, VisualBasic Script,&nbsp; ActiveX, or Java applets can be =
used for a=20
different purposes (including arbitrary remote code execution in =
vulnerable or=20
misconfigured browsers), but a Cross Site Scripting has as its main =
purpose the=20
retrieval of session information that involves the remote site. This =
session=20
information (usually in the form of cookies) can then be used to hijack =
the=20
user's session (before it expires) if the remote site does not control =
where the=20
session was established form.</P>
<P>XSS attacks can be produced both by sending crafted URLs to potential =
targets=20
through e-mail or by storing them in web-pages and hiding them as =
legitimate=20
links, images, or other type of information. Common sources of these =
kind of=20
attacks are webmail applications and web-based bulletin boards, but in =
some=20
cases even web servers are directly affected.<BR></P>
<H1>How to Test</H1>
<P>Testing the success of XSS attempts involves determining where does =
the web=20
application introduce information generated by an attacker into an HTML =
response=20
which might be viewed by a user (and interpreted by his browser). =
Usually,=20
determining that JavaScript will be run is enough to demonstrate that an =
XSS=20
attack is possible. Although whether or not this kind of attack can =
succeed=20
depends on if this attack can be introduced permanently into the web =
server (for=20
example through a comment posted in the web site) or needs to be done =
through=20
social engineering attacks (like e-mail).<BR></P>
<H2>Black Box</H2>
<P>In order to do black box testing of XSS the tester first needs to =
determine=20
which parts of the web server application introduces information =
generated by an=20
attacker. This can include either dynamically generated HTML pages based =
on POST=20
or GET calls to applications (including error pages they generate which =
include=20
said content) or content generated based on information stored in the =
website=20
that is later on presented to legitimate users. Once these places have =
been=20
determined the tester should try to inject code and see if it is =
included=20
verbatim in the HTML page returned and whether it is filtered or not. If =

filtered, the tester needs to make sure that all potential sources for =
XSS=20
attacks are appropiately filtered.<BR></P>
<P>Notice that it is also necessary to determine if it is possible to do =
session=20
replays (using retrieved cookies) from sources different from the one =
that=20
started the session. Session hijacking might be hindered and even if XSS =
attacks=20
are possible, their impact might be limited.<BR></P>
<P></P>
<H2>White Box <BR></H2>
<P>Source code analysis must be focused on how the application manages=20
information that is presented to the end user or stored to later =
generate HTML=20
pages . Any such information must be sanitized before storage and must =
be also=20
santized before it is given to the user; this includes filtering all =
HTML tags.=20
If HTML tags are allowed in the content, the reviewer must make sure =
that the=20
filter implement to sanitize these tags only allows tags (and =
parameters) which=20
are not subject to XSS attacks. </P>
