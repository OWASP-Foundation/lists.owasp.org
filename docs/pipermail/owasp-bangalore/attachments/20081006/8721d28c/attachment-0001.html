<h2><a href="http://www.cgisecurity.org/2008/10/interview-jerem.html#">Interview: Jeremiah Grossman provides more details on clickjacking attack</a></h2>
<div class="line"></div>
<div class="entry-content">
<div class="entry-body">
<p>Little information has been provided on <a href="http://www.securityfocus.com/news/11534">ClickJacking</a> so I decided to go digging a little bit and talk to the source to find out some additional information. Here&#39;s my interview with <a href="http://jeremiahgrossman.blogspot.com/">Jeremiah Grossman</a> on Friday October 3rd.</p>

<p><strong>How did you find this flaw exactly? Was it something you were digging for or was it by accident? </strong><br>ClickJacking as an attack technique <a href="http://ha.ckers.org/blog/20080915/clickjacking/">Robert</a> and I discovered around a year and a half ago. Recently we&#39;re told we've been told that its been known by the browser vendors since 2002. In any case the attack has been essentially underestimated and largely undefended by the web security community in general. Post <br>
Black Hat 2008 some research we were conducting was furthered by using ClickJacking, as a result we felt warranted more attention. What we didn&#39;t know didn&#39;t realize at first was that one of our proof-of-concept examples used a zero-day in an Adobe product. When we found out, because Adobe informed us, that&#39;s when we decided to postpone our OWASP conference talk.</p>

<p><strong>Can ClickJacking reach out of the browser? </strong><br>Clickjacking can be used to exploit just about anything between the browser walls, or perhaps more specifically, anything that is part of the DOM.</p>
<p><strong>Are applications other than browsers affected, if so what?</strong><br>We haven&#39;t researched that aspect of ClickJacking.<br><strong><br>Is it likely this has been used by attackers to actively exploit people?</strong><br>
We believe ClickJacking may have been used by advertising click fraudsters, but we don't know for sure. Beyond that ClickJacking attacks would be incredibly hard for the average user to detect, and even if they did, it would be tough for them to describe.</p>

<p><strong>Knowing that after you discuss this bad guys are going to use it to do you feel it is better to still talk about it?&nbsp; </strong><br>Bad guys tend to use the attack techniques that are the easiest to monetize. While ClickJacking is somewhat trivial, they're already vested in using attack techniques like SQL Injection until that stops working. From my experience in Web security, the bad guys start taking advantage of new techniques 12-18 months after initial disclosure if their able to monetize it will enough. </p>

<p>When we discuss new attacks, it evens the playing field for everyone. Those who want to defend themselves quickly now have the information available to do so.</p>
<p><strong>Have you received negativity from anyone for wanting to disclose this?</strong><br>There has been some yes, but that's to be expected. It's impossible to please everyone all the time when it comes to matters of vulnerability disclosure. Everyone has they're preference. What we're doing is trying to keep the end user as our #1 priority when discussing these matters publicly. </p>

<p><strong>Does this flaw still work if you&#39;re using a keyboard with no mouse? </strong><br>Yes. If you can "click", you can be "ClickJacked".</p>
<p><strong>Does this flaw affect other technologies such as Silverlight, Javafx, applets, etc? Is anything immune?</strong><br>We are unable to say for sure, more research would need to be perform by Robert and myself, or others in the industry.</p>

<p><strong>Do iframes offer any sort of protection? </strong><br>No, they are one source of the problem.</p>
<p><strong>Does this break protections for flaws such as Cross-Site Request Forgery? </strong><br>Yes. Clickjacking has the potential of breaking <a href="http://www.cgisecurity.com/articles/csrf-faq.shtml">CSRF</a> token-based protections.</p>

<p><strong>Boxers or briefs? </strong><br>I prefer No Disclosure. ;)</p>
<p>The fact that <a href="http://www.cgisecurity.com/articles/csrf-faq.shtml">CSRF</a> token based protection may be busted and that there is no clear fix for browser makers is sure to stir things up in the industry. Full details will be published at the <a href="https://conference.hackinthebox.org/">HITB</a> conference later this month. Certainly some interesting research.</p>
