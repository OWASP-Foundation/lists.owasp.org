== Talk ==</b></font><font size=3> <br>
</font><font size=2 face="Arial"><b><br>
Title: </b>Define Security Requirements - A practical approach</font><font size=3>
<br>
</font><font size=2 face="Arial"><b><br>
Abstract: &nbsp;</b> </font><font size=3><br>
</font><font size=2 face="Arial"><br>
The Data Protection Act states that &quot;appropriate security measures&quot;
must be taken to protect personal data. How do you specify the appropriate
security measures for a website which processes personal data? It is an
important step in a development project, but is often neglected. In this
talk, Alexis will descibe his own experiences of assessing web application,
and will also look in more detail at what the Data Protection Commissioner
says. He will then take a fictional website and look at a practical approach
to specifying the security requirements that the fictional application
should meet. This will use the kind of risk-based techniques outlined by
OWASP or the Microsoft Secure Development Lifecycle (SDL). Issues discussed
will include encryption, authentication, access control, audit, etc. The
result will be a list of security requirements that can be carried into
the design and development phases. Attendees should be able to apply the
ideas to their own development projects.</font><font size=3> <br>

2. OWASP CSRF Tester: How to test your apps for CSRF vulnerabilities</b></font><font size=3><br>
</font><font size=3 color=blue><u><br>
</u></font><a href="http://www.owasp.org/index.php/Cross-Site_Request_Forgery"><font size=2 color=blue face="Arial"><u>Cross-Site
Request Forgery</u></font></a><font size=2 face="Arial"> (CSRF) is an attack
whereby the victim is tricked into loading information from or submitting
information to a web application for which they are currently authenticated.
The problem is that the web application has no means of verifying the integrity
of the request. The OWASP CSRFTester Project attempts to give developers
the ability to test their applications for CSRF flaws. </font><font size=3><br>
</font><font size=2 face="Arial"><br>
More info here: </font><a href=http://www.owasp.org/index.php/Category:OWASP_CSRFTester_Project target=_blank><font size=2 color=blue face="Arial"><u>http://www.owasp.org/index.php/Category:OWASP_CSRFTester_Project</u></font></a><font size=3>

3. OWASP AntiSamy: How to prevent malicious input in your apps</b></font><font size=3>
<br>
</font><font size=2 face="Arial"><br>
The OWASP AntiSamy project is a few things. Technically, it is an API for
ensuring user-supplied HTML/CSS is in compliance within an application's
rules. Another way of saying that could be: It's an API that helps you
make sure that clients don't supply malicious cargo code in the HTML they
supply for their profile, comments, etc. that gets persisted on the server.
The term malicious code in terms of web applications is usually regarded
only as JavaScript. Cascading Stylesheets are only considered malicious
when they invoke the JavaScript engine. However, there are many situations
where &quot;normal&quot; HTML and CSS can be used in a malicious manner.
</font><font size=3><br>
</font><font size=2 face="Arial"><br>
More info here: </font><a href=http://www.owasp.org/index.php/Category:OWASP_AntiSamy_Project target=_blank><font size=2 color=blue face="Arial"><u>http://www.owasp.org/index.php/Category:OWASP_AntiSamy_Project</u></font></a><font size=3>

4. HTML5: Security Facts developers should keep in mind</b></font><font size=3>
<br>
</font><font size=2 face="Arial"><br>
The war on the words between Apple and Adobe Systems has prompted plenty
of speculation about the fate of HTML5. But while HTML5 remains a work
in progress, the one thing that is certain is developers who adopt HTML5
will have a new set of features to consider as part of the application
security development life cycle.</font><font size=3> <br>
