I'd suggest that a major risk from a fat client is a badly-implemented<br>
official version of the client, as opposed to some hacked one, as the<br>
article discusses. There's nothing inherently wrong with using a fat<br>
client, as long as all communication goes via some secure channel. But,<br>
a development error could break that principle. <br>
<br>
With a thin client (ideally pure, basic html) you have the browser's<br>
pretty well accepted security available. This generic security<br>
implementation is most likely better than any custom implementation.<br>
Granted some custom code might also be needed, but the less of it (and<br>
the more isolated) the better! <br>
