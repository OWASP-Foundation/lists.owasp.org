</div><div style><br></div><div style>From the release notes:</div><div style><br></div><div style><div style="color:rgb(0,0,0);font-family:sans-serif;line-height:19.18402862548828px;border-bottom-color:rgb(153,153,153);border-bottom-style:solid;border-bottom-width:1px;margin-bottom:3px;padding-bottom:3px;font-size:18.88888931274414px">
What Changed From 2010 to 2013?</div><p style="font-size:13.333333969116211px;margin:0.4em 0px 0.5em;line-height:19.18402862548828px;color:rgb(0,0,0);font-family:sans-serif">The threat landscape for applications security constantly changes. Key factors in this evolution are advances made by attackers, the release of new technologies with new weaknesses as well as more built in defenses, and the deployment of increasingly complex systems. To keep pace, we periodically update the OWASP Top 10. In this 2013 release, we made the following changes:</p>
<ol style="font-size:13.333333969116211px;line-height:19.18402862548828px;margin:0.3em 0px 0px 3.2em;padding:0px;color:rgb(0,0,0);font-family:sans-serif"><li style="margin-bottom:0.1em">Broken Authentication and Session Management moved up in prevalence based on our data set,. Probably because this area is being looked at harder, not because issues are actually more prevalent. This caused Risks A2 and A3 to switch places.</li>
<li style="margin-bottom:0.1em">Cross-Site Request Forgery (CSRF) moved down in prevalence based on our data set from 2010-A5 to 2013-A8. We believe this is because CSRF has been in the OWASP Top 10 for 6 years, and organizations and framework developers have focused on it enough to significantly reduce the number of CSRF vulnerabilities in real world applications.</li>
<li style="margin-bottom:0.1em">We broadened Failure to Restrict URL Access from the 2010 OWASP Top 10 to be more inclusive:<p style="margin:0.4em 0px 0.5em;line-height:1.5em;padding-left:2em">+     2010-A8: Failure to Restrict URL Access is now <u>2013-A7: Missing Function Level Access Control</u> – to cover all of function level access control. There are many ways to specify which function is being accessed, not just the URL.</p>
</li><li style="margin-bottom:0.1em">We merged and broadened 2010-A7 & 2010-A9 to CREATE: <u>2013-A6: Sensitive Data Exposure</u>:<p style="margin:0.4em 0px 0.5em;line-height:1.5em;padding-left:2em">-     This new category was created by merging 2010-A7 – Insecure Cryptographic Storage & 2010-A9 - Insufficient Transport Layer Protection, plus adding browser side sensitive data risks as well. This new category covers sensitive data protection (other than access control which is covered by 2013-A4 and 2013-A7) from the moment sensitive data is provided by the user, sent to and stored within the application, and then sent back to the browser again.</p>
</li><li style="margin-bottom:0.1em">We added: <u>2013-A9: Using Components with Known Vulnerabilities</u>:<p style="margin:0.4em 0px 0.5em;line-height:1.5em;padding-left:2em">+     This issue was mentioned as part of 2010-A6 – Security Misconfiguration, but now has a category of its own as the growth and depth of component based development has significantly increased the risk of using components with known vulnerabilities.</p>
</li></ol></div><div style><br></div><div style>Regards,</div><div style>Darren Fitzpatrick</div><div style>OWASP Ireland Board</div><div style><br></div><div style><br></div></div>
