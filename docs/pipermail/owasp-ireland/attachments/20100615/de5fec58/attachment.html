== Talk ==</b></font><font size=3> <br>
</font><font size=2 face="Arial"><b><br>
Title: </b>Define Security Requirements - A practical approach</font><font size=3>
<br>
</font><font size=2 face="Arial"><b><br>
Abstract: &nbsp;</b> </font><font size=3><br>
</font><font size=2 face="Arial"><br>
The Data Protection Act states that &quot;appropriate security measures&quot;
must be taken to protect personal data. How do you specify the appropriate
security measures for a website which processes personal data? It is an
important step in a development project, but is often neglected. In this
talk, Alexis will descibe his own experiences of assessing web application,
and will also look in more detail at what the Data Protection Commissioner
says. He will then take a fictional website and look at a practical approach
to specifying the security requirements that the fictional application
should meet. This will use the kind of risk-based techniques outlined by
OWASP or the Microsoft Secure Development Lifecycle (SDL). Issues discussed
will include encryption, authentication, access control, audit, etc. The
result will be a list of security requirements that can be carried into
the design and development phases. Attendees should be able to apply the
ideas to their own development projects.</font><font size=3> <br>

<br>
<br><font size=2 face="Arial"><b><i>Title:</i></b></font>
<br><font size=2 face="Arial">Secure Application Development: Writing secure
code (and testing it)</font>
<br>
<br><font size=2 face="Arial"><b><i>Trainers: </i></b></font>
<br><font size=2 face="Arial">Eoin Keary &nbsp;Senior Manager, Ernst &amp;
Young, OWASP Board Member </font>
<br><font size=2 face="Arial">Rahim Jina Senior Consultant, Ernst &amp;
Young, OWASP Ireland chapter board. </font>
<br>
<br><font size=2 face="Arial"><b><i>Abstract:</i></b></font>
<br><font size=2 face="Arial">Writing Secure code is the most effective
method to securing your web applications. Writing secure code takes skill
and know-how but results in a more stable and robust application and assists
in protecting an organisations brand. </font>
<br>
<br><font size=2 face="Arial">Application security is not commonly a part
of many computer science curricula today and most organizations have not
focused on instituting a culture that includes application security as
a core part of their software development training efforts. This intensive
one-day course focuses on the most common web application security problems,
including aspects of both the OWASP Top Ten (2010) and the MITRE Top 25.
&nbsp;The course will introduce and demonstrate application assessment
techniques, illustrating how application vulnerabilities can be exploited
so students really understand how to avoid introducing such vulnerabilities
in their code. </font>
<br>
<br><font size=2 face="Arial">This training can be booked when getting
a ticket to the event.</font>

5. Cross-Site Tracing (XST): The misunderstood vulnerability</b></font><font size=3>
<br>
</font><font size=2 face="Arial"><br>
In January 2003 Jeremiah Grossman divulged a method to bypass the HttpOnly
cookie restriction. He named it Cross-Site Tracing (XST), unwittingly starting
a trend to attach &quot;cross-site&quot; to as many web-related vulnerabilities
as possible.</font>
<br>
<br><font size=2 face="Arial">Alas, the &quot;XS&quot; in XST evokes similarity
to XSS (Cross-Site Scripting) which has the consequence of leading people
to mistake XST as a method for injecting JavaScript. (Thankfully, character
encoding attacks have avoided the term Cross-Site Unicode, XSU.) Although
XST attacks rely on browser scripting to exploit the vulnerability, the
vulnerability is not the injection of JavaScript. XST is a means for accessing
headers normally restricted from JavaScript.</font>
<br>
<br><font size=2 face="Arial">Confused yet? You could find the full article
here: </font><a href="http://www.deadliestwebattacks.com/2010/05/cross-site-tracing-xst-misunderstood.html" target=_blank><font size=2 color=blue face="Arial">http://www.deadliestwebattacks.com/2010/05/cross-site-tracing-xst-misunderstood.html</font></a>
