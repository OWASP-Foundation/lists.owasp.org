My phone line is always open :-) I agree with this line of thought and
perhaps can offer an extension which I see as gap in our current
understanding of why and where security vulnerabilities are coded into our
web applications. At present there is no specific Web Design and
Architecture methodology that has a maturity level commensurate with
traditional software development. Therefore I would suggest that most web
applications are not 'designed' in a manner that would allow security to be
addressed as a normal part of the process of creating the application. It is
this lack of maturity and  the failure to design web applications that is
the start of the end for security. No matter how many lists, vulnerabilities
or education developers receive, security issues will still be with us until
we start to engage in the area of design.
./C

> I have one as well: "The path to secure software does not start with
> specific vulnerabilities".
>
> I think anyone on the Irish mailing list will know I have been saying since
> the release of the Sans Top 25 list earlier this year that I don't think
> lists of vulnerabilities is the optimal approach to developer education and
> ultimately secure software. We as software security professionals are guilty
> of telling to developers to prevent a list of vulnerabilities instead of
> telling to develop securely, yes I do feel there is a difference here. Think
> of it this way, when you are learning to drive does the instructor give you
> a list of ways to crash a car and hope you figure out how to avoid all of
> those different ways of crashing or does he teach you how to drive safely
> and within the rules of the road?
>
> Everyone without fail in the application security community is guilty of
> doing this yet we blame the developers when the applications are insecure,
> are the security professionals not just as culpable for not educating
> developers correctly? Whether it be the OWASP top ten, PCI DSS Requirement
> 6.5 or companies offering secure development training they all focus on a
> small set of vulnerabilities instead of focusing on a set of secure
> development principles such as Input Validation, Error Handling and Secure
> Communications.
