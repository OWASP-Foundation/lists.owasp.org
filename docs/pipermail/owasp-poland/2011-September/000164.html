>> On 09/15/11 17:27, Przemyslaw Skowron wrote:
>>> Czy w związku z rekordową dyskusją jest szansa na streszczenie,
>>> podsumowanie lub coś w ten deseń?
>>
>> Proponuję zacząć od publikacji Marka, treść dostępna na jego 
>stronie tutaj:
>>
>> http://marek.zachara.name/marek/publ/2011_ares.pdf
>>
>> Nie zawiera ona wyników testów praktycznych, które były pokazane 
>na
>> slajdach w samej prezentacji.
>> Wyniki są ciekawe - przy poprawnie nauczonym WAF-ie pokazują 
>bardzo
>> dobrą skuteczność ochrony przed atakami, przy znikomej ilości
>> false-positive (istotna uwaga - skala pionowa jest 
>logarytmiczna).
>> Mnie osobiście zaskoczyło, że wielokrotnie słabszą skuteczność 
>ochrony
>> przed atakami miała klasyfikacja typu parametru (czy jest to e-
>mail,
>> wartość liczbowa, słowo, tekst...), w porównaniu do skuteczności
>> klasyfikacji chociażby wg długości parametru oraz jego rozkładu 
>znaków.
>> Niewykluczone że częściowo wpływ na takie wyniki miało dobranie 
>zestawu
>> ataków testowych.
>>
>>
>> Zaproponowane dość sensownie 2 główne tryby pracy WAF - każda z 
>nich ma
>> swoje wady i zalety:
>>
>> 1. Tryb "triggered learning" - mający zastosowanie głównie w 
>aplikacjach
>> klasy taka "bankowość elektroniczna". W trakcie testów 
>akceptacyjnych -
>> gdy i tak cała aplikacja jest przeklikiwana - firewall 
>przestawia się w
>> tryb uczenia prawidłowego ruchu. Następnie jak już firewall 
>nauczy się
>> prawidłowych schematów, włącza się go w tryb ochrony, i wszystko 
>co
>> odbiega od znanych wartości jest blokowane.
>> Główna wada - konieczność pełnego i dokładnego przeklikania 
>aplikacji w
>> trybie uczenia (i za każdym razem przy nowych wersjach).
>>
>> 2. Tryb "continuous learning" - bardziej dla aplikacji typu duży 
>portal.
>> Takie aplikacje mają ogromną ilość prawidłowego ruchu, z którego 
>na
>> podstawie analizy statystycznej i behawioralnej, robionej w 
>dłuższym
>> odcinku czasu, WAF może wyłowić niestandardowe zachowania.
>> Zaletą tego rozwiązania jest praktycznie bezobsługowość, wadą - 
>że musi
>> być duża próbka prawidłowego ruchu, a także mniejsza skuteczność 
>w mało
>> używanych zakamarkach aplikacji (atakujący może tam łatwiej 
>"nauczyć"
>> firewall że długi i wielokrotnie powtarzany atak jest 
>prawidłowym
>> zachowaniem).
>>
>>
>> W tym momencie jest to projekt typowo badawczy, a nie nastawiony 
>wprost
>> na wypuszczenie konkretnego, łatwo używalnego produktu. Nie 
>zawiera i
>> nie ma więc w planach na razie ficzerów typu ładny interfejs
>> użytkownika, konfigurator, wpinka do ręcznej edycji regułek itp.
>> Nie ma też jeszcze zrobionego w praktyce modułu szyfrującego 
>zapamiętane
>> wzorce - w pewnych środowiskach mogą one zawierać np. dane 
>osobowe czy
>> inne, wymagające przechowywania w sposób bezpieczny.
>>
>> Roadmapa to m.in. obsługa sesyjności, operacji kilkukrokowych a 
>także
>> jednoczesnej analizy wartości różnych parametrów - w tej chwili
>> analizuje każdy parametr osobno, a nie wnika w zależności między 
>nimi.
