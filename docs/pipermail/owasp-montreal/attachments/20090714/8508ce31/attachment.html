<span class="Apple-style-span" style="border-collapse: collapse; "><div>Before I answer, here are a fee reference from the OWASP:</div><div><br></div><div>OWASP guide on Session Management can be found here: <a href="http://www.owasp.org/index.php/Session_Management" target="_blank" style="color: rgb(119, 153, 187); ">http://www.owasp.org/index.php/Session_Management</a></div>
<div><br></div><div>The guide talks about when to rotate a session identifier here:</div><div><a href="http://www.owasp.org/index.php/Session_Management#Rotate_Session_Identifiers" target="_blank" style="color: rgb(119, 153, 187); ">http://www.owasp.org/index.php/Session_Management#Rotate_Session_Identifiers</a></div>
<div><br></div><div><br></div><div>Here the paragraph on when to Rotate Session Identifiers:</div><div><span style="font-family: -webkit-sans-serif; font-style: italic; line-height: 19px; ">For high value sites, session identifiers should be regenerated prior to any significant transaction, after a certain period of time and after a certain number of requests. For medium and low value sites, Token regeneration should be performed after a change in user privilege, such as moving from an anonymous visitor to a logged in user or moving from a logged in user to an administrator. If a user is moving from an insecure page on the site to a more secure section of the site that uses HTTPS, the session ID should also be regenerated so that the secure session ID has never been transmitted in an unencrypted state. An additional session ID might be used instead of regenerating the original session ID. Unless this capability is built in to the application framework, it must implemented in addition to the application framework session controls. The prior recommendations about first leveraging platform security mechanisms still apply – because this control measure often includes writing additional custom code for an application the application should require the application framework session management features to be in operation as well as this additional rotating session identifier.</span></div>
<div><br></div><div>Now, if I understand correctly, your question is: &quot;Is it a good practice to terminate the current session on a password change?&quot;</div><div><br></div><div>I assume that what you mean by terminate is more &quot;rotate&quot; i.e. change the session identifier while keeping the rest of the session information intact. If that case and that changing the password is a significant operation in your application, the guide agrees with you right in the first sentence: &quot;<span style="font-family: -webkit-sans-serif; font-style: italic; line-height: 19px; ">For high value sites, session identifiers should be regenerated prior to any significant transaction[...]</span><span style="font-family: -webkit-sans-serif; line-height: 19px; ">&quot;. </span></div>
<div><font face="-webkit-sans-serif"><span style="line-height: 19px; "><br></span></font></div><div><font face="-webkit-sans-serif"><span style="line-height: 19px; ">This being said, if by terminate you meant actually logging the user out I would not recommend it since it would create great confusion from the user stand point. Also it would require creating a special exit page to indicate that the password change operation was successful otherwise you may have user complaining they cannot log back into the system.</span></font></div>
