Pravir, Dennis,

 

That’s a good point. 50% is initial value, assuming zero knowledge about an application and reflecting “full” uncertainty about its vulnerability status. The whole point is that you then apply the whole battery of tests against the application to see what is its real vulnerability status.

 

Also note that I’ve defined vulnerability as something that directly leads to a compromise, or more precisely one of three outcomes leading to loss (data breach, opportunistic hack, client attacks). With this assumption I wasn’t really able to confirm many industry reports that claimed that 70% applications on Internet are vulnerable. 

 

Of course, if you run a scanner against virtually any website you’ll get a number of “medium” and “high” rated “issues”, but if I was asking myself “can I hack this application right away” in most cases the answer was be “no”.

I went through the model very quickly and I'm wondering why you chose 50% for the vulnerability likelihood… In coin-flip prediction calcs this would make sense, but in your model it's not used as such (if I'm understanding it correctly). It seems like a pretty fundamental constant that directly derives cost tolerances and such, meaning that if you changed it to ~100% (a far more likely figure based on my experience with security testing), wouldn't all of the cost projections pretty much default to the max value of potential loss? Like I said, I may have misunderstood, but just some thoughts… Interesting ideas overall.
