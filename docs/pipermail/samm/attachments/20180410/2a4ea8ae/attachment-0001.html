            moz-do-not-send="true">https://github.com/OWASP/samm/issues</a>).</p>
        <p>Remarks made in xls, see below:<br>
        </p>
        <p>Cel: C4<br>
          Imho this should be limited to either measure value of:<br>
          - data<br>
          - IS-systems(applications /software) or<br>
          - determine business IT risks<br>
          It is now not clear and far too wide.<br>
          Cel: D5<br>
          Develop a business case (instead of build …)<br>
          Building a business case is a proces on its own.<br>
          Cel: C6<br>
          Not Clear how 'undersanding' can be made objective. A SAMM
          assessment should not deliver discussions, imho it should be a
          measurements of risks for this point towards clear objective
          criteria.<br>
          Cel: D6<br>
          Suggestion: Develop a security strategy plan (instead of
          'strategic plan')<br>
          Cel: C7<br>
          Most of the time a default approved business risk profile is
          not present. It is also in many companies common practice to
          make a clear seperation between IT risks and business risks.
          IT risks are part of business risk, but business risks are far
          more wider. So<br>
          maybe rephrase this to: Align or update the IT risks to the
          business risks profile if present.<br>
          (seeing G1, SAMM is for IT risks) to<br>
          Cel: C8<br>
          For G1 most of the time a decent classification schema for IT
          risks is not present. So if the aim of SAMMv2 is to create
          more consultancy work, excelllent job, else consider this
          improvement:<br>
          Classify determined IT risks using the business classification
          schema if present. If a standard classification schema is not
          present consider creating one based on an open industry
          standard classification schema for the business under
          consideration.<br>
          Cel: E10<br>
          Suggestion: Keep it tailored to security complliance. Business
          compliance is an other world. Even so with privacy compliance.<br>
          Cel: C12<br>
          Most compliance rules have nothing to do with IT security. So
          why a 'single' panel-of-glass…?<br>
          Q:'What is meant with the term 'patchwork' in this context?<br>
          Suggestion: Concentrate IT standards and policies is often
          already hard enough.<br>
          Cel: C13<br>
          Suggestion:Tailor it down to security relevant compliance
          drivers.<br>
          Cel: D17<br>
          Suggestion: Not only to application devs, but to all who are
          in the devops or developement proces active. IT and not IT
          people.<br>
          Cel: E17<br>
          Centralize should not be the goal. Imho with large enterprises
          it is better to steer on security knowlegde available within
          all business units and coordinate this. Central coordination
          is different than having all knowlegde centralized.<br>
          Cel: C18<br>
          Suggestion: Security education.<br>
          So ommit the word 'technical'<br>
          Cel: C19<br>
          'build-security-in ..' : No clue what is meant here.<br>
          Suggestion: Develop a security by design mindset<br>
          Cel: D30<br>
          Using use-cases within a agile devops teams can work just
          well. Collecting and tracing requirements is a bit
          'old-skool'. Key is to have security use-cases when developing
          a product.<br>
          Cel: C33<br>
          This will be opionion based. So suggestion: Ask for
          certifications or training or security audit reports.<br>
          Cel: D38<br>
          Standardization can be a huge thread for security. Why
          standardize?<br>
          Suggestion: Evaluate and determine if and where
          standardization on IT building blocks (software, framworks, OS
          etc) lowers security risks.<br>
          Cel: E38<br>
          Biased opinion. What is 'standard?'<br>
          Suggestion: Use technologies that meet security standards and
          certifications. E.g. for OSS there is the badge-program
          present (bestpractices.coreinfrastructure.org)(<br>
          Cel: C49<br>
          So devops is not possible? Maybe change this to:<br>
          Make use of different roles, account and implement
          multiple-eye principles where needed for (manual) deployment.<br>
          Cel: B51<br>
          Opmerking: (Bart) should we say something about configuration
          file integrity ? <br>
        </p>
        <p>> Imho: Not too much, this can be done on multiple ways,
          but if you prefer to add something: Use an approved and method
          that is recommend by standard bodies (e.g. NIST)<br>
        </p>
        <p>(Yan) by requiring treating configuration files as code and
          using versioning, we are starting to focus on integrit. It's
          also addressed in the Deployment process. I played with the
          langugae, and it always comes of a little redundant.<br>
        </p>

                style="line-height:1.38;margin-top:0pt;margin-bottom:0pt"><span style="font-size:11pt;font-family:Arial;background-color:transparent;font-variant-numeric:normal;font-variant-east-asian:normal;vertical-align:baseline;white-space:pre-wrap">Hi,</span></p>
              <p dir="ltr"
                style="line-height:1.38;margin-top:0pt;margin-bottom:0pt"><span style="font-size:11pt;font-family:Arial;background-color:transparent;font-variant-numeric:normal;font-variant-east-asian:normal;vertical-align:baseline;white-space:pre-wrap">
</span></p>
              <p dir="ltr"
                style="line-height:1.38;margin-top:0pt;margin-bottom:0pt"><span style="font-size:11pt;font-family:Arial;background-color:transparent;font-variant-numeric:normal;font-variant-east-asian:normal;vertical-align:baseline;white-space:pre-wrap">After a year of work by the SAMMv2 team, we now release the alpha version of our new SAMMv2 framework.</span></p>
              <br>
              <p dir="ltr"
                style="line-height:1.38;margin-top:0pt;margin-bottom:0pt"><span style="font-size:11pt;font-family:Arial;background-color:transparent;font-variant-numeric:normal;font-variant-east-asian:normal;vertical-align:baseline;white-space:pre-wrap">Our objective is to update the SAMM framework taking into account the following improvements:</span></p>
              <p dir="ltr"
                style="line-height:1.38;margin-top:0pt;margin-bottom:0pt"><span style="font-size:11pt;font-family:Arial;background-color:transparent;font-variant-numeric:normal;font-variant-east-asian:normal;vertical-align:baseline;white-space:pre-wrap">1) clean out inconsistencies from the previous release;</span></p>
              <p dir="ltr"
                style="line-height:1.38;margin-top:0pt;margin-bottom:0pt"><span style="font-size:11pt;font-family:Arial;background-color:transparent;font-variant-numeric:normal;font-variant-east-asian:normal;vertical-align:baseline;white-space:pre-wrap">2) a more logical flow of maturity levels of the security activities as part of the SAMM practices;</span></p>
              <p dir="ltr"
                style="line-height:1.38;margin-top:0pt;margin-bottom:0pt"><span style="font-size:11pt;font-family:Arial;background-color:transparent;font-variant-numeric:normal;font-variant-east-asian:normal;vertical-align:baseline;white-space:pre-wrap">3) take into account </span><span style="font-family:Arial;font-size:14.6667px;white-space:pre-wrap">agile </span><span style="background-color:transparent;font-family:Arial;font-size:11pt;white-space:pre-wrap">software development and DevOps practices;</span></p>
              <p dir="ltr"
                style="line-height:1.38;margin-top:0pt;margin-bottom:0pt"><span style="font-size:11pt;font-family:Arial;background-color:transparent;font-variant-numeric:normal;font-variant-east-asian:normal;vertical-align:baseline;white-space:pre-wrap">4) decrease the number of "audit or quality gate" activities in the previous framework.</span></p>
              <br>
              <p dir="ltr"
                style="line-height:1.38;margin-top:0pt;margin-bottom:0pt"><span style="font-size:11pt;font-family:Arial;background-color:transparent;font-variant-numeric:normal;font-variant-east-asian:normal;vertical-align:baseline;white-space:pre-wrap">A core team - with the help of many volunteers - has worked on the new framework since the last OWASP summit in June 2017.</span></p>
              <p dir="ltr"
                style="line-height:1.38;margin-top:0pt;margin-bottom:0pt"><span style="font-size:11pt;font-family:Arial;background-color:transparent;font-variant-numeric:normal;font-variant-east-asian:normal;vertical-align:baseline;white-space:pre-wrap">During that summit, we lay the foundations of version 2.0. </span></p>
              <p dir="ltr"
                style="line-height:1.38;margin-top:0pt;margin-bottom:0pt"><span style="font-size:11pt;font-family:Arial;background-color:transparent;font-variant-numeric:normal;font-variant-east-asian:normal;vertical-align:baseline;white-space:pre-wrap">We then added more details to the draft version with a SAMM summit in Reykjavik and extra online conference calls throughout 2017 and the last months.</span></p>
              <br>
              <p dir="ltr"
                style="line-height:1.38;margin-top:0pt;margin-bottom:0pt"><span style="font-size:11pt;font-family:Arial;background-color:transparent;font-variant-numeric:normal;font-variant-east-asian:normal;vertical-align:baseline;white-space:pre-wrap">The major changes are:</span></p>
              <p dir="ltr"
                style="line-height:1.38;margin-top:0pt;margin-bottom:0pt"><span style="font-size:11pt;font-family:Arial;background-color:transparent;font-variant-numeric:normal;font-variant-east-asian:normal;vertical-align:baseline;white-space:pre-wrap">1) The addition of a new business function "Implementation", covering 3 security practices: Secure Build, Secure Deployment and Defect Management.</span></p>
              <p dir="ltr"
                style="line-height:1.38;margin-top:0pt;margin-bottom:0pt"><span style="font-size:11pt;font-family:Arial;background-color:transparent;font-variant-numeric:normal;font-variant-east-asian:normal;vertical-align:baseline;white-space:pre-wrap">2) The introduction of 2 "Activity Streams" per security practice (this replaces the A and B activities which did not have a logical relation in the previous release). Each activity stream consists of 3 security activities (increasing in maturity level).</span></p>
              <br>
              <p dir="ltr"
                style="line-height:1.38;margin-top:0pt;margin-bottom:0pt"><span style="font-size:11pt;font-family:Arial;background-color:transparent;font-variant-numeric:normal;font-variant-east-asian:normal;vertical-align:baseline;white-space:pre-wrap">The current draft framework is created in a spreadsheet as this provides a better overview.</span></p>
              <p dir="ltr"
                style="line-height:1.38;margin-top:0pt;margin-bottom:0pt"><span style="font-size:11pt;font-family:Arial;background-color:transparent;font-variant-numeric:normal;font-variant-east-asian:normal;vertical-align:baseline;white-space:pre-wrap">The final framework will be released in yaml format, with an updated toolbox and document.</span></p>
              <br>
              <p dir="ltr"
                style="line-height:1.38;margin-top:0pt;margin-bottom:0pt"><span style="font-size:11pt;font-family:Arial;background-color:transparent;font-variant-numeric:normal;font-variant-east-asian:normal;vertical-align:baseline;white-space:pre-wrap">We now release the draft version for feedback, available here: </span></p>
