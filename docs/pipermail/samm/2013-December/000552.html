Hi,

I have updated the BSIMM mapping with the observed activity data and mapped
it back to SAMM activities with some Excel VBA scripting.

The updated Excel spreadsheet is available here:
https://www.owasp.org/images/f/f9/OpenSAMM-BSIMM-V-Mapping-data.xlsx

The resulting overview is available in a presentation here:
https://www.owasp.org/images/6/66/OpenSAMM_-_BSIMM-V_mapping.pptx

Some interesting results I would like to share are:

The *most observed SAMM activities *per security practice are:

   - SM Build and maintain assurance program roadmap
   - PC Establish project audit practice
   - EG Build and maintain technical guidelines
   - TA Develop attacker profile from software architecture
   - SR Evaluate security and compliance guidance for requirements
   - SA Explicitly apply security principles to design
   - DR Analyze design against known security requirements
   - CR Perform point-review of high-risk code
   - ST Conduct penetration testing on software releases
   - VM Create informal security response team(s)
   - EH Monitor baseline environment configuration status
   - OE Maintain formal operational security guides

No real surprises here, although the low number of observed developer
awareness training seems odd.

SAMM activities that are not observed are:

   - SM 3.A Conduct periodic industry-wide cost comparisons
   - TA 2.B Adopt a weighting system for measurement of threats
   - TA 3.B Elaborate threat models with compensating controls
   - SR 2.A Build an access control matrix for resources and capabilities
   - SR 2.B Specify security requirements based on known risks
   - SR 3.B Expand audit program for security requirements
   - DR 3.B Establish release gates for design review
   - CR 2.B Integrate code analysis into development process
   - VM 2.B Adopt a security issue disclosure process
   - EH 1.A Maintain operational environment specification
   - EH 1.B Identify and install critical security upgrades and patches
   - EH 3.B Expand audit program for environment configuration
   - OE 1.A Capture critical security information for deployment
   - OE 1.B Document procedures for typical application alerts
   - OE 2.A Create per-release change management procedures
   - OE 3.A Expand audit program for operational information

At first sight this seem to be a lot of SAMM activities which are not
observed as part of the 67 leading software security initiatives?
Some of these seem surprising to me: e.g. not integrating code analysis
into development process? or no identification and installation of critical
security upgrades and patches?

We should probably review the current mappings to see if these activities
are not buried in one of the BSIMM activities and update the mapping?
If we end up with a list of SAMM activities that are really not observed,
we should probably consider removing them in SAMM v2.0?

As always: your feedback and review is greatly appreciated!

Kind regards,

Seba

Thanks to BSIMM and Luis for the data and the updated mapping!
