Hi All,

I am taking this discussion in a bit different direction. We in security
hear "false positives" and "false negatives" a lot. We need a to understand
these terms in depth, as during any application security assessment exercise
we find vulnerabilities which are nothing but some sort of application
coding/configuration errors. We need to understand more about these errors.

"False positives" and "False negatives" are statistical terms for Type I or
Type II errors respectively. In statistics, the terms *Type I error* (also,
*α error*, or *false positive*) and *type II error* (*β error*, or a *false
negative*) are used to describe possible errors made in a statistical
decision process.
Type I error (False Positive)

*Type I error*, also known as an "*error of the first kind*", an *α error*,
or a "*false positive*": the error of rejecting a null hypothesis when it is
actually true. Plainly speaking, it occurs when we are observing a
difference when in truth there is none. An example of this would be if a
test shows that a woman is pregnant when in reality she is not. Type I error
can be viewed as the error of excessive credulity.

Type II error (False Negative)

*Type II error*, also known as an "*error of the second kind*", a *β error*,
or a "*false negative*": the error of failing to reject a null hypothesis
when it is in fact not true. In other words, this is the error of failing to
observe a difference when in truth there is one. An example of this would be
if a test shows that a woman is not pregnant when in reality she is. Type II
error can be viewed as the error of excessive skepticism.
For more details, please refer following link and mentioned related links:
http://en.wikipedia.org/wiki/Type_I_and_type_II_errors

> I think you are referring to actual SQL injection and XSS
> vulnerabilities that were missed by the automated scanning tool.
> Those are called as 'false negatives'.
> Tools miss plenty of them that is why it is very important that
> testers manually check the injection points.
>
> However it is also possible that testers could miss out
> vulnerabilities sometimes.
> This is why defense in depth is important.
>
> Make sure that the passwords and other sensitive information are not
> stored in clear in the database.
> The rights of the application account on the DB should be as minimal
> as possible and dangerous functions should be turned off at the DB
> level.
>
> The cookie parameters like secure flag, http-only flag, domain and
> path should be set as per recommendations.
> If possible use out of band authentication like tokens sent over SMS
> etc before users can make critical processes like making money
> transfer or changing profile information etc.
>
> Installing a WAF, DB firewall could tip you off when some large scale
> exploitation of these vulnerabilities is going on.
> Check out ESAPI, it has an Intrusion Detection module, if rightly
> tuned to your application environment it could prevent/alert on
> exploitation of vulnerabilities.
