<br><font size=2 face="sans-serif">&gt; </font><font size=3>salted MD5
hashing protects the authentication credentials rightly from eavesdropping
on the network.</font>
<br><font size=2 face="sans-serif">Salting is used to protect MD5 hashes
from rainbow table attacks. </font>
<br>
<br><font size=2 face="sans-serif">&gt; </font><font size=3>SSL does the
same thing.</font>
<br><font size=2 face="sans-serif">Yes, SSL is used to ensure confidentiality
and trust in the network communiation. Confidentiality (or prevention of
network eavesdropping) is ensured by encryption. Trust is accomplished
by Server and/or client certificates.<br>
</font>
<br><font size=2 face="sans-serif">&gt; </font><font size=3>However, in
some scenarios SSL might not be feasible. For example, causing heavy load
on the server or may be some applications don't support it, etc.</font>
<br><font size=2 face="sans-serif">In such cases SSL Accelerators are used.
They are devices which can be fit into the architecture without changing
the application or affecting the load on the application server.</font>
<br>
<br><font size=2 face="sans-serif">&gt; </font><font size=3>We can protect
the authentication credentials using salted MD5 hashing or by using SSL.</font>
<br><font size=2 face="sans-serif">To protect authentication credentials
in HTTP, we have to rely on SSL. Hashes are a secure (aka not in clear
text) way to store authentication credentials.</font>
<br>
<br><font size=2 face="sans-serif">&gt; </font><font size=3>In order to
protect the Session credentials (Session ID, tokens, cookies, etc) on a
non-SSL channel what measures can be taken?</font>
<br><font size=2 face="sans-serif">To protect either auth or session credentials
we have to ensure confidentiality of the communication channel. If we dont
use HTTPS, then VPN might be another option. While authentication credentials
can be protected by some challenge handshake mechanism (similar to CHAP),
we would need to protect session creds by encrypting the channel. </font>
