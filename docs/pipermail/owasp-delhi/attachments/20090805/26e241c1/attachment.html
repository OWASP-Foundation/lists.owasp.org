Dear Suresh,<br><br>1) Direct Object Reference Attack works as, <br><br>Most Of the web Applications often refer data stored on the web server using some kind of ID, it might be database key, Some Objet String, many times even included in hidden fields values,&nbsp; File name Of Might some other kind of Object Referrer ID which they expose to users in web page, ( for example accid, DocID,referID -- www.acme.com/account?<b>cartid=123</b>) <br><br>You can use Tamper Data Or proxies like Burp to find and replace the values.&nbsp; <br><br>Now the attacker will locate these values and try to brutforce it to see can he penetrate into other accounts or can access other data.<br><br>For example,<br><br>User 1 = www.acme.com/account?<b>cartid=123<br><br></b>Attacker = www.acme.com/account?<b>cartid=125<br><br></b>So if the attacker change his id from 125 to 123 he can access the User 1 Account and the attack will get successful. Now this is just one example there are many other ways to do it.<br><br>The main reason is because there no real access control check on those ID's . and the solution is very simple, is Never expose this ID's to users,<br><br>For example,<br><br>Instead Of www.acme.com/account?<b>cartid=123</b> just use <b>www.acme.com/account?askdjlk23423kjasjl </b><br><br>So You need to replace the ID's with the Indirect reference to that object and on the server side you need some kind of mapping which will map it with the stored ID of Authenticated user. Only the server application should now the mapping.<br><br><br>2) Broken Access Control <br><br>It is about how a web application grants access to content and functions to some
users and not others. These checks are performed after authentication,
and govern what ‘authorized’ users are allowed to do. example forcefull browesing, Access without any rights, Access to critical data without Authorization, etc...<br><br>So in simple terms - <b>Authorization - What I Can Do ?<br><br>Authentication - Who M I ?</b><br><br>Broken Access Control is one of the reason for Direct Object reference Attack.<br><br>And Patching is simple just Implement Good Access Control <img src="http://gfx2.hotmail.com/mail/w3/ltr/emoticons/smile_tongue.gif" alt="" width="19" height="19"><br><br><br><br><br><hr id="stopSpelling">Date: Wed, 5 Aug 2009 09:57:48 +0000<br>To: owasp-delhi@lists.owasp.org<br>From: sureshtiwary@rediffmail.com<br>Subject: [Owasp-delhi] Fw: "Broken Access Control" v/s "Insecure Direct        Object Reference"<br><br>Dear OWASP Delhi,<br>
