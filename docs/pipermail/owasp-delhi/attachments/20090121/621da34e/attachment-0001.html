&gt; So here is how it works.<br>
&gt;<br>
&gt; No SSL. All traffic in clear text.<br>
&gt; Initially we need to communicate an OOB password/code/nonce (whatever you<br>
&gt; call it) to the client - only *once* not each time.<br>
&gt;<br>
&gt;<br>
&gt; 1. The client initiates a connection request to the server. As a result,<br>
&gt; server will initiate a handshake process to synchronize the time-stamps<br>
&gt; between the client and itself. Then the server will analyze tolerance time<br>
&gt; of packets leaving the client workstation till it reaches the server. The<br>
&gt; tolerance time is configured so to manage slow hosts or slow traffic. In<br>
&gt; plain words, server synchronizes date and time (in seconds, milliseconds
or<br>
&gt; even nanoseconds based on the criticality of the application) with the<br>
&gt; client.<br>
&gt;<br>
&gt; 2. At the client side: the message, instantaneous time-stamp and the<br>
&gt; password are concatenated and hashed via SHA-1/SHA-256, whatever one
wishes.<br>
&gt; This ensuing hash is combined with the username to form an authentication<br>
&gt; header and sent to the server.<br>
&gt;<br>
&gt; 3. At the server side, the server sees the username and perceives which<br>
&gt; user is trying to authenticate. Based on the tolerance configured
initially<br>
&gt; at the handshake time, it picks the time-stamp from its own clock and
using<br>
&gt; the password and the message it generates the hash in the same way as it
was<br>
&gt; done at the client side.<br>
&gt;<br>
&gt; 4. If the hash matches it generates a Session ID and maps it to the<br>
&gt; username of the user. This Session ID will be sent from the client to the<br>
&gt; server along with the computed hash (which is different for each request -<br>
&gt; see next Step)<br>
&gt;<br>
&gt; 5. Now if the client sends another request, it appends and then hashes the<br>
&gt; message, password and the time-stamp (which will be apparently because of<br>
&gt; the different instantaneous time-stamp i.e. date and time). This resulting<br>
&gt; string is sent to the server along with the Session ID assigned to the
user<br>
&gt; earlier. The resulting string contains a different hash as the time has<br>
&gt; changed. This will be validated at the server the same way as in Step 3.<br>
&gt;<br>
&gt; Now how can an adversary exploit this scenario? What I think of are two<br>
&gt; ways:<br>
&gt;<br>
&gt; 1. Either he will capture one of the hashes and try to predict the next<br>
&gt; hash by brute forcing different time-stamps.<br>
&gt;<br>
&gt; - But this can not be possible. He doesn't have the password to generate<br>
&gt; the hash.<br>
&gt;<br>
&gt; 2. What if he captures a series of requests and responses and replays them<br>
&gt; back to the server at a later stage.<br>
&gt;<br>
&gt; - This can't be possible either. In order to interact with the server, one<br>
&gt; needs to handshake so to synchronize itself with the server's clock. The<br>
&gt; attacker would not be able to synchronize because he would not be able to<br>
&gt; synchronize the *past-timestamp* (in the captured traffic) with the
server's<br>
&gt; current time.<br>
&gt;<br>
&gt; 3. Lastly for the newbies, what if he captures the Session ID.<br>
&gt;<br>
&gt; - But he would not be able to produce the matching Hash for the request,
so<br>
&gt; no luck.<br>
&gt;<br>
&gt; A few things to contemplate:<br>
&gt;<br>
&gt; 1. This will be considerably be CPU intensive but can be acknowledged if<br>
&gt; compared to the processing cycles SSL takes.<br>
&gt;<br>
&gt; 2. Time synchronization can be complex. I have no idea how to do that.<br>
