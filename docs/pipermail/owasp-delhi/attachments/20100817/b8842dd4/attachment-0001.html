    This actually depends on the type of application. <br>
<br>
Is this a java applet kind of application? If so, you will not have to
worry about the &quot;transport layer insecurities&quot; because applet
communication is tunnelled. That makes it a little difficult. Also, if
it runs on SSL, then its a biot m,ore safer in terms of MITM attacks
etc.<br>
<br>
If this is going to be a desktop application or even an applest based application look for these things:<br>
<br>
0. Command Injection flaws (OS Injection, VM Injections etc)<br>
1. Logic Flaws<br>
2. Authentication &amp; Authorization Flaws<br>
3. Improper configurations<br>
4. Single/Multi- Factor Authentication<br>
5. Unsafe code execution<br>
6. Memory Leaks<br>
7. Race Conditions<br>
8. De-compilable code<br>
9. Untrusted execution<br>
10. Compliance &amp; Regulatory Laws being ignored if any<br>
<br>
These are just some of the points that I could come up with. Thinking along these lines could get you still many more ideas.<br>
