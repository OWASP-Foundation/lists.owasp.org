<br>
Around August of 2004 certain Researchers in China Published Research<br>
documents that they could hash collisions within MD5.. This was not very<br>
meaningful threat as they concentrated on creating two random blobs of<br>
data with colliding hashes. The researchers did not publish the details of<br>
their algorithm instead they published &#39;X&#39; and &#39;Y&#39; samples with colliding<br>
MD5 hashes for proving their theory.<br>
<br>
<a href="http://eprint.iacr.org/2004/199.pdf" target="_blank">http://eprint.iacr.org/2004/199.pdf</a><br>
<br>
In 2005 Dan Kaminsky created a tool called &#39;stripwire&#39;which could create<br>
certain perl scripts with colliding hashes,<br>
<br>
In early 2006 some guys figured out that they could do the same to<br>
PostScript documents.<br>
<br>
Some time later, Dan Kaminsky came back with &#39;ConFoo&#39; showing that he<br>
could do it with web pages.<br>
<br>
&#39;MD5coll&#39; is a tool you can use to create your very own &#39;X&#39; and &#39;Y&#39; with<br>
colliding hashes.<br>
<br>
Researchers have known for a very long time that if &#39;X&#39; is not equal to<br>
&#39;Y&#39; but share a hash collision then you can append &#39;Q&#39; to both &#39;X&#39; and &#39;Y&#39;<br>
and the hash collision will be preserved provided X and Y are increments<br>
of 64 bytes in length.<br>
<br>
If one uses this information very cleverly some very interesting things<br>
can be made possible. Suppose we have evil code &#39;P&#39; which opens up a<br>
backdoor, here we take &#39;X&#39; which has a collission with &#39;Y&#39;, We take SHA-1<br>
of &#39;X&#39; (this is not an attack against SHA-1 (we&#39;re using it simply to<br>
shrink it down to be able to use it as a key) to use it as an AES key to<br>
encrypt &#39;P&#39; the evil code thus getting &#39;Q&#39; (Which is the encrypted evil<br>
code &#39;P&#39;, encrypted using a key derived from &#39;X&#39; which happens to have a<br>
hash collision with &#39;Y&#39;).<br>
<br>
Now we take the encrypted evil code &#39;Q&#39; and append it to &#39;X and &#39;Y&#39;...<br>
<br>
Dan Kaminsky calls the derived &#39;X&#39;+&#39;Q&#39; and &#39;Y&#39;+&#39;Q&#39; as &#39;Fire&#39; and &#39;Ice&#39;.<br>
Ice is completely innocuous blob of bits followed by something which is<br>
encrypted, which really cant fo anything.<br>
Fire, however is evil..<br>
<br>
To make fire work you need to do the following<br>
<br>
1. Slice off the initial &#39;X&#39; from &#39;X&#39;+&#39;Q&#39;<br>
2. Generate SHA-1 of &#39;X&#39;.<br>
3. Use the derived SHA-1 as the AES key to decrypt &#39;Q&#39; and get &#39;P&#39;<br>
4. Execute &#39;P&#39;.<br>
<br>
This functionality is incorporated into StripWire and is called as &#39;Evil<br>
Execution Harness&#39;..<br>
Problem: Fire has the potential to do something evil but Fire can&#39;t do<br>
much without Ice as it is dependent on &#39;X&#39;..<br>
<br>
Research suggests that MD5 collisions can be preserved by pre-pending evil<br>
data:<br>
<a href="http://www.doxpara.com/research/md5/232-md5-considered-harmful-slides.pdf" target="_blank">http://www.doxpara.com/research/md5/232-md5-considered-harmful-slides.pdf</a><br>
<br>
If I could do pre-pending, then I could put the &#39;Evil execution Harness&#39;<br>
upfront which upon execution would start cryptographic function and<br>
depending upon the result of the cryptographic function would start<br>
branching execution either launching benign code or malicious one.<br>
<br>
ConFoo can demonstrate this very nicely, it works by taking two separate<br>
web pages, convolving them together, adding the appropriate &#39;X&#39; and &#39;Y&#39;,<br>
adding a javascript to implement the &#39;execution harness&#39; and creating<br>
pages t1 and t2 with MD5 collisions.<br>
