Hi All,<br><br>I am taking this discussion in a bit different direction. We in security hear &quot;false positives&quot; and &quot;false negatives&quot; a lot. We need a to understand these terms in depth, as during any application security assessment exercise we find vulnerabilities which are nothing but some sort of application coding/configuration errors. We need to understand more about these errors.<br>
<br>&quot;False positives&quot; and &quot;False negatives&quot; are statistical terms for Type I or Type II errors respectively. In statistics, the terms <b>Type I error</b> (also, <b>α error</b>, or <b>false positive</b>) and <b>type II error</b> (<b>β error</b>, or a <b>false negative</b>) are used to describe possible errors made in a statistical decision process.<br>
<h3><span class="mw-headline">Type I error (False Positive)<br></span></h3>
<p><b>Type I error</b>, also known as an &quot;<b>error of the first kind</b>&quot;, an <b>α error</b>, or a &quot;<b>false positive</b>&quot;: the error of rejecting a null hypothesis
when it is actually true. Plainly speaking, it occurs when we are
observing a difference when in truth there is none. An example of this
would be if a test shows that a woman is pregnant when in reality she
is not. Type I error can be viewed as the error of excessive credulity.</p>
<p><a name="Type_II_error" id="Type_II_error"></a></p>
<h3><span class="editsection"></span><span class="mw-headline">Type II error (False Negative)<br></span></h3>
<p><b>Type II error</b>, also known as an &quot;<b>error of the second kind</b>&quot;, a <b>β error</b>, or a &quot;<b>false negative</b>&quot;:
the error of failing to reject a null hypothesis when it is in fact not
true. In other words, this is the error of failing to observe a
difference when in truth there is one. An example of this would be if a
test shows that a woman is not pregnant when in reality she is. Type II
error can be viewed as the error of excessive skepticism.</p>For more details, please refer following link and mentioned related links:<br><a href="http://en.wikipedia.org/wiki/Type_I_and_type_II_errors">http://en.wikipedia.org/wiki/Type_I_and_type_II_errors</a><br>

Hi Suresh,<br>
<br>
I think you are referring to actual SQL injection and XSS<br>
vulnerabilities that were missed by the automated scanning tool.<br>
Those are called as &#39;false negatives&#39;.<br>
Tools miss plenty of them that is why it is very important that<br>
testers manually check the injection points.<br>
<br>
However it is also possible that testers could miss out<br>
vulnerabilities sometimes.<br>
This is why defense in depth is important.<br>
<br>
Make sure that the passwords and other sensitive information are not<br>
stored in clear in the database.<br>
The rights of the application account on the DB should be as minimal<br>
as possible and dangerous functions should be turned off at the DB<br>
level.<br>
<br>
The cookie parameters like secure flag, http-only flag, domain and<br>
path should be set as per recommendations.<br>
If possible use out of band authentication like tokens sent over SMS<br>
etc before users can make critical processes like making money<br>
transfer or changing profile information etc.<br>
<br>
Installing a WAF, DB firewall could tip you off when some large scale<br>
exploitation of these vulnerabilities is going on.<br>
Check out ESAPI, it has an Intrusion Detection module, if rightly<br>
tuned to your application environment it could prevent/alert on<br>
exploitation of vulnerabilities.<br>
