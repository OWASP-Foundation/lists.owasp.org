<blockquote style="border-left: 1px solid rgb(204, 204, 204); margin: 0pt 0pt 0pt 0.8ex; padding-left: 1ex;" class="gmail_quote">Can input sanitization mitigate sql injection ? If yes how and if no why ?<br></blockquote><br>

No. <br>Suppose you have a &quot;first name&quot; input field, and the user enters the text d&#39;souza. Its a valid name, so input sanitization will pass. But if you take the name and directly create a query, you will get a SQL injection because of the extra single quote.<br>

<br>In general, you may have inputs that are valid for your domain, but if you directly use them to create a query - they will cause injection. I<br><br>To prevent SQL Injection, use parameterized queries (eg. PreparedStatement). Avoid dynamically created queries; but if you must, make sure you escape users input. The reference ESAPI library provides escape methods for popular databases.<br>

<br><blockquote style="border-left: 1px solid rgb(204, 204, 204); margin: 0pt 0pt 0pt 0.8ex; padding-left: 1ex;" class="gmail_quote">For resolving xss, why output sanitization is considered ? Why not input sanitization ?<br>

</blockquote><div>Because you don&#39;t know whether the data is going to end up in HTML or Javascript or CSS or in a PDF file. Each output format has a different escape function.<br> <br></div>Consider the subject of your email &quot;SQL Injection+XSS &lt; Input Sanitization&quot;. It contains the special characters &lt;. <br>

Now, you can approach this problem in several ways. I&#39;ll try to explain the problem with each of them.<br><ol><li>Don&#39;t allow special characters - this is too restrictive, and not always possible.</li><li>Encode &lt; as &amp;lt; and store in database - Problem is that &amp;lt; only works in HTML. If you want to use the same string in Javascript or in your javacode, you will have a problem (the text will be shown as &amp;lt;). In general, you don&#39;t know where you are going to use the data, so you can&#39;t encode it at input time.</li>

</ol>The solution is to always escape the data just before you rendering the output. If you are creating HTML, escape according to HTML rules. If its javascript, escape according to javascript rules. This <a href="http://www.530geeks.com/encode.jsp">page explains how output encoding works</a>. <br>
