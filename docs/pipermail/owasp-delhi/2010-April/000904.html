> Can input sanitization mitigate sql injection ? If yes how and if no why ?
>

No.
Suppose you have a "first name" input field, and the user enters the text
d'souza. Its a valid name, so input sanitization will pass. But if you take
the name and directly create a query, you will get a SQL injection because
of the extra single quote.

In general, you may have inputs that are valid for your domain, but if you
directly use them to create a query - they will cause injection. I

To prevent SQL Injection, use parameterized queries (eg. PreparedStatement).
Avoid dynamically created queries; but if you must, make sure you escape
users input. The reference ESAPI library provides escape methods for popular
databases.

For resolving xss, why output sanitization is considered ? Why not input
> sanitization ?
>
Because you don't know whether the data is going to end up in HTML or
Javascript or CSS or in a PDF file. Each output format has a different
escape function.

Consider the subject of your email "SQL Injection+XSS < Input Sanitization".
It contains the special characters <.
Now, you can approach this problem in several ways. I'll try to explain the
problem with each of them.

   1. Don't allow special characters - this is too restrictive, and not
   always possible.
   2. Encode < as &lt; and store in database - Problem is that &lt; only
   works in HTML. If you want to use the same string in Javascript or in your
   javacode, you will have a problem (the text will be shown as &lt;). In
   general, you don't know where you are going to use the data, so you can't
   encode it at input time.

The solution is to always escape the data just before you rendering the
output. If you are creating HTML, escape according to HTML rules. If its
javascript, escape according to javascript rules. This page explains how
output encoding works <http://www.530geeks.com/encode.jsp>.
