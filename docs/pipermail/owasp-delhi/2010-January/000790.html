For more clarity:  

 

Proper input validation can ensure that malicious data is not inserted into
the system. However, it fails to provide the assurance that validated data
will remain consistent throughout its lifetime. For instance, if an insider
is allowed to insert data into a database without validation, it is possible
to glean unauthorized information or execute arbitrary code on the client
side by means of attacks such as Cross Site Scripting (XSS). Consequently,
output filtering is as important as input validation.

 

To ensure that any data that bypasses the validation does not cause
vulnerabilities, it is highly recommended that output characters be encoded,
except those that are known to be safe.

 

Here is an example of java code..

 

public class BadOutput {

  // description and input are String variables containing values obtained
from a database

  // description = "description" and input = "<script> executable code
</script>"

  public static void display(String description, String input) {

    // Display to the user or pass description and input to another system

  }

}

 

 

Example displays input obtained from a database directly to the user without
performing any output validation or encoding.

Well I'm no expert on XSS, but generally I'd prefer input sanitization.

 

Consider the following:

1 with multiple interfaces (outputs) its easier and more secure to clean it
at the source before persistence.

 

2 Many times the interfaces or outputs change. Developers may forget
security constraints over time and buld sexy UIs that disregard earlier
sanitization code.

 

3. Unsanitized data sitting in your DB can be atime bomb.

 

The points are correlated, but gives you a perspective why I'd rate input
better but it depends on your scenario.

 

The only down of input sanitization is u gotta be careful how to do it;
tougher than escaping XML on output. But theb again that's what OWASP
AntiSamy tools are for :)
