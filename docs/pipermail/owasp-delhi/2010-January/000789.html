Perfectly explained!!!

 

Just want to add few notes on it...

 

1.       If you want to restrict user to enter unwanted inputs then you can
go for Input Validation other wise use Input Sanitization. Only difference
is whether you want to show a note "Rights of Admission Reserved" or not
:-)

2.       Well...In my opinion both should be implemented and Input
sanitization must occur before validation if it is impossible to validate
unsanitized data.

3.       White-listing approach should be used in accepting valid user
inputs, both at client and server side.

4.       Now about output ..if you are capable to validate all source of
inputs and you are 100% confident then actually it is not required but as
said below very accurately "Unsanitized data sitting in your DB can be a
time bomb" so just for safe side output encoding is recommended.

Well I'm no expert on XSS, but generally I'd prefer input sanitization.

 

Consider the following:

1 with multiple interfaces (outputs) its easier and more secure to clean it
at the source before persistence.

 

2 Many times the interfaces or outputs change. Developers may forget
security constraints over time and buld sexy UIs that disregard earlier
sanitization code.

 

3. Unsanitized data sitting in your DB can be atime bomb.

 

The points are correlated, but gives you a perspective why I'd rate input
better but it depends on your scenario.

 

The only down of input sanitization is u gotta be careful how to do it;
tougher than escaping XML on output. But theb again that's what OWASP
AntiSamy tools are for :)
