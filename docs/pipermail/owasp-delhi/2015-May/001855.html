> The very purpose of salted hash implementation is to avoid transport of
> plain text password from client side to the server with/without HTTPS.
>
> The salted hash password is computed on client side and sent to the server
> which is compared with the computed (with hashed password retrieved from
> DB) salted hash on the server side. If the salted hash from client compares
> ok with salted hash on server side logic then  login is successful.
>

> > small correction
> >
> > 1.The user submits the UID & Password through HTTPS-POST method, which
> is received by Web/AppServer - at the time of registration
> > 2.Application logic takes the "plain password" and generates an output
> based on Keyed/Salted-HMAC algorithm & stores it in DB
> > 3.When user wants to authenticate - user submits the UID & Password
> through HTTPS-POST method, which is received by Web/AppServer
> > 4a. Application logic takes the "plain password" and generates an
> output based on Keyed/Salted-HMAC algorithm
> > 4b. Application logic loads the corresponding HMACed password of that
> specific user from DB & compares the generated HMAC & Stored HMAC
> > 4c. If both matches then login is successful - else authentication
> failed message is shown.
> >
> > Advantages:
> > 1. Internal employees/DB admins cannot use the stored HMACs to login to
> a user's account
> > 2. BruteForce or replay attacks can be avoided
> > 3. Rainbow tables can be avoided
