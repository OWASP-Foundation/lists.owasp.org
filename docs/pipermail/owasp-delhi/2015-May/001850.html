> small correction
>
> 1.The user submits the UID & Password through HTTPS-POST method, which is
> received by Web/AppServer - at the time of registration
> 2.Application logic takes the "plain password" and generates an output
> based on Keyed/Salted-HMAC algorithm & stores it in DB
> 3.When user wants to authenticate - user submits the UID & Password
> through HTTPS-POST method, which is received by Web/AppServer
> 4a. Application logic takes the "plain password" and generates an output
> based on Keyed/Salted-HMAC algorithm
> 4b. Application logic loads the corresponding HMACed password of that
> specific user from DB & compares the generated HMAC & Stored HMAC
> 4c. If both matches then login is successful - else authentication failed
> message is shown.
>
> Advantages:
> 1. Internal employees/DB admins cannot use the stored HMACs to login to a
> user's account
> 2. BruteForce or replay attacks can be avoided
> 3. Rainbow tables can be avoided

> Salt is used in web app and also in Unix password files.
>
> 1.In webapp
>
> In web applications salted hash is used to secure the password from replay
> attack.
>
> Salt or any random number is generated at server and sent to browser.
> There hash is derived using the hash of password plus salt.
>
> Hash(hash(password))+salt)
>
> This is sent on net and received by webserver.
>
> Server already knows your salt . It gets the password or hash of password
> from database table. Then does operation. Hash(hash(password)+salt) it
> compares it with received hash from client ie browser.
>
> Does the authentication as usual.
>
> 2. In Unix password file salt is part of the password. See net for that.
