Understood the confusion & here is my gentle response with some corrections

1] Salted Hash are not generated on client side - WHY ?

    a] To generate salted hash on client side - you need to send the "salt or the key" to client as part of web page

         which is a bad practice in a web application - whether you use http or https(ssl)

    b] an attacker can steal the "salt" and conduct rainbow attacks & you should never send your secrets out of your server

    c] if HTTPS is not used then a man-in-middle attack can capture the "HMAC" output and user-id and replay the same

        thereby account takeover can happen

2] The Salted HMAC implementation that you were referring might be happening in "Unix/Linux" environment and not in

     web application systems

3] On unix/linux environments this is possible since the keys or salts are pre-shared as part of "Installation/Setup" of domain logins

     or network login setup

4] In Web Application the Salted HMAC doesn't work like this

5] Whether it is Paypal/Ebay/Amazon web apps all store their passwords as an output of Keyed/Salted HMACs

6] Just check the login page of any webpage -- you'll never find any HMAC Code(Javascript) in the login page


Hope this helps.

Dear Sir/MadameThe very purpose of salted hash implementation is to avoid transport of plain text password from client side to the server with/without HTTPS.The salted hash password is computed on client side and sent to the server which is compared with the computed (with hashed password retrieved from DB) salted hash on the server side. If the salted hash from client compares ok with salted hash on server side logic then  login is successful.For information.Thanks

Application logic takes the "plain password" and generates an output based on Keyed/Salted-HMAC algorithm> 4b. Application logic loads the corresponding HMACed password of that specific user from DB & compares the generated HMAC & Stored HMAC> 4c. If both matches then login is successful - else authentication failed message is shown.>
