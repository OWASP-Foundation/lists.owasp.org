<br>
</span>It depends on your threat model.<br>
<br>
The net-net is:<br>
<br>
You and your customers can't accidentally end up using HTTP instead of<br>
HTTPS.<br>
<br>
The details are:<br>
<br>
That means cookies which you failed to set the "secure" (?!?) flag on<br>
don't accidentally get sent via HTTP.<br>
<br>
They don't accidentally send a password or sensitive information over<br>
HTTP via an "unsecure" form, possibly via a redirect or something<br>
where the form action is autogenerated.<br>
<br>
You can avoid accidentally sending a login cookie or something in<br>
an unencrypted response.<br>
<br>
An active MITM can spoof HTTP/TCP/80 without talking to the real site,<br>
clearly.<br>
<br>
A MOTS can only race against the real site which probably sends back a<br>
TCP RST.<br>
<br>
Passive attackers are SOL if you don't ever do HTTP/TCP/80.<br>
<br>
You can easily splice into HTTP requests via dsniff and whatnot<br>
to insert a fake HSTS - it's harder if they never use HTTP in the<br>
first place and you never allow it.<br>
<br>
There's probably a dozen other minor implications and scenarios that<br>
would need to be run through to generate a proper analysis, including<br>
interactions with user controllable content, XSS, CSRF, header<br>
splicing and header injection, but this is all you get for free from<br>
me after 2200 on a worknight. :-)<br></blockquote><div><br></div><div>I suppose I didn't consider "accidentally" using HTTP.  Given that I was considering any mechanism that does not protect against a MITM as flawed, I still see no benefit, but I do suppose there is a benefit against e.g., the passive attacker.  Not serving on port 80 does come with a usability cost though.  Many users will not type in the protocol and will be surprised when it doesn't work.</div><div><br></div><div>Some of the concerns you listed could be mitigated by a well-designed framework: redirecting all requests to HTTPS won't protect against the data sent by the user, but it can prevent sending the login cookie in the response, and your framework can ensure the secure attribute is set on cookies.</div><div> </div><blockquote class="gmail_quote" style="margin:0 0 0 .8ex;border-left:1px #ccc solid;padding-left:1ex">
