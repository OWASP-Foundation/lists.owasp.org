Thoughts?<BR>&gt;&nbsp;&nbsp; 1.. Http Basics -- use this one to demonstrate 
that HTTP is stateless, request-reply protocol. Click show parameters to show 
what's going with each request.<BR>&gt;&nbsp;&nbsp; 2.. Fail Open Authentication 
-- try removing the password parameter entirely using Achilles.&nbsp; Or you can 
do it on the URL with (<A 
href="http://localhost:8080/WebGoat/attack?Username=jeff">http://localhost:8080/WebGoat/attack?Username=jeff</A>) 
-- when there's no password, an exception gets thrown and the authentication 
step gets skipped.<BR>&gt;&nbsp;&nbsp; 3.. HTML Clues -- view HTML and search 
for a comment with admin:adminpw -- use this to authenticate<BR>&gt;&nbsp;&nbsp; 
4.. Parameter Injection -- show how shell calls can be used to run arbitrary 
code -- use the &amp; character to chain commands on windows -- blah.txt &amp; 
netstat -a &amp; ipconfig<BR>&gt;&nbsp;&nbsp; 5.. Unchecked Email -- enter a 
message that contains linked offensive content -- Hi, check this out....&lt;A 
href="<A href='http://www.google.com">Click'>http://www.google.com"&gt;Click</A> 
here for Google"&lt;/A&gt; and it looks like it came from the 
website.<BR>&gt;&nbsp;&nbsp; 6.. SQL Injection -- 101' or 1=1 to get the whole 
contents of the db<BR>&gt;&nbsp;&nbsp; 7.. Thread Safety -- this one's tricky -- 
use 'new window' to get two browsers.&nbsp; Enter 'dave' in one window and 
'jeff' in the other.&nbsp; Then set it up so you click on the submit boxes as 
fast as possible.&nbsp; If you do it fast enough, the application gets confused 
and returns the second clicked account to both browsers.&nbsp; This is a 
massively underreported problem in many many web apps.<BR>&gt;&nbsp;&nbsp; 8.. 
Weak Authentication Cookie -- use achilles to add 'AuthCookie=ggfkggfk;' to the 
Cookie: header using Achilles. Note that the cookie is username:password 
(jeff:jeff) backwords and one-letter up.<BR>&gt;&nbsp;&nbsp; 9.. Database XSS -- 
enter &lt;script language="javascript" type="text/javascript"&gt;alert("Ha Ha 
Ha");&lt;/script&gt; in the message and then click on the name of the message 
below to show that anyone who reads the message is attacked<BR>&gt;&nbsp;&nbsp; 
10.. Hidden Field Tampering -- there are a number of ways to edit a hidden 
field.&nbsp; I like IEBooster, but you can use Achilles, or even try to modify 
it in the URL (e.g. <A 
href="http://localhost:8080/WebGoat/attack?Price=9.99">http://localhost:8080/WebGoat/attack?Price=9.99</A>).&nbsp; 
Change the price of the TV from 4999.99 to 9.99.&nbsp; <BR>&gt;&nbsp;&nbsp; 11.. 
Weak Access Control -- this page is supposed to let you access files in a 
certain directory.&nbsp; But if you use ../topsecret.txt you can get other 
files.&nbsp; Would be nice to enhance this with some url, hex, unicode encoding 
attacks.<BR>&gt;&nbsp;&nbsp; 12.. Challenge 1 -- the challenge is tricky.&nbsp; 
To get past the first page, you have to remember how to turn on showing of the 
java code. Check the lesson plan link to find out that the lesson plan is called 
ChallengeScreen.html.&nbsp; Then force browse to 
/WebGoat/src/lessons/ChallengeScreen.java.&nbsp; Find the username and password 
(youaretheweakestlink:goodbye).&nbsp; Note: there is another option -- if you 
sniff the network, you'll see a UDP packet with the username and password in the 
clear.&nbsp; Cool huh?<BR>&gt;&nbsp;&nbsp; 13.. Challenge 2 -- you'll need to 
modify the 's' hidden field to contain 101' or 1=1 (like the database lesson). 
Use Achilles or IEBooster.&nbsp; This will give you all the 
passwords.<BR>&gt;&nbsp;&nbsp; 14.. Challenge 3 -- you'll need to know a bit 
about Server Side Includes here -- syntax is &lt;!--#exec tag1="dir" 
--&gt;&nbsp; So you'll want to do something like &lt;!--#exec tag1="echo defaced 
&gt; [PATH]\index.html" --&gt;.&nbsp; This will overwrite the index.html with 
your content and win the challenge.<BR>&gt; --Jeff<BR>&gt;
