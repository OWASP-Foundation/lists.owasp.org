The phishing with XSS solution does work.  You may have to remove the "." after localhost.  I don't have acesss to WebGoat right now to verify but will do so soon.

You are correct about the reflected XSS trace.  That only works on older browsers and should be removed from WebGoat.

As far as checking for a successful XSS attack, it is very difficult to do unless the attack posts to the catcher servlet.  The catcher servlet was introduced in 5.1 for the phishing lesson, however it is unreasonable to have the user craft such a difficult XSS for some of the simpler XSS lessons.  That is why we  check for the word 'script', not the most effective way of detecting a "valid" attack but it is the simplest.  I'm open to better solutions... And yes I realize that simly having the word 'script' does not constitute a valid attack.

Hi Daniel,

I had the exact same problem on Kubuntu 7.10.

I got rid of the error messages by modifying
'/etc/tomcat5.5/policy.d/04webapps.policy':
grant codeBase
"file://var/lib/tomcat5.5/webapps/WebGoat/WEB-INF/classes/org/owasp/webgoat/session/*"
{
    permission java.lang.RuntimePermission "getenv.*";
};

You can't edit '/etc/tomcat5.5/catalina.policy' directly because it's
auto-generated by the files in '/etc/tomcat5.5/policy.d/*'.

That fixed the exception but then I got an "Invalid session" error when
accessing the WebGoat home page so, after almost a day spent on that
(somebody can jump in here if they know the answer to this problem), I said
screw it and turned off Java Security Manager in Tomcat.

I rolled back the change to '04webapps.policy' and turned off Security
Manager in the file '/etc/default/tomcat5.5' by setting
'TOMCAT5_SECURITY=no' and it works fine ever since.

Of course the web server shouldn't be Internet-facing for WebGoat so it
shouldn't be a big deal. I have a LAN using a wireless router and the
easiest way for me is that in the router configuration there is a "Block
Services" section that I set for ports 80,443, & 8080 on that server (a 6 yr
old Dell laptop). It's dynamic - no reboot of the router necessary - so it's
easy to toggle on and off when I need the server to access the Internet.

Stephen

> I had the exact same problem on Kubuntu 7.10.
>
> I got rid of the error messages by modifying
> '/etc/tomcat5.5/policy.d/04webapps.policy':
> grant codeBase
> "file://var/lib/tomcat5.5/webapps/WebGoat/WEB-INF/classes/org/owasp/webg
> oat/session/*"
> {
>    permission java.lang.RuntimePermission "getenv.*";
> };
>
> You can't edit '/etc/tomcat5.5/catalina.policy' directly because it's
> auto-generated by the files in '/etc/tomcat5.5/policy.d/*'.
>
> That fixed the exception but then I got an "Invalid session" error when
> accessing the WebGoat home page so, after almost a day spent on that
> (somebody can jump in here if they know the answer to this problem), I
> said
> screw it and turned off Java Security Manager in Tomcat.
>
> I rolled back the change to '04webapps.policy' and turned off Security
> Manager in the file '/etc/default/tomcat5.5' by setting
> 'TOMCAT5_SECURITY=no' and it works fine ever since.
>
> Of course the web server shouldn't be Internet-facing for WebGoat so it
> shouldn't be a big deal. I have a LAN using a wireless router and the
> easiest way for me is that in the router configuration there is a "Block
> Services" section that I set for ports 80,443, & 8080 on that server (a
> 6 yr
> old Dell laptop). It's dynamic - no reboot of the router necessary - so
> it's
> easy to toggle on and off when I need the server to access the Internet.
