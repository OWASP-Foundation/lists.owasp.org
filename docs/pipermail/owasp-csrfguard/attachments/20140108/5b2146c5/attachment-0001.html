<blockquote class="gmail_quote" style="margin:0 0 0 .8ex;border-left:1px #ccc solid;padding-left:1ex">Hello Ragha,<br>
<br>
I recently implemented CSRFGuard in an application using IBM WebSphere 8.0. There are some changes that need to be made to the code in order to get it to work with the WebSphere server. I found that the Installation Guide and other documentation on the OWASP site does not address this.<br>

<br>
In particular, in the csrfguard.properties file, I had to change the org.owasp.csrfguard.PRNG property<br>
From: org.owasp.csrfguard.PRNG =SHA1PRNG<br>
To: org.owasp.csrfguard.PRNG=IBMSecureRandom<br>
<br>
And in the CsrfGuard.java file, defaults are hard-coded for the following two values, which have to be changed for WebSphere servers<br>
From: org.owasp.csrfguard.PRNG=SHA1PRNG<br>
To: org.owasp.csrfguard.PRNG=IBMSecureRandom<br>
---<br>
From: org.owasp.csrfguard.PRNG.Provider=SUN<br>
To: org.owasp.csrfguard.PRNG.Provider=IBMJCE<br>
---<br>
<br>
Because of the change to the .java file, the code had to be recompiled. The OWASP CSRFGuard Install Guide shows how to list the CSRFGuardListener class in its example web.xml file. And the CSRFGuard.jar file on my company's application security web site contains this class file. But I was not able to find this java code anywhere. So instead, I used the CsrfGuardServletContextListener and CsrfGuardHttpSessionListener classes, and put them in my web.xml as follows:<br>

    <listener><br>
       <listener-class>org.owasp.csrfguard.CsrfGuardServletContextListener</listener-class><br>
    </listener><br>
    <listener><br>
       <listener-class>org.owasp.csrfguard.CsrfGuardHttpSessionListener</listener-class><br>
    </listener><br>
<br>
I would think that it would be possible to put both properties in csrfguard.properties and forgo the recompile of the Java code, but I didn't try this myself. It would look like this:<br>
org.owasp.csrfguard.PRNG=IBMSecureRandom<br>
org.owasp.csrfguard.PRNG.Provider=IBMJCE<br>
<br>
This is where info is on these properties for WebSphere:<br>
<a href="http://files.jscape.com/secureftpserver/docs/index.html?running_under_ibm_jvm.htm" target="_blank">http://files.jscape.com/secureftpserver/docs/index.html?running_under_ibm_jvm.htm</a><br>
<br>
Since I only needed to protect a couple pages in the application I worked on, I found it much simpler to use the provided JSP Tag Library, csrfguard.tld, and insert tokens into specific links. For me, this was preferable to using the default JavaScript token injection, which for my application, which has many links and older and faulty JavaScript code, surfaced a lot of previously unseen JS errors.<br>

<br>
Feel free to contact me if you have additional questions about CSRFGuard in WebSphere.<br>

Dear Sir,<br>
I ma the Leader of this project.<br>
You can use the CSRFGuard library with IBM WS as JavaEE filter.<br>
<br>
OWASP CSRFGuard utilizes request tokens to address Cross-Site Request Forgery.  CSRF is an attack where the victim is tricked into interacting with a website where they are already authenticated.<br>
CSRFGuard Provides code to generate unique request tokens to mitigate CSRF risks.<br>
<br>
You can integrate the source code using you own method (if you don't use Maven).<br>
<br>
The following link show how to integrate CSRFGuard with you application :<br>
<br>
<a href="https://www.owasp.org/index.php/CSRFGuard_3_Installation" target="_blank">https://www.owasp.org/index.php/CSRFGuard_3_Installation</a><br>
<br>
