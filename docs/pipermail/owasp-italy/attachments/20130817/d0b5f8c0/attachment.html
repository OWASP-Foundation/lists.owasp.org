<div><br></div><div>Prima di tutto una nota per Lorenzo e Paolo. Le password che dobbiamo proteggere non sono quelle degli utenti, ma quelle "applicative", usate dalle applicazioni per autenticarsi su altre applicazioni. Per esempio se ho un gestore documentale che deve usare un servizio di marcatura temporale, il gestore deve utilizzare una user-id e una password per farsi riconoscere dal servizio di marcatura.</div>
<div><br></div><div>Per Giorgio e Nicola: abbiamo pensato anche noi ad una soluzione basata su HSM, anche perché usiamo già dispositivi simili in azienda e anche pesantemente. Ma per interfacciarsi ad un HSM c'è bisogno di un PIN, quindi il problema è solo spostato: proteggo una chiave con un'altra chiave che non so come proteggere.</div>
<div><br></div><div>Per Luca: fornire la password al boot dell'applicazione? Purtroppo per noi non è applicabile. Abbiamo esigenze molto alte di disponibilità dei servizi che ha portato al proliferare di istanze (anche 30 per una singola applicazione) che si riavviano al primo cenno di problema (a qualunque ora del giorno e della notte). Non possiamo dedicare una persona all'inserimento delle password ad ogni riavvio.</div>
<div><br></div><div>Per Canaro: anche noi abbiamo scelto di cifrare i file di configurazione: è la soluzione attualmente in essere. E anche noi abbiamo scelto di annegare la chiave di (de)cifratura nel codice. Peccato che il codice vada sul repository dei sorgenti, in cui è leggibile da tutto il gruppo di sviluppatori e amministratori. E' troppo esposta e dobbiamo trovare un'altra soluzione.</div>
<div><br></div><div>Per Antonio: interessante JDPAPI. Ma noi abbiamo solo sistemi Linux. Ma una cosa che ho chiesto ai nostri sistemisti è vedere se esiste qualcosa di simile anche in Linux. O più generalmente: quali sono i servizi di sicurezza che un Linux può mettere a disposizione delle nostre applicazioni?</div>
<div><br></div><div>Per Luca: anche JSVC è interessante. Ne terremo sicuramente conto anche se, come leggerai sotto, stiamo puntando ad un'altra strada.</div><div><br></div><div>Per Giorgio: anche il sandboxing dell'application server è un tema interessante, su cui però siamo completamente a digiuno. Sai darmi qualche dritta per gli application server Java ed in particolare JBoss? Grazie!!!</div>
<div><br></div><div>Ora. Una veloce presentazione di cosa stiamo pensando adesso. Ogni vostra opinione è gradita.</div><div><br></div><div>1) Abbandonare la configurazione su file e portarla, per esempio, su un Directory Server (la struttura ad albero si sposa bene con le configurazioni e avendo un server centrale possiamo pensare ad un configurazione servizio-centrica e non applicazione-centrica).</div>
<div>2) Configurare il Directory Server in modo che sia possibile entrarci solo tramite LDAP/S con mutua autenticazione di certificato.</div><div>3) Fare in modo che non sia l'applicazione ad autenticarsi al Directory Server, bensì il sistema operativo. In questo modo la chiave privata sarebbe protetta dall'OS e non si necessitano altri archivi o file di configurazione in cui metterla. Sempre che l'OS abbia simili strumenti...</div>
<div>4) Per modificare le configurazioni predisponiamo un'applicazione che richiede un'autenticazione (forse forte... stiamo valutando). Quando poi si inserisce la password per un servizio, questa non viene memorizzata così com'è. L'applicazione fa un cambio password impostando una password molto robusta (lunga e con un'ampia gamma di caratteri) che verrà salvata sul Directory Server e non mostrata all'operatore. In questo modo la password sarebbe conosciuta solo dal servizio, dal Directory Server e dall'applicazione e, a meno di vulnerabilità di quest'ultima, non dovrebbe mai uscire da questa triade. E la password iniziale che l'operatore aveva non è più valida.</div>

<br>
- in un file di configurazione, ovvero in chiaro sulle macchine su cui<br>
gira l'applicazione -> non è accettabile, perché se un attaccante prende<br>
il controllo della macchina o riesce ad accedere al file può leggere il<br>

- annegate nel codice dell'applicazione -> non è accettabile<br>

sviluppatori potrebbero leggere tali password; inoltre essendo le nostre<br>
applicazioni scritte in Java (e non potendo offuscare tutti i binari) è<br>
