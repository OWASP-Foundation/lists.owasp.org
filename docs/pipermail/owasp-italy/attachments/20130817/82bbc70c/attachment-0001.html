<html><head><meta http-equiv="content-type" content="text/html; charset=utf-8"></head><body dir="auto"><div>La tematica è interessante e si presta a delle riflessioni estive sotto l'ombrellone.</div><div><br></div><div>Concordo praticamente con tutto quello che è stato detto.</div><div><br></div><div>Come diceva un mio collega, "non si può proteggere il software con il software" e per quanto possa essere sofisticato il sistema di protezione adottato (molti dei quali citati nelle mail precedenti), l'ultimo anello della catena deve essere necessariamente hardware. Per un server, questo significa un hsm. Anche se l'adozione di hsm comporta una complessità notevole: al momento del boot le chiavi devono essere sbloccate da uno o più amministratori inserendo delle smartcard e digitando un pin (cosa che in un contesto enterprise dove la continuità è fondamentale può costituire un problema); per non parlare che l'applicativo si deve integrare con l'hsm per la gestione delle chiavi e delle operazioni di cifratura. Per non parlare degli aspetti legislativi legati alla validità della firma digitale...</div><div><br></div><div>È chiaro quindi che una simile complessità è giustificata se l'applicativo tratta informazioni critiche per il business, altrimenti è come sparare con un cannone ad una formica. Un conto è realizzare una CA, un'altro un serverino con informazioni pubbliche per il quale è sufficiente un hardening standard. L'analisi dei rischi è sempre la prima cosa da fare, per evitare di mettere in campo controlli di sicurezza troppo complessi che poi vengono utilizzati male, creando in più negli utenti un falso senso di sicurezza. Personalmente ho visto hsm con smartcard sempre inserite e senza pin: il che vuol dire che l'applicativo tratta materiale cifrato con password blank... Un classico esempio dove l'analisi dei rischi avrebbe aiutato!</div><div><br></div><div>Alla fine penso che il modo migliore per trattare le password (che statisticamente sono sempre deboli o prevedibili...) è ove possibile di evitarle, delegandone la gestione ad un sistema esterno (vedi iam o l'autenticazione integrata di Windows).</div><div><br></div><div>Bye</div><div><br></div><div>L.<br><br>Inviato da iPhone</div><div><br>Il giorno 16/ago/2013, alle ore 08:39, antonio parata <<a href="mailto:aparata@gmail.com">aparata@gmail.com</a>> ha scritto:<br><br></div><blockquote type="cite"><div><div dir="ltr"><div><div><div><div>Ciao Claudio,<br><br></div>premetto che sono un windows guy quindi il mio consiglio potrebbe non trovare un equivalente in altri OS. Ti consiglio di dare un occhio alle api DPAPI (<a href="http://msdn.microsoft.com/en-us/library/ms995355.aspx">http://msdn.microsoft.com/en-us/library/ms995355.aspx</a>) utilizzate proprio per proteggere dati utente, così come scritto nella documentazione:<br>

<div dir="ltr">Ciao a tutti,<div>cercando in giro ho capito che il tema è sentito, ma sinceramente non ho trovato una risposta valida che sia una.</div><div>Parto dall'inizio.</div><div><br></div><div>Per lo più la mia azienda si occupa dello sviluppo di servizi web. Quindi web application Java che esponiamo in internet a cui i clienti accedono, si autenticano, lavorano, etc etc etc.</div>

<div>I nostri servizi hanno spesso bisogno di utilizzare altri servizi: alcuni interni altri esterni. Ma è normale, per accedere ad un servizio, che sia necessario utilizzare una password o una chiave privata (nel senso più generico del termine).</div>

<div>Il problema che stiamo cercando di risolvere è: dove è sicuro scrivere tali password/chiavi?</div><div><br></div><div>Alcune risposte sono state:</div><div><br></div><div>- in un file di configurazione, ovvero in chiaro sulle macchine su cui gira l'applicazione -> non è accettabile, perché se un attaccante prende il controllo della macchina o riesce ad accedere al file può leggere il file di configurazione e leggere pertanto la password;</div>

<div><br></div><div>- annegate nel codice dell'applicazione -> non è accettabile perché verrebbero scritte nei repository dei sorgenti e quindi tutti gli sviluppatori potrebbero leggere tali password; inoltre essendo le nostre applicazioni scritte in Java (e non potendo offuscare tutti i binari) è come scrivere le stesse password in chiaro su un file di configurazione;</div>

<div><br></div><div>- in un file di configurazione ma cifrate -> non è accettabile: per cifrare serve una chiave e dove scrivo la chiave? il problema è solo stato spostato;</div><div><br></div><div>- in un archivio (database relazionale, directory service, ...) accessibile solo da pochi: ma per accedere all'archivio avrei bisogno di una password, e siamo di nuovo al punto di prima.</div>

<div><br></div><div>Immagino che questo problema non sia solo della nostra azienda. E, come dicevo, ho trovato altri che hanno posto domande simili. Ma non ho trovato risposte valide.</div><div>Voi cosa ne dite? Come avete risolto voi questo problema?</div>
