<div dir="ltr">Even if you have a single client and a single resource server, you can still benefit from REST APIs. My example would be building a single page web app. By using a REST API you can separate frontend from backend properly and deploy them independently. You can completely rewrite the frontend with no changes to the backend. But why does that imply the need for an STS/IdP? Why is sessions a problem here?<div>Resources can still work as they are supposed to, without depending on session state. The session is for authentication information only. It carries authentication information, but that informationÂ doesn't affect the way resources are processed (except for the auth, but that holds for basic auth, tokens, and all other types of authentication as well).</div><div><br></div><div>Is it possible to design an STS/IdP with a REST API?</div><div><div><br></div></div></div><div class="gmail_extra"><br><div class="gmail_quote">On 14 November 2017 at 10:56, Johan Peeters <span dir="ltr"><<a href="mailto:yo@johanpeeters.com" target="_blank">yo@johanpeeters.com</a>></span> wrote:<br><blockquote class="gmail_quote" style="margin:0 0 0 .8ex;border-left:1px #ccc solid;padding-left:1ex">No sooner did Jim move the REST cheat sheet from draft to approved<br>

Here I want to discuss what the deal is with Security Token Services (STS).<br>
Erlend asked me: can't you have a REST service without an STS? Sure,<br>
if no authN/Z is needed, obviously. If the REST API is sensitive,<br>
serves multiple clients and there are several resource servers, on the<br>
other hand, then I would strongly recommend using an STS. This<br>
centralises authentication logic in a dedicated component and avoids<br>
passing credentials around.<br>
If you do not have and do not anticipate having multiple clients or<br>
resource servers, maybe REST APIs are not the right choice - probably<br>
there are simpler designs that serve your purpose.<br>
