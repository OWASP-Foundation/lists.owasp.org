    <div class="moz-cite-prefix">Dear Kenneth,<br>
      <br>
      thank you for your mail and your points for discussion.<br>
      <br>
      About your question: DHE versus ECDHE:<br>
      <ul>
        <li>DHE and ECDHE, both do support Forward Secrecy. So they
          should be offered by a server with hightest priority.</li>
        <ul>
          <li>"Highest priority for ciphers that support 'Forward
            Secrecy' (-> Support ephemeral Diffie-Hellman key
            exchange, see rule above)."</li>
        </ul>
        <li>But there is still the need to start with one cipher. There
          have been a lot of discussions about the reliability of
          Elliptic Curves, and finally two Curves are about to be
          standardized for TLS
(<a class="moz-txt-link-freetext" href="https://datatracker.ietf.org/doc/draft-ietf-tls-rfc4492bis/?include_text=1">https://datatracker.ietf.org/doc/draft-ietf-tls-rfc4492bis/?include_text=1</a>).
          Until now, only some crypto libraries support them and some of
          them have had vulnerabilities by implementing them, e.g.
          <a class="moz-txt-link-freetext" href="http://www.cvedetails.com/cve/CVE-2017-0379/">http://www.cvedetails.com/cve/CVE-2017-0379/</a> <br>
          So I'd still favor DHE over ECDHE in general, but it's only a
          fine nuance:Â  </li>
      </ul>
      <blockquote>
        <ul>
          <ul>
            <li>"Favor DHE over ECDHE (and monitor the CPU usage, see
              Notes below), ECDHE lacks now of really reliable Elliptic
              Curves, see discussion about secp{224,256,384,521}r1 and
              secp256k1, cf. <a rel="nofollow" class="external
                autonumber" href="http://safecurves.cr.yp.to">[3]</a>, <a
                rel="nofollow" class="external autonumber"
href="https://www.schneier.com/blog/archives/2013/09/the_nsa_is_brea.html#c1675929">[4]</a>.
              The solution might be to use <a rel="nofollow"
                class="external text"
href="http://www.researchgate.net/profile/Johannes_Merkle/publication/260050106_Standardisierung_der_Brainpool-Kurven_fr_TLS_und_IPSec/file/60b7d52f36a0cc2fdd.pdf">Brainpool
                Curves [German]</a>, defined for TLS in <a
                rel="nofollow" class="external text"
                href="https://tools.ietf.org/html/rfc7027">RFC 7027</a>,
              or <a rel="nofollow" class="external text"
                href="http://eprint.iacr.org/2007/286">Edwards Curves</a>.
              The most promising candidates for the latter are <a
                rel="nofollow" class="external text"
                href="https://tools.ietf.org/html/draft-josefsson-tls-curve25519-06">'Curve25519'</a>
              and <a rel="nofollow" class="external text"
                href="http://sourceforge.net/p/ed448goldilocks/wiki/Home/">Ed448-Goldilocks</a>
              (see <a rel="nofollow" class="external text"
                href="https://tools.ietf.org/html/rfc7748">RFC 7748 -
                Elliptic Curves for Security</a>), that are about to be
              defined by RFC for TLS, cf. <a rel="nofollow"
                class="external text"
                href="https://tools.ietf.org/html/draft-ietf-tls-rfc4492bis-11">DRAFT-ietf-tls-rfc4492bis</a>
              and <a rel="nofollow" class="external text"
href="http://www.iana.org/assignments/tls-parameters/tls-parameters.xhtml#tls-parameters-8">IANA
                (temporary registrations for ecdh_x25519, ecdh_x448)</a>,
              furthermore some crypto libraries support <a
                rel="nofollow" class="external text"
href="https://en.wikipedia.org/wiki/Comparison_of_TLS_implementations#Supported_elliptic_curves">already
                Curve 25519</a>." </li>
          </ul>
        </ul>
        Meanwhile other influences get more weight, e.g. to favor GCM
        over CBC. So, I've discussed this with Achim Hoffmann and we
        have developed the DRAFT for
        '<a class="moz-txt-link-freetext" href="https://www.owasp.org/index.php/TLS_Cipher_String_Cheat_Sheet">https://www.owasp.org/index.php/TLS_Cipher_String_Cheat_Sheet</a>'.
        There has been a Cheat Sheet workshop in Germany the last two
        days and I've sent our draft as input. So now, I am awaiting
        feedback if they had time to discuss it. And yes, it is planned
        to link to the Transport Layer Protection Cheat Sheet with the
        TLS Cipher String Cheat Sheet if the feedback is positive.
      </blockquote>
      <ul>
        <li><b>Additional remarks</b>: The overall setup for DHE/ECDHE
          needs to be up to date: 'Use cryptographic parameters (like
          DH-parameter) that use a secure length that match to the
          supported keylength of your certificate (>=2048 bits or
          equivalent Elliptic Curves).' <br>
          -> As a consequence do not use DHE if you can not use DH
          parameters with at least 2048 bits. Same for ECDHE, if you can
          not use curves with at least 250 bits, e.g.
          secp{256,384,521}r1, x25519 (weaker curves should be
          deactivated, which is still a challenge. In many cases you
          might get a standard list, mostly undocumented/or with a hard
          to find documentation). <br>
        </li>
      </ul>
