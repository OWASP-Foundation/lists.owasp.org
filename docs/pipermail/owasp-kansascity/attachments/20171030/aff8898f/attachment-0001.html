<div dir="auto">Greetings KC chapter,</div><div dir="auto"><br></div><div dir="auto">I have been comparing static analysis software and found the following to be insightful, so passing it along...</div><div dir="auto"><br></div><div dir="auto"><a href="https://www.owasp.org/index.php/Code_Review_Metrics">https://www.owasp.org/index.php/Code_Review_Metrics</a><br></div><div dir="auto"><br></div><div dir="auto"><br></div><div dir="auto"><br></div><div dir="auto">Mat</div><div dir="auto"><br></div><div dir="auto"><br></div><div dir="auto"><br></div><div dir="auto"><h2 style="background-image:none;font-weight:normal;margin:1em 0px 0.25em;overflow:hidden;padding:0px;border-bottom-width:1px;border-bottom-style:solid;border-bottom-color:rgb(170,170,170);font-family:"Linux Libertine",Georgia,Times,serif;line-height:1.3"><span class="mw-headline" id="Secure_Development_Metrics">Secure Development Metrics</span></h2><h3 style="background-image:none;margin:0.3em 0px 0px;overflow:hidden;padding-top:0.5em;padding-bottom:0px;border-bottom-style:none;font-size:1.2em;line-height:1.6;font-family:sans-serif"><span class="mw-headline" id="Defect_Density">Defect Density</span></h3><p style="margin:0.5em 0px;line-height:inherit;color:rgb(37,37,37);font-family:sans-serif">The average occurrence of programming faults per Lines of Code (LOC). This gives a high level view of the code quality but not much more. Fault density on its own does not give rise to a pragmatic metric. Defect density would cover minor issues as well as major security flaws in the code; all are treated the same way. Security of code can not be judged accurately using defect density alone.</p><h3 style="background-image:none;margin:0.3em 0px 0px;overflow:hidden;padding-top:0.5em;padding-bottom:0px;border-bottom-style:none;font-size:1.2em;line-height:1.6;font-family:sans-serif"><span class="mw-headline" id="Lines_of_Code_.28LOC.29">Lines of Code (LOC)</span></h3><p style="margin:0.5em 0px;line-height:inherit;color:rgb(37,37,37);font-family:sans-serif">The count of the executable lines of code. Commented-out code or spaces don't count. This is another metric in an attempt to quantify the size of the code. This gives a rough estimate but is not particularly scientific. Some circles of thinking believe that the estimation of an application size by virtue of LOC is professional malpractice!</p><h3 style="background-image:none;margin:0.3em 0px 0px;overflow:hidden;padding-top:0.5em;padding-bottom:0px;border-bottom-style:none;font-size:1.2em;line-height:1.6;font-family:sans-serif"><span class="mw-headline" id="Function_Point">Function Point</span></h3><p style="margin:0.5em 0px;line-height:inherit;color:rgb(37,37,37);font-family:sans-serif">The estimation of software size by measuring functionality. The combination of a number of statements which perform a specific task, independent of programming language used or development methodology. </p><h3 style="background-image:none;margin:0.3em 0px 0px;overflow:hidden;padding-top:0.5em;padding-bottom:0px;border-bottom-style:none;font-size:1.2em;line-height:1.6;font-family:sans-serif"><span class="mw-headline" id="Risk_Density">Risk Density</span></h3><p style="margin:0.5em 0px;line-height:inherit;color:rgb(37,37,37);font-family:sans-serif">Similar to defect density, but discovered issues are rated by risk (high, medium & low). In doing this we can give insight into the quality of the code being developed via a [<i><b>X Risk / LoC</b></i>] or [<i><b>Y Risk / Function Point</b></i>] value. (X&Y being high, medium or low risks) as defined by your internal application development policies and standards. </p><p style="margin:0.5em 0px;line-height:inherit;color:rgb(37,37,37);font-family:sans-serif">Example:<br>4 High Risk Defects per 1000 (Lines of Code)<br>2 Medium Risk Defects per 3 Function Points<br></p><h3 style="background-image:none;margin:0.3em 0px 0px;overflow:hidden;padding-top:0.5em;padding-bottom:0px;border-bottom-style:none;font-size:1.2em;line-height:1.6;font-family:sans-serif"><span class="mw-headline" id="Path_complexity.2Fcomplexity-to-defect.2Fcyclomatic_complexity">Path complexity/complexity-to-defect/cyclomatic complexity</span></h3><p style="margin:0.5em 0px;line-height:inherit;color:rgb(37,37,37);font-family:sans-serif">Cyclomatic complexity can help establish risk and stability estimations on an item of code, such as a class or method or even a complete system. It was defined by Thomas McCabe in the 70's and it easy to calsulate and apply, hence its usefulness. </p><p style="margin:0.5em 0px;line-height:inherit;color:rgb(37,37,37);font-family:sans-serif">CC = Number of decisions +1</p><p style="margin:0.5em 0px;line-height:inherit;color:rgb(37,37,37);font-family:sans-serif">A decision could be considered commands such as:</p><p style="margin:0.5em 0px;line-height:inherit;color:rgb(37,37,37);font-family:sans-serif">If....else<br>switch<br>case<br>catch<br>while<br>do<br>and so on.....</p><p style="margin:0.5em 0px;line-height:inherit;color:rgb(37,37,37);font-family:sans-serif">As the decision count increases, so does the complexity. Complex code leads to less stability and maintainability. </p><p style="margin:0.5em 0px;line-height:inherit;color:rgb(37,37,37);font-family:sans-serif">The more complex the code, the higher risk of defects. One could establish thresholds for Cyclomatic complexity: </p><p style="margin:0.5em 0px;line-height:inherit;color:rgb(37,37,37);font-family:sans-serif">0-10: Stable code. Acceptable complexity<br>11-15: Medium Risk. More complex<br>16-20: High Risk code. Too many decisions for a unit of code.<br></p><h2 style="background-image:none;font-weight:normal;margin:1em 0px 0.25em;overflow:hidden;padding:0px;border-bottom-width:1px;border-bottom-style:solid;border-bottom-color:rgb(170,170,170);font-family:"Linux Libertine",Georgia,Times,serif;line-height:1.3"><span class="mw-headline" id="Review_Process_Metrics">Review Process Metrics</span></h2><h3 style="background-image:none;margin:0.3em 0px 0px;overflow:hidden;padding-top:0.5em;padding-bottom:0px;border-bottom-style:none;font-size:1.2em;line-height:1.6;font-family:sans-serif"><span class="mw-headline" id="Inspection_Rate">Inspection Rate</span></h3><p style="margin:0.5em 0px;line-height:inherit;color:rgb(37,37,37);font-family:sans-serif">This metric can be used to get a rough idea of the required duration to perform a code review. The inspection rate is the rate of coverage a code reviewer can cover per unit of time. From experience, a rate of 250 lines per hour would be a baseline. This rate should not be used as part of a measure of review quality, but simply to determine duration of the task.</p><h3 style="background-image:none;margin:0.3em 0px 0px;overflow:hidden;padding-top:0.5em;padding-bottom:0px;border-bottom-style:none;font-size:1.2em;line-height:1.6;font-family:sans-serif"><span class="mw-headline" id="Defect_Detection_Rate">Defect Detection Rate</span></h3><p style="margin:0.5em 0px;line-height:inherit;color:rgb(37,37,37);font-family:sans-serif">This metric measures the defects found per unit of time. Again, can be used to measure performance of the code review team, but not to be used as a quality measure. Defect detection rate would normally increase as the inspection rate (above) decreases.</p><h3 style="background-image:none;margin:0.3em 0px 0px;overflow:hidden;padding-top:0.5em;padding-bottom:0px;border-bottom-style:none;font-size:1.2em;line-height:1.6;font-family:sans-serif"><span class="mw-headline" id="Code_Coverage">Code Coverage</span></h3><p style="margin:0.5em 0px;line-height:inherit;color:rgb(37,37,37);font-family:sans-serif">Measured as a % of LoC of function points, the code coverage is the proportion of the code reviewed. In the case of manual review we would aim for close to 100%, unlike automated testing wherein 80-90% is considered good. In order to ensure that the code coverage standards are met, some organizations might implement a safety check during the build process, so the build will fail if there are any piece of code that has not been tested or if it the coverage is under the desired percentage. The higher the percentage of code coverage, the better to ensure quality and prevent logic errors.</p><h3 style="background-image:none;margin:0.3em 0px 0px;overflow:hidden;padding-top:0.5em;padding-bottom:0px;border-bottom-style:none;font-size:1.2em;line-height:1.6;font-family:sans-serif"><span class="mw-headline" id="Defect_Correction_Rate">Defect Correction Rate</span></h3><p style="margin:0.5em 0px;line-height:inherit;color:rgb(37,37,37);font-family:sans-serif">The amount of time used to correct detected defects. This metric can be used to optimise a project plan within the SDLC. Average values can be measured over time, producing a measure of effort which must be taken into account in the planning phase. </p><h3 style="background-image:none;margin:0.3em 0px 0px;overflow:hidden;padding-top:0.5em;padding-bottom:0px;border-bottom-style:none;font-size:1.2em;line-height:1.6;font-family:sans-serif"><span class="mw-headline" id="Reinspection_Defect_Rate">Reinspection Defect Rate</span></h3><p style="margin:0.5em 0px;line-height:inherit;color:rgb(37,37,37);font-family:sans-serif">The rate at which upon re-inspection of the code more defects exist, some defects still exist, or other defects manifest through an attempt to address previously discovered defects.</p></div>
