<blockquote cite="mid:1731936916.1251223339445.JavaMail.nobody@james0"
 type="cite">I've handled this type of incident a couple of times.<br>
  <br>
Server-side AV software is not useless and it can detect well-known
injected strings.<br>
The administrator should schedule the  daily update and scan, which
causes server overhead for hundred of web sites.<br>
Also this is a huge need to analyze found malwares for false positives,
which makes server AV software not able to delete/remove anything they
have found.<br>
Due to this, you'll never see AV scanners (daily scan) in web server
environments till a malicious virus spread occurs.<br>
  <br>
Client-side AV can be effective for well-known injected strings. <br>
  <br>
But not all AV can detect such. You can check it out by saving the
following sample as a text file and submitting to <a
 moz-do-not-send="true" href="http://virustotal.com">virustotal.com</a>.<br>

The main reason why JavaScript malware injection is always successful
at attackers' heart is that<br>
AV softwares today lack JavaScript emulation engine (as far as I know)
for malware analysis. <br>
Most use signatures. Most run at default - lowest heuristic settings.<br>
I've made a demo movie here: <a moz-do-not-send="true"
 href="http://yehg.net/lab/pr0js/files.php/DefeatingSignature-BasedAVScanners.zip">http://yehg.net/lab/pr0js/files.php/DefeatingSignature-BasedAVScanners.zip</a><br>
  <br>
  <br>
Causes:<br>
==========<br>
  <br>
There is a number of possible causes - security weaknesses on the
target. <br>
Common are insecure server configurations (Application/Web/Database),
web applications and weak FTP passwords. <br>
  <br>
  <br>
Counter-measures<br>
=============<br>
  <br>
It's all security hardening/patching/monitoring. There is nothing more
to say. <br>
  <br>
So, what about against z-day exploits ?<br>
  <br>
A lot of security companies have launched remote malware monitoring
sevice:<br>
  <a moz-do-not-send="true"
 href="http://www.google.com.sg/search?q=malware+monitoring+service">http://www.google.com.sg/search?q=malware+monitoring+service</a><br>
A well-known one is <a moz-do-not-send="true"
 href="http://hackalert.armorize.com/">http://hackalert.armorize.com/</a><br>
A cheap solution is that you can write a script/software to<br>
periodically search "This site may harm ..." in  Google search term
with your site.<br>
  <br>
Again, they all have limitations like AV softwares. They detect only
known injections.<br>
But you'll be notified as soon as injection occurs via periodic scans,
which gives you <br>
time to save from Google Blacklisting. ()<br>
  <br>
What about other tricks you can use?<br>
  <br>
- You can periodically scan files for injection with clean md5 hash
database like what Tripwire does.<br>
- You can encrypt server-side files (aspx,php..etc) which contains
self-md5 checking that automatically<br>
   mail/sms you  If mismatch is found. For this, you have to use a
(commercial) source code encrypter.<br>
  <br>
  <br>
You can individually feel free to contact &amp; get help from me.<br>

I will like to seek feedback from the group on how you will address
such a scenario:<br>
    <br>
That the web server is infected with malicious Javascript which does
not affect the infected web server, but affects all visitors to the

elsewhere.<br>

    <br>
1) Server-side AV software is useless here as the malware is not on the
affected web server.<br>
2) Client-side AV software is not effective if the malware could be a
zero-day exploit.<br>
(The successful infection of UniSIM students in Jan this year shows
that client-side AV is not foolproof.)<br>
    <br>
There are tons of ways for the malicious Javascript to get into the
affected web server. <br>

counter-measures.<br>
