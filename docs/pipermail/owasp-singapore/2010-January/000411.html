>>     I mean no disrespect, but your questions below show that you lack

>>     algorithms in use today.
>>
>>     This is a problem. Why?
>>
>>     Well, if the data for the project is important enough to be
>>     encrypted, it is important that it be done right. You clearly
>>     need an expert to assist you.
>>
>>     Cryptography is hard. Even just using crypto API's correctly is
>>     hard. Don't be misled by the way it has been commoditized. You
>>     are going to need help with things like proper padding,
>>     mode-of-operation selection and secure key-handling.
>>
>>     As an example, due to an attack published last year 128-bit AES
>>     is in fact now more secure than 256-bit AES. Surprised? But you
>>     didn't need a 256-bit key anyway. Nobody is going to brute-force
>>     128-bit AES before our sun burns out.
>>
>>     But to answer your question, there is no practical limit to the
>>     length of data that can be encrypted with AES. AES and RSA are
>>     fundamentally different, both in how they are applied, but also
>>     in how each algorithm operates on data.
