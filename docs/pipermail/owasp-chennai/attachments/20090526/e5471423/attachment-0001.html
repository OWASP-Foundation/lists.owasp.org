hi arun,<div><br></div><div>there are quite a few things you would have to consider when testing a thick client application.</div><div>some of the most basic checks that you can do with your existing understanding of web security are as below:</div>
<div><br></div><div>1) check for SQL Injection</div><div>2) look at the communication channel between the client and server, is it encrypted</div><div>3) check for any hard-coded passwords or encryption keys or any other sensitive information in the client-side.</div>
<div>4) if there is any kind of access control based on accounts try to see if you can bypass that by directly communicating with the server.</div><div>5) check out what files and registries this client reads from/writes to and see if they contain any sensitive information</div>
<div><br></div><div>if the application is coded in native language or its managed code calling native code internally then buffer overflows are a possibility, so you either end up with a source code review or fuzzing the application, if you have the required skills.</div>
<div><br></div><div>dont get too carried away by the technology part of it...in all probability most real life abuses of the application would be an intelligent abuse of some seemingly naive feature.</div><div><br></div><div>
so look at all the feature of the applications and how they work and see how they can be abused.</div><div><br></div><div>this would not be a perfect approach but it would get you quite far in terms of locking it down.</div>
<div><br></div><div>if its a very critical application and the associated paranoia is high then get an expert to test it.</div><div><br></div><div>hope this helps</div><div><br></div><div>cheers</div><div>lava</div><div><br>
