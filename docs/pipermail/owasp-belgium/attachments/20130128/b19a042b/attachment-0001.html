<b>Topic:</b> Data events, or why security is cloudier than you think<br>
<br>
Data security doesn't involve just securing data at rest or in transit. It also
needs to be secured in use ­ which means that at any point, the characteristics
of the data can change. We call this situation a "data event," and it
can mean that security requirements have to change as a result. <br>
<br>
This is not the same thing as logging event data; this is taking into account
changes in the combination, use or business context surrounding specific data.
For example, a press release is confidential and requires a certain set of
security policies in the areas of access control, DLP, key management (if
encryption is involved), and so on. But once the business event occurs, the
press release suddenly becomes the opposite of confidential, and all the
policies have to change immediately as a result. <br>
<br>
Data events can also occur when data elements are combined in particular ways
so that they become covered by regulations. A query might produce a small
enough sample size that it needs to be treated as protected information, or a
doctor becomes a patient so that her name is now protected by HIPAA. Data
events are often tied closely to the business context, and as such can mirror
transactions and workflows. <br>
<br>
Data events are important because traditional security policies have been
applied to the current container of the data: this database is confidential
because some rows are confidential, or this Word document requires access
control (but its content can be copied and pasted somewhere else).
Container-centric security is too static for today's high-speed, big-data,
cloud-based (pick as many buzzwords as you like) processing. <br>
<br>
This talk will describe the concept of data events, and will invite audience
discussion on how security controls can be adapted to them. <br>
