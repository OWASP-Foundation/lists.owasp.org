I agree that TDD doesnt usually address security, mostly because many
(most?) developers dont understand security (and I'm a developer;).

But how about if you use those tests to drive the security tools?
Thats what I'm introducing where I work.
Developers write UI tests using tools like Selenium.
Then we proxy Selenium through ZAP, run the Selenium tests then run the ZAP
spider, passive and active scanners.
Using the tests in this way means you should get better coverage and drive
the application in a more realistic way.
And developers dont have to know about all the different attack vectors.
A simple example is here: https://code.google.com/p/bodgeit/wiki/RegTests
No automated scanner will find all security vulnerabilities, but the above
setup will hopefully catch silly mistakes within hours of the code being
committed, so the pentesters can concentrate on the more interesting
problems.

Thats my theory anyway :)

> is further compounded by the fact that TDD is about tests to make sure

> > To inspire secure software I think we need to inspire thoroughly correct

> must be told how to do the job well.

> > If I had to pick 3 points, one of them would be from the test-driven
> development camp. Basically provoking every developer who says her
> software/module/function does X to provide some reason (e.g., a unit test,

> entirely too prone to believe things work, without applying a healthy
> filter of skepticism. Children are brought up amongst technology that looks
> like magic to OAPs. They see it work over and over and don't develop the

> hardware fails, and people make mistakes. Testing while developing is vital
> for catching problems before they grow into larger issues. I will bet that
