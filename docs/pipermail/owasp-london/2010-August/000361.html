Hi Sureshbabu,

Are there any particular threats you're looking to address (i.e. are you fixing a known issue?), or are you looking to secure this in general?

The problem of handling user input will generally involve some combination of input validation and output encoding to get where you probably want to be.  As an example, you're outlining a black-list input validation approach (defining "known bad" characters) which may be susceptible to bypass in certain circumstances if an alternative encoding can be used.  Additionally, the HTML encoding noted may also be susceptible to the same problem.

In both these cases, you would need to take a more thorough approach to be more secure - for example only accepting known good characters (white-list validation) and encoding all characters except for those that are known to be safe (OWASP Encoding and Microsoft Anti-XSS take this approach). Of course, doing this may not be justified for all situations.

My 2c

Justin
